요즘 백엔드 씬에서 그냥 SQLite 쓰세요라는 말이 꽤 자주 들립니다. Litestream이나 LiteFS 같은 도구들이 발전하면서, VPS 한 대에 SQLite를 올려두고 꽤 규모 있는 트래픽을 감당하는 아키텍처가 실용적인 선택지가 되었죠.

하지만 이 단일 노드 SQLite 아키텍처를 운영하다 보면 반드시 거대한 벽을 마주하게 됩니다. 바로 백그라운드 작업 큐(Task Queue)와 Pub/Sub 시스템이 필요해지는 순간입니다. 보통 이 시점에 도달하면 관성적으로 Redis 띄우고 Celery 붙이자라는 결론을 내리게 됩니다. 그리고 그 순간, 우리는 두 개의 데이터스토어를 관리해야 하고, 비즈니스 로직과 큐 사이의 이중 쓰기(Dual-write) 문제와 분산 트랜잭션의 악몽에 발을 들이게 되죠.

오늘 Hacker News에서 이 문제를 정면으로 돌파하려는 흥미로운 프로젝트를 발견했습니다. Russell Romney가 공개한 **Honker** 입니다.

## Honker: 데몬 없는 SQLite Pub/Sub 및 Task Queue

Honker는 한마디로 SQLite에 Postgres의 NOTIFY/LISTEN 시맨틱을 추가해주는 확장(Extension)입니다. 별도의 브로커나 데몬 프로세스 없이, 기존에 사용하던 .db 파일 하나만으로 내구성 있는 Pub/Sub, 작업 큐, 이벤트 스트림을 구현합니다.

제가 이 프로젝트에서 가장 높게 평가하는 부분은 비즈니스 로직과 이벤트 발행을 단일 트랜잭션으로 묶어버렸다는 점입니다.

```python
with db.transaction() as tx:
    tx.execute("INSERT INTO orders (user_id) VALUES (?)", [42])
    emails.enqueue({"to": "alice@example.com"}, tx=tx)
```

위 코드처럼 주문을 생성하는 INSERT 쿼리와 이메일 발송 큐를 하나의 트랜잭션으로 묶습니다. 만약 비즈니스 로직에서 에러가 발생해 롤백되면, 큐에 들어간 작업도 깔끔하게 날아갑니다. 외부 메시지 브로커를 쓸 때 흔히 발생하는 DB는 롤백됐는데 메시지는 이미 발행되어 워커가 돌아가는 끔찍한 정합성 깨짐 문제가 원천적으로 차단되는 것이죠. 이 패턴은 과거 제가 금융권 프로젝트에서 Outbox 패턴을 쌩고생하며 구현했던 기억을 떠올리게 하더군요. 이걸 프레임워크 레벨에서 기본 지원한다는 건 엄청난 메리트입니다.

## 어떻게 폴링 없이 1ms Latency를 달성했나?

SQLite는 Postgres처럼 서버 데몬이 떠서 클라이언트의 커넥션을 관리하는 구조가 아닙니다. 그렇다면 어떻게 변경 사항을 감지해서 다른 프로세스에게 푸시해줄 수 있을까요?

Honker의 핵심 트릭은 SQLite의 WAL(Write-Ahead Log) 모드와 OS 레벨의 stat(2) 시스템 콜에 있습니다.

데이터베이스를 직접 폴링하는 대신, 백그라운드 스레드가 1ms 간격으로 .db-wal 파일의 크기와 수정 시간을 stat(2)로 감시합니다. 파일에 변경이 감지되면 구독자들에게 신호를 보내고, 구독자들은 부분 인덱스를 태워 변경된 row만 빠르게 SELECT 해옵니다.

여기서 시니어 엔지니어라면 당연히 이런 의문이 들 겁니다. 아니, 파일 변경 감지면 inotify나 kqueue 같은 우아한 OS API를 놔두고 왜 무식하게 1ms마다 stat(2) 폴링을 때리지?

저자도 바보는 아니었습니다. macOS의 FSEvents는 동일 프로세스 내에서 발생한 Write 이벤트를 조용히 씹어버리는 고질적인 문제가 있다고 합니다. 크로스 플랫폼에서 일관된 동작을 보장하기 위해, 가장 무식하지만 가장 확실한 stat(2)를 선택한 것이죠.

## HN 커뮤니티의 반응과 나의 시선

Hacker News 쓰레드에서는 예상대로 이 폴링 구조에 대한 갑론을박이 뜨거웠습니다.

- **Syscall Overhead:** 1ms마다 시스템 콜을 호출하는 것이 캐시를 더럽히고 성능을 저하시킨다는 지적이 있었습니다. 누군가는 io_uring을 쓰라고 조언하기도 했죠. 하지만 현대 하드웨어에서 stat(2)는 1µs 이하로 떨어지며, 실제 DB 쿼리 폴링에 비하면 새 발의 피 수준의 오버헤드입니다.
- **더 나은 대안:** 쓰레드에서 나온 가장 영양가 있는 피드백은 SQLite의 내장 기능인 PRAGMA data_version이나 C API인 SQLITE_FCNTL_DATA_VERSION을 사용하라는 것이었습니다. 저자 역시 이를 인정하고 추후 이 방향으로 최적화하겠다고 답변했습니다. 파일시스템에 의존하는 것보다 DB 엔진이 보장하는 API를 타는 것이 WAL Checkpoint 같은 엣지 케이스를 다룰 때 훨씬 안전합니다.

또한, Java나 Go 같은 언어면 그냥 메모리 큐 쓰면 되는데 왜 굳이 이런 게 필요하냐는 질문도 있었습니다. 하지만 Python, Ruby 생태계에서는 여전히 프로세스 기반의 동시성 모델이 주류이며, 무중단 배포 시 구버전과 신버전 프로세스가 공존하는 상황을 고려하면 프로세스 간 통신을 DB가 중재해주는 이 모델은 매우 실용적입니다.

## 총평: 프로덕션에 쓸 수 있을까?

솔직히 말씀드리면, 아직 프로덕션의 메인 워크로드에 올리기엔 이릅니다. 저자 본인도 실험적 단계라고 명시했고, WAL 파일이 Truncate 될 때의 폴링 안정성 등 검증해야 할 엣지 케이스가 남아있습니다.

하지만 아키텍처의 방향성만큼은 박수를 쳐주고 싶습니다. 모든 것을 하나의 SQLite 파일로 해결한다는 철학을 극단으로 밀어붙인 훌륭한 엔지니어링 시도입니다. 사이드 프로젝트나 내부 백오피스 툴링, 혹은 트래픽이 예측 가능한 소규모 서비스라면 Redis를 걷어내고 Honker를 도입해 보는 것도 꽤 재밌는 도전이 될 것입니다.

앞으로 C API 기반으로 엔진이 개선되고 메이저 버그들이 잡힌다면, SQLite 생태계에서 Celery를 대체할 강력한 표준으로 자리 잡을 잠재력이 충분해 보입니다.
