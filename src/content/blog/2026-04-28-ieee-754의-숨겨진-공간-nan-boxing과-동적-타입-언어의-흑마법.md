엔지니어로서 우리는 NaN(Not-a-Number)을 별로 좋아하지 않습니다. 모니터링 대시보드나 결제 로직에 NaN이 뜨는 순간 등골이 서늘해지니까요. 하지만 동적 타입 언어인 JavaScript나 Lua의 런타임 성능이 사실 이 NaN의 숨겨진 공간 덕분에 유지되고 있다는 사실을 아시나요? 오늘은 Annie Cherkaev의 훌륭한 글과 Hacker News의 열띤 토론을 바탕으로, 부동소수점 표준의 가장 기괴하고 아름다운 해킹인 NaN-boxing에 대해 깊게 파헤쳐보겠습니다.

## NaN의 해부학: 51비트의 공터

IEEE 754-2008 표준에서 64비트 double precision을 살펴봅시다. 부호 1비트, 지수 11비트, 가수(mantissa) 52비트로 구성됩니다. NaN은 지수부의 모든 비트가 1로 채워진 상태를 말합니다.

여기서 흥미로운 점은 가수부의 최상위 비트로 Quiet NaN과 Signaling NaN을 구분하고 나면, 무려 51비트의 잉여 공간(payload)이 남는다는 것입니다. 51비트면 현대의 48비트 메모리 포인터나 32비트 정수를 구겨 넣기에 충분하고도 남는 공간입니다.

## 실전 압축 흑마법: JavaScriptCore의 NaN-boxing

실무에서 이 공간을 어떻게 쓸까요? WebKit을 구동하는 **JavaScriptCore** (JSC) 코드를 보면 이 흑마법의 정수를 볼 수 있습니다. 동적 타입 언어는 런타임에 변수의 타입이 바뀔 수 있으므로 값과 타입 태그를 함께 들고 다녀야 합니다. JSC는 이 51비트 페이로드에 포인터, 정수, 불리언 같은 비 부동소수점 값들을 전부 때려 넣습니다.

```cpp
// Pointer { 0000:PPPP:PPPP:PPPP
// Double  { ...
// Integer { FFFF:0000:IIII:IIII
```

JSC는 포인터 접근을 최적화하기 위해 비트 범위를 기발하게 틀어버립니다. 2^48 값을 의미하는 **DoubleEncodeOffset** 을 더해서 진짜 double 값을 위로 밀어버리고, `0x0000`으로 시작하는 영역을 포인터 전용으로 비워둡니다. 덕분에 포인터는 별도의 마스킹 없이 그대로 꺼내 쓸 수 있죠.

## 나의 생각: 천재적이면서도 시한폭탄 같은 패턴

솔직히 제가 처음 이 패턴을 접했을 때, 천재적이라는 감탄과 동시에 경악을 금치 못했습니다. 가비지 컬렉션(GC) 부하를 줄이기 위해 모든 원시 타입과 포인터를 64비트 레지스터 하나에 구겨 넣는 발상은 경이롭지만, 구조적으로는 시한폭탄이나 다름없습니다.

실제로 WebKit에서 발생했던 수많은 취약점(예: CVE-2010-1807)이 바로 이 NaN 페이로드 검증 로직의 버그에서 비롯되었죠. 게다가 미래에 하드웨어가 64비트 메모리 주소 공간을 온전히 사용하게 되는 날이 오면, 상위 16비트를 타입 태그로 쓰는 수많은 VM들은 대대적인 재작업을 피할 수 없을 겁니다.

## Hacker News의 시선: D 언어의 우아한 철학

이번 Hacker News 스레드에서 제 눈길을 사로잡은 건 커뮤니티 유저들의 다양한 실무 경험이었습니다. 특히 눈에 띄는 통찰들을 정리해 보았습니다.

- **D 언어의 철학:** D 언어는 부동소수점 변수를 초기화하지 않으면 0.0이 아니라 NaN으로 초기화합니다. 컴파일러 경고를 끄기 위해 생각 없이 0을 집어넣는 주니어 시절의 실수를 막기 위함입니다. 0.0으로 인한 계산 오류는 production 환경에서 조용히 시스템을 망가뜨리지만, NaN은 즉각적으로 시스템에 이상이 있음을 비명 지르듯 알려줍니다.
- **센서 데이터 처리:** 센서가 고장 났다면 0이 아니라 NaN을 반환해야 수신 측에서 데이터가 오염되었음을 명확히 알 수 있습니다.
- **GC 최적화 이점:** 부동소수점 객체를 매번 메모리에 할당할 필요가 없으므로 가비지 컬렉션 타임이 획기적으로 줄어듭니다.

## 결론 (Verdict)

결론적으로 **NaN-boxing** 기법은 극한의 메모리 최적화와 성능이 필요한 언어 런타임 레벨에서는 여전히 유효하고 매력적인 무기입니다.

하지만 일반적인 비즈니스 로직이나 애플리케이션 레벨에서 이런 꼼수를 부리는 것은 명백한 오버엔지니어링입니다. 애플리케이션 엔지니어라면 NaN을 본연의 목적, 즉 '시스템의 이상 상태'나 '유효하지 않은 데이터'를 명시적으로 나타내는 데에만 사용하는 것이 현명합니다.
