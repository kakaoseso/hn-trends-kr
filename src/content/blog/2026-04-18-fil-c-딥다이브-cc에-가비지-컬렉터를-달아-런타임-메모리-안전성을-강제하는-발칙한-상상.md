요즘 어딜 가나 'Rust로 재작성(Rewrite it in Rust)'하라는 이야기가 들린다. 메모리 안전성(Memory Safety)이 업계의 가장 중요한 화두가 된 지 오래고, C/C++는 마치 시한폭탄 취급을 받고 있다. 하지만 수십 년간 쌓여온 수백만 줄의 레거시 C/C++ 코드를 하루아침에 Rust로 포팅하는 것이 현실적으로 가능할까? 당연히 불가능하다.

이런 상황에서 매우 흥미롭고, 한편으로는 논쟁적인 프로젝트가 Hacker News를 뜨겁게 달궜다. 바로 **Fil-C** 다. Fil-C는 C/C++ 소스 코드를 변환하여 **완벽한 런타임 메모리 안전성** 을 보장하겠다고 나선 프로젝트다. 놀랍게도 이 목표를 달성하기 위해 그들은 C/C++에 **가비지 컬렉터(GC)** 를 도입했다.

15년 넘게 시스템 프로그래밍과 인프라 아키텍처를 다뤄온 엔지니어의 관점에서, 처음 이 아이디어를 접했을 때는 솔직히 코웃음을 쳤다. 'C/C++의 장점인 제어력과 성능을 다 포기하면서까지 굳이?'라는 생각이 들었기 때문이다. 하지만 그 내부 구현 원리와 커뮤니티의 토론을 깊게 파고들수록, 이것이 단순한 장난감이 아니라 특정한 도메인에서는 꽤나 현실적인 타협안이 될 수 있겠다는 생각이 들었다.

이번 글에서는 Fil-C가 도대체 어떻게 C/C++를 메모리 안전하게 만드는지 그 기술적 디테일을 파헤쳐보고, 이것이 과연 실무에 도입할 만한 가치가 있는지 내 개인적인 시각과 HN 커뮤니티의 반응을 곁들여 정리해 보겠다.

## 어떻게 동작하는가: Fat Pointer의 극단적 진화

Fil-C의 핵심 아이디어는 사실 시스템 프로그래밍에서 꽤 오래된 개념인 'Fat Pointer'를 극단적으로 밀어붙인 것이다. 컴파일러(LLVM IR 패스)가 개입하여 모든 포인터 변수에 메타데이터를 담은 `AllocationRecord*`를 강제로 짝지어 버린다.

단순화된 C 코드로 보면 아래와 같은 변환이 일어난다.

```c
// Original Source
void f() {
  T1* p1;
  T2* p2;
  uint64_t x;
  // ...
}

// After Fil-C Transform
void f() {
  T1* p1; AllocationRecord* p1ar = NULL;
  T2* p2; AllocationRecord* p2ar = NULL;
  uint64_t x;
  // ...
}
```

여기서 `AllocationRecord`는 메모리 경계(Bounds)를 추적하기 위한 구조체다.

```c
struct AllocationRecord {
  char* visible_bytes;    // 실제 데이터가 있는 곳
  char* invisible_bytes;  // 중첩된 포인터의 메타데이터가 있는 곳
  size_t length;          // 할당된 길이
};
```

### 3중 할당을 수행하는 filc_malloc

Fil-C 환경에서 `malloc`을 호출하면 단순히 메모리 한 덩어리를 받는 것이 아니다. 내부적으로 `filc_malloc`으로 치환되며, 무려 3번의 할당이 일어난다.

1. `AllocationRecord` 자체를 위한 할당
2. 실제 데이터가 저장될 `visible_bytes` 할당
3. 포인터 내부에 또 다른 포인터가 저장될 경우, 그 메타데이터를 추적하기 위한 `invisible_bytes` 할당

가장 흥미로운 부분은 **invisible_bytes** 의 존재다. 지역 변수에 있는 포인터는 컴파일러가 쉽게 메타데이터를 끼워 넣을 수 있지만, 힙(Heap)에 저장된 구조체 내부의 포인터는 추적하기 까다롭다. Fil-C는 힙 메모리의 특정 오프셋 `i`에 포인터가 저장되면, 정확히 `invisible_bytes + i` 위치에 해당 포인터의 `AllocationRecord*`를 저장하는 방식으로 이 문제를 우회한다. 굉장히 무식해 보이지만 런타임에 메모리 레이아웃을 완벽히 섀도잉(Shadowing)하는 확실한 방법이다.

### 경계 검사 (Bounds Checking)

포인터를 역참조(`*p1 = x;`)할 때마다 컴파일러는 아래와 같은 무시무시한 어서션(Assertion)을 삽입한다.

```c
assert(p1ar != NULL);
uint64_t i = (char*)p1 - p1ar->visible_bytes;
assert(i < p1ar->length);
assert((p1ar->length - i) >= sizeof(*p1));
*p1 = x;
```

모든 메모리 접근에 대해 할당된 범위를 벗어나는지 런타임에 검사하는 것이다. 범위를 벗어나면 자비 없이 프로그램은 크래시(Crash)된다.

## 코끼리를 냉장고에 넣는 방법: C/C++에 GC 도입하기

이 아키텍처를 보다 보면 필연적으로 발생하는 문제가 있다. `filc_malloc`에서 3개의 객체를 할당했는데, 사용자가 `free`를 호출하면 어떻게 될까? `visible_bytes`와 `invisible_bytes`는 해제되지만, `AllocationRecord` 자체는 해제되지 않는다. 누군가 여전히 그 레코드를 참조하고 있을 수 있기 때문이다(Dangling Pointer 방지).

이 간극을 메우기 위해 Fil-C는 **가비지 컬렉터(GC)** 를 도입했다. 프로덕션 버전에서는 동시성 점진적 GC(Concurrent Incremental GC)가 동작하며, 도달할 수 없는(Unreachable) `AllocationRecord`를 백그라운드에서 정리한다.

이로 인해 얻는 부수적인 효과는 엄청나다. C/C++ 프로그래머의 영원한 숙제인 **메모리 누수(Memory Leak)가 더 이상 치명적이지 않게 된다.** `free`를 호출하는 것을 잊어버려도 GC가 알아서 메모리를 수거해 간다. 물론 `free`를 명시적으로 호출하면 메모리를 더 빨리 반환할 수 있으므로 여전히 유용하지만 말이다.

## 엔지니어의 시선: 이것은 혁신인가, 끔찍한 혼종인가?

이 기술을 실무 관점에서 바라보면 몇 가지 치명적인 트레이드오프가 눈에 띈다.

### 1. 성능과 메모리 오버헤드

당연하게도 이 방식은 **엄청나게 느리고 무겁다.** 모든 포인터가 두 배 이상의 공간을 차지하고, 모든 메모리 접근에 분기문(Bounds Check)이 추가되며, 캐시 지역성(Cache Locality)은 박살 난다. HN 댓글에서도 지적하듯, 이 정도의 성능 하락을 감수할 거라면 차라리 처음부터 Java나 C#을 쓰는 게 맞지 않냐는 비판이 나온다.

하지만 Fil-C 제작자의 목표는 명확하다. 성능을 20~30% 하락 수준으로 방어하는 것이다. 만약 당신이 초당 수백만 건의 트랜잭션을 처리하는 HFT(고빈도 매매) 엔진을 만든다면 Fil-C는 최악의 선택이다. 하지만 이따금씩 호출되는 레거시 이미지 파싱 라이브러리나 텍스트 프로세서라면? 보안 취약점을 막기 위해 30%의 성능 하락을 기꺼이 지불할 기업은 널려 있다.

### 2. 런타임 크래시 vs 컴파일 타임 증명

HN 커뮤니티에서 가장 치열하게 맞붙은 논쟁 중 하나는 **"크래시가 나는 것을 메모리 안전하다고 부를 수 있는가?"** 였다.

- **비판 측:** Rust나 WUFFS처럼 컴파일 타임에 안전성을 증명하는 것이 진짜 안전성이다. Fil-C는 잘못된 메모리 접근 시 프로그램을 그냥 죽여버린다(`exit(-1)`). 웹 서버라면 단일 요청의 오류로 전체 프로세스가 죽는 서비스 거부(DoS) 상태가 되는데, 이것이 어떻게 안전하냐는 것이다.
- **옹호 측:** 보안 관점에서 메모리 오염(Memory Corruption)으로 인한 임의 코드 실행(RCE)을 허용하는 것보다 깔끔하게 죽는(Fail-fast) 것이 100배 낫다. Go나 Java 역시 배열 인덱스를 벗어나면 런타임 패닉을 발생시킨다.

내 생각은 이렇다. Fil-C의 방식은 **보안(Security)** 측면에서는 완벽한 방어막이지만, **가용성(Availability)** 측면에서는 약점이 명확하다. 예외 복구(Exception Recovery) 메커니즘이 부족한 것은 아쉽지만, C의 언어적 한계를 생각하면 어쩔 수 없는 선택이다.

### 3. 생태계와 ABI 호환성 (BYOlibc)

가장 뼈아픈 단점은 Fil-C로 컴파일된 코드는 기존의 미리 컴파일된 C 라이브러리(심지어 표준 libc조차)와 링크할 수 없다는 점이다. 포인터의 크기와 호출 규약(ABI)이 완전히 달라지기 때문이다. 이로 인해 Fil-C를 사용하려면 의존하는 모든 라이브러리를 Fil-C로 다시 컴파일해야 한다. 제작자가 freetype, fontconfig 등을 직접 포팅하고 있는 이유이기도 하다.

## 결론: 언제 Fil-C를 써야 할까?

솔직히 말해서, 새로운 프로젝트를 시작한다면 주저 없이 Rust나 Go를 선택할 것이다. 하지만 현실 세계에는 여전히 동작하지만 누구도 건드리고 싶어 하지 않는, 수백만 줄의 C/C++ 코드가 존재한다.

Fil-C는 다음과 같은 상황에서 빛을 발할 수 있다.

- **레거시 샌드박싱:** 도저히 Rust로 재작성할 예산과 시간은 없지만, 외부 입력을 처리해야 해서 보안이 극도로 중요한 C/C++ 모듈이 있을 때.
- **ASan을 대체하는 디버깅 툴:** AddressSanitizer(ASan)보다 더 강력하고 엄격하게 메모리 버그를 잡아내고 싶을 때.
- **컴파일 타임 평가:** Zig처럼 컴파일 타임과 런타임 언어가 동일한 경우, 컴파일 타임에 코드를 안전하게 실행하기 위한 샌드박스 엔진으로 활용.

Fil-C는 "C/C++는 본질적으로 안전할 수 없다"는 업계의 고정관념에 정면으로 도전하는 흥미로운 프로젝트다. 비록 주류 언어나 컴파일러를 완전히 대체하지는 못하겠지만, 시스템 프로그래머들에게 포인터 출처(Pointer Provenance)와 메모리 모델에 대해 깊은 영감을 주는 훌륭한 레퍼런스임은 틀림없다.
