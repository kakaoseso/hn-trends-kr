사이버 보안 업계에 15년 이상 몸담으면서 수많은 멀웨어와 APT(지능형 지속 위협) 보고서를 읽어왔습니다. 대부분은 뻔합니다. 데이터를 빼돌리거나, 시스템을 암호화하고 랜섬을 요구하거나, 디도스 공격을 위한 봇넷을 구축하죠. 하지만 가끔 엔지니어로서의 상식을 뒤흔드는 우아하고 소름 돋는 코드를 발견할 때가 있습니다. 

최근 SentinelLabs에서 공개한 **fast16** 에 대한 분석이 바로 그런 사례입니다. Stuxnet이 세상에 알려지기 무려 5년 전인 2005년에 컴파일된 이 프레임워크는, 데이터를 훔치는 대신 물리적 세계를 시뮬레이션하는 소프트웨어의 **수학적 계산을 조작** 하도록 설계되었습니다. 오늘은 이 경이로운(그리고 끔찍한) 소프트웨어 사보타주 프레임워크의 기술적 디테일과, Hacker News 커뮤니티에서 오간 흥미로운 통찰들을 엔지니어의 시각에서 파헤쳐 보겠습니다.

## 2005년에 등장한 Lua 기반의 모듈러 아키텍처

솔직히 처음 이 기사를 읽었을 때 가장 놀랐던 점은 아키텍처의 현대성입니다. `svcmgmt.exe`라는 이름의 이 2005년형 캐리어 바이너리는 내부에 **Lua 5.0 가상 머신** 을 탑재하고 있습니다. 

우리가 보통 C++로 고성능 멀웨어를 짤 때 가장 골치 아픈 것이 유연성입니다. 새로운 기능이 필요할 때마다 전체 임플란트를 다시 컴파일하고 배포하는 것은 리스크가 크죠. fast16의 개발자들은 이 문제를 해결하기 위해 핵심 로직을 암호화된 Lua 바이트코드로 분리했습니다. Flame이나 Project Sauron 같은 최상위 국가급 해킹 툴들이 Lua를 내장한 것으로 유명한데, fast16은 이들보다 최소 3년에서 10년 이상 앞서 이 패턴을 구현했습니다.

![Composition of the Carrier payload](https://www.sentinelone.com/wp-content/uploads/2026/04/fast16_7.jpg)

위 다이어그램에서 볼 수 있듯, 캐리어는 실행 래퍼 역할만 하고 실제 환경 설정, 전파 로직, 그리고 페이로드(Kernel Driver)는 철저히 모듈화되어 있습니다. 당시 유행하던 조잡한 매크로 바이러스나 웜들과는 차원이 다른, 철저히 엔터프라이즈급 소프트웨어 아키텍처를 따르고 있습니다.

## 커널 레벨 파일시스템 필터: 정밀 타격을 위한 인내심

이 프레임워크의 진짜 무기는 `fast16.sys`라는 커널 드라이버입니다. 이 드라이버는 부팅 초기 단계에 로드되어 모든 파일시스템 디바이스(NTFS, FAT 등) 위에 자신의 워커 디바이스를 붙입니다. `IoRegisterFsRegistrationChange`를 사용해 `IRP_MJ_READ` 같은 I/O 요청을 가로채는 전형적인 루트킷 방식이죠.

하지만 제가 엔지니어로서 가장 감탄한 부분은 이들의 **인내심** 입니다. 이 드라이버는 부팅 시 로드되지만, 시스템이 `explorer.exe`를 띄울 때까지 코드 인젝션 엔진을 활성화하지 않습니다. 부팅 과정에서 무거운 패칭 작업을 수행하면 I/O 병목이 발생해 타겟이 이상함을 눈치챌 수 있기 때문입니다. 성능 최적화와 스텔스를 완벽하게 이해하고 있는 개발자의 솜씨입니다.

### 인메모리 바이너리 패칭과 FPU 조작

엔진이 활성화되면 디스크에서 읽혀 메모리로 올라가는 실행 파일들을 검사합니다. 타겟 조건은 매우 구체적입니다.

- **확장자:** `.EXE` 파일일 것
- **컴파일러 마커:** PE 섹션 헤더 직후에 `Intel`로 시작하는 ASCII 문자열이 있을 것

즉, Intel C/C++ 컴파일러로 빌드된 특정 고성능 연산 프로그램을 노린 것입니다. 조건이 맞으면 드라이버는 인메모리 상태에서 PE 헤더를 수정해 `.xdata`와 `.pdata` 섹션을 주입합니다. 그리고 가장 소름 돋는 작업이 시작됩니다.

![Injected FPU-based calculations](https://www.sentinelone.com/wp-content/uploads/2026/04/fast16_6.jpg)

일반적인 멀웨어처럼 실행 흐름을 훔쳐 쉘코드를 실행하는 것이 아닙니다. 이들은 **부동소수점 연산 장치(FPU)** 명령어 블록을 주입하여, 내부 배열의 스케일링 값을 미세하게 비틉니다. 타겟으로 추정되는 소프트웨어는 LS-DYNA(핵무기 시뮬레이션 및 충돌 테스트에 사용됨), PKPM(건축 구조 해석), MOHID(유체 역학 모델링) 등입니다.

시뮬레이션 결과가 확 틀어지면 과학자들은 버그를 의심할 것입니다. 하지만 아주 미세하게, 그리고 웜을 통해 네트워크 내의 모든 PC가 동일하게 조작된 오답을 내놓는다면? 그들은 그 오답을 진실로 믿고 결함이 있는 교량이나 원자로를 짓게 될 것입니다. 이것은 단순한 해킹이 아니라 현실 세계에 대한 물리적 사보타주입니다.

## 코드에 남겨진 화석: RCS/SCCS와 유닉스 그레이비어드

Hacker News 스레드에서 가장 뜨거웠던 주제 중 하나는 바이너리 내부에 남겨진 `@(#)par.h $Revision: 1.3 $` 같은 문자열이었습니다. 

이것은 70~80년대 유닉스 환경에서 쓰이던 SCCS(Source Code Control System)나 RCS의 마커입니다. 2005년의 윈도우 커널 개발자들은 보통 SourceSafe나 SVN을 썼습니다. HN의 한 유저가 남긴 비유가 아주 찰떡입니다. "이건 최첨단 현대식 사무실에서 다이얼 전화기를 발견한 것과 같다."

이 힌트가 의미하는 바는 명확합니다. fast16의 개발팀은 윈도우 생태계에서 자라난 해커들이 아닙니다. 수십 년 전부터 정부나 군사 기관의 메인프레임과 유닉스 시스템을 다루던, 이른바 '그레이비어드(Graybeard)' 엔지니어들이 주축이 된 국가급 조직이라는 뜻입니다. 

## 시스템의 불일치를 노리는 우아한 공격

또 다른 HN 유저의 통찰력 있는 댓글이 제 눈길을 끌었습니다. 이 공격의 본질은 토마스 프타첵(Thomas Ptacek)이 1998년 논문에서 지적한 "시스템 간의 해석 불일치(Mismatch)"를 극한으로 밀어붙인 형태라는 것입니다.

디스크에 있는 파일의 해시를 백신이 검사할 때는 정상입니다. 하지만 메모리에 로드되는 순간 커널 드라이버가 이를 조작합니다. 보안 시스템과 운영체제가 같은 데이터를 다르게 해석하는 간극을 파고든 것이죠. 더 나아가, 연산 자체를 오염시켜 검증 시스템과 실제 물리계 사이의 불일치를 만들어냅니다. 현대의 LLM 가드레일 우회(Jailbreak) 기법들도 결국 이러한 '해석의 불일치'를 노린다는 점에서, 보안의 근본적인 취약점은 20년 전이나 지금이나 변하지 않았습니다.

```yara
rule apt_fast16_driver {
    meta:
        author = "SentinelLABS/vk"
        description = "Catches fast16 driver or related project files"
    strings:
        $a1 = "@(#)foo.c : "
        $dev1 = "\\Device\\fast16"
        // ... (생략) ...
    condition:
        filesize < 10MB and uint16(0)==0x5a4d and ( ... )
}
```
*SentinelLabs가 공개한 YARA 룰 일부. SCCS 마커($a1)를 탐지 조건으로 사용하고 있다.*

## 결론: 우리는 무엇을 놓치고 있는가?

Fast16은 단순한 레거시 멀웨어가 아닙니다. 이것은 국가급 사이버 무기가 우리가 생각했던 것보다 훨씬 오래전부터, 훨씬 더 정교한 형태로 존재해왔음을 증명하는 디지털 화석입니다. Stuxnet이 우라늄 농축 원심분리기를 물리적으로 파괴했다면, fast16은 과학자들의 머릿속과 시뮬레이션 데이터를 조용히 파괴했습니다.

현업에 있는 시니어 엔지니어로서 이 분석을 보며 깊은 회의감과 경외감을 동시에 느낍니다. 우리는 매일 CI/CD 파이프라인을 최적화하고 마이크로서비스를 구축하지만, 인프라의 가장 밑바닥(커널, 컴파일러, FPU)에 대한 신뢰가 무너졌을 때 시스템이 어떻게 붕괴할 수 있는지 잊고 지냅니다. 

여러분의 서버 리포지토리 어딘가에도, 아직 분석되지 않은 채 '오래된 미확인 샘플'로 분류된 10년 전의 조용한 사보타주 코드가 잠들어 있을지도 모릅니다. 때로는 최신 AI 보안 솔루션을 도입하는 것보다, 우리가 당연하게 믿고 있는 인프라의 하부 구조를 의심해 보는 긱(Geek)스러운 집요함이 필요할 때입니다.
