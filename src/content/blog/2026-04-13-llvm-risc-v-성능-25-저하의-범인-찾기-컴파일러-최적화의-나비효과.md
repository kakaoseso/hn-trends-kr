컴파일러는 종종 마법처럼 느껴집니다. 우리가 작성한 조잡한 코드를 분석하고, 불필요한 연산을 날려버리며, 타겟 아키텍처에 딱 맞는 효율적인 기계어로 번역해 주니까요. 하지만 15년 넘게 엔지니어링을 하면서 뼈저리게 느낀 점이 하나 있다면, **'완벽한 컴파일러는 없다'** 는 것입니다. 가끔은 컴파일러의 똑똑한 최적화가 오히려 성능을 깎아먹는 흑마법이 되기도 합니다.

최근 아주 흥미로운 포스트를 하나 읽었습니다. Kaving이라는 엔지니어가 LLVM의 RISC-V 타겟에서 발생한 25%의 성능 저하(Regression) 원인을 추적하고, 직접 패치까지 기여한 사례입니다. 단순한 버그 픽스를 넘어, 컴파일러의 Middle-end 파이프라인에서 패스(Pass) 간의 상호작용이 어떻게 의도치 않은 결과를 낳는지 보여주는 훌륭한 엔지니어링 딥다이브입니다. 

오늘은 이 사례를 해부해 보면서, 왜 이런 문제가 발생했고 어떻게 해결했는지, 그리고 Hacker News에서 벌어진 흥미로운 논쟁에 대한 제 개인적인 생각까지 나누어 보겠습니다.

## 단 14 Cycle의 차이가 만든 나비효과

문제의 발단은 Igalia의 LLVM vs GCC 벤치마크 결과였습니다. 특정 벤치마크에서 LLVM이 GCC보다 SiFive P550 CPU 기준으로 약 8% 더 많은 Cycle을 소모하고 있었습니다.



어셈블리를 뜯어보니 원인은 명확했습니다. 루프 내부에서 GCC는 `fdiv.s` (Single precision, float 나눗셈)를 사용한 반면, LLVM은 `fdiv.d` (Double precision 나눗셈)를 생성하고 있었던 것이죠.

SiFive P550은 Out-of-Order 실행을 지원하는 CPU입니다. `llvm-mca`를 통해 분석한 결과, `fdiv.s`의 Latency는 19 cycle인 반면, `fdiv.d`는 무려 33 cycle이었습니다. 컴파일러가 float으로 처리해도 될 연산을 굳이 double로 승격시켜 처리하면서 병목이 발생한 것입니다. 

흥미로운 점은 과거 버전의 LLVM에서는 정상적으로 `fdiv.s`를 생성했다는 사실입니다. 즉, 최근 누군가 추가한 패치가 이 최적화를 망가뜨린 **Regression** 이었습니다.

## 원인: 너무 똑똑해진 InstCombine

문제의 C 코드는 다음과 같았습니다.

```c
ixran = iprod - 74383*(long int)(iprod/74383);
x = (float)ixran / 74383.0;
```

여기서 `74383.0`은 C 언어 스펙상 `double` 리터럴입니다. 하지만 `ixran`의 최대값은 74383이므로, 굳이 double로 계산하지 않고 float으로 처리해도 정밀도 손실이 없습니다. 과거의 LLVM은 이를 정확히 파악하고 나눗셈을 float으로 좁히는(Narrowing) 최적화를 수행했습니다.

과거 LLVM IR의 파이프라인 진행 과정은 대략 이랬습니다.

1. `sitofp` (int -> float)
2. `fpext` (float -> double) : 74383.0과 연산하기 위해 승격
3. `fdiv` (double 나눗셈)
4. `fptrunc` (double -> float) : 결과값을 x에 넣기 위해 강등

여기서 최적화 패스인 `visitFPTrunc`가 등장해 "어? 어차피 float으로 다시 자를 거면, 그냥 처음부터 float 나눗셈을 하자!" 라며 `fpext`와 `fptrunc`를 날려버리는 최적화를 수행했습니다.

**그런데 최근 추가된 패치가 이 평화를 깨뜨렸습니다.**

누군가 `InstCombine` 패스의 `isKnownExactCastIntToFP` 함수를 개선했습니다. 정수를 FP로 캐스팅할 때 비트 분석을 더 정교하게 하도록 만든 훌륭한 패치였죠. 이 패치 덕분에 컴파일러는 앞단의 `sitofp`와 `fpext`를 합쳐서 단일 `uitofp` (int -> double) 명령어로 더 깔끔하게 접어버렸습니다(Fold).

결과적으로 IR은 이렇게 변했습니다.

```llvm
%conv2 = uitofp nneg i64 %0 to double
%div3 = fdiv double %conv2, 7.438300e+04
%conv4 = fptrunc double %div3 to float
```

코드는 더 짧아졌지만, 치명적인 문제가 생겼습니다. 하위 패스인 `visitFPTrunc`는 앞단에 `fpext`가 있어야만 동작하는 패턴 매칭 로직을 가지고 있었습니다. `fpext`가 사라지자 컴파일러는 "이거 float으로 좁혀도 되나?" 판단할 근거를 잃었고, 결국 무거운 `fdiv.d`를 그대로 남겨둔 것입니다. 

솔직히 말해서, 이런 류의 버그는 컴파일러 개발자들의 악몽입니다. A라는 패스를 개선했더니, 전혀 상관없어 보이던 B 패스의 전제조건이 깨지면서 타겟 아키텍처의 성능이 박살나는 전형적인 나비효과죠.

## 우아한 해결책과 오픈소스 생태계

Kaving은 이 문제를 해결하기 위해 `isKnownExactCastIntToFP`를 리팩토링했습니다. 연산의 범위를 분석하는 핵심 로직을 `canBeCastedExactlyIntToFP`라는 별도 함수로 분리하고, `getMinimumFPType`에서 이를 호출하도록 만들었습니다. 

즉, 앞단에 `fpext`가 없더라도, 원본 정수값(`Src`)의 범위를 직접 분석하여 "이 값은 float에 구겨 넣어도 정보 손실이 없다"는 것을 컴파일러가 스스로 증명하게 만든 것입니다.

결과는 대성공이었습니다. 다시 `fdiv.s`가 생성되었고, 벤치마크 사이클은 25%나 개선되었습니다.



## Hacker News의 논쟁: 프로 vs 아마추어

이 글이 Hacker News에 올라오자 기술적인 내용만큼이나 흥미로운 토론이 벌어졌습니다. 누군가 "이런 고도의 작업을 무료로 해서는 안 된다"라고 댓글을 달자, 엄청난 논쟁이 불붙었죠.

- **의견 A:** "무료로 일하는 사람들은 순수한 즐거움 때문에 한다. 돈을 받고 일하는 사람은 이들만큼 잘할 가능성이 낮다."
- **의견 B:** "아니다. 프로페셔널은 풀타임으로 집중할 수 있기 때문에 아마추어보다 더 뛰어나다."

시니어 엔지니어로서 제 생각을 덧붙이자면, **소프트웨어 엔지니어링은 '프로'와 '아마추어'의 경계가 가장 모호한 분야** 입니다. 저 역시 회사에서 수억 달러짜리 인프라를 만지지만, 주말에 잠을 줄여가며 오픈소스 커널 코드를 뜯어볼 때 가장 큰 희열과 성장을 느낍니다. 

돈을 받고 하는 일은 필연적으로 '일정'과 '비즈니스 가치'와 타협하게 됩니다. Jira 티켓에 "컴파일러 파이프라인의 엣지 케이스를 분석해 14 cycle을 줄이시오"라는 업무가 할당될 확률은 0에 가깝습니다. 이런 집요한 최적화는 99% 순수한 호기심과 완벽주의를 가진 '덕후'들에 의해 이루어집니다.

하지만 동시에, 현대의 거대한 IT 인프라가 이런 자발적인 '무료 노동'에 크게 의존하고 있다는 사실은 다소 서늘하게 다가옵니다. 우리가 매일 사용하는 수많은 툴체인들이 누군가의 주말 헌신으로 지탱되고 있으니까요.

## 총평 (Verdict)

이번 사례는 RISC-V 생태계가 마주한 현재 진행형의 과제를 잘 보여줍니다. 하드웨어 스펙은 훌륭하게 발전하고 있지만, 이를 뒷받침할 소프트웨어 툴체인(특히 LLVM 백엔드)은 여전히 수많은 튜닝을 필요로 합니다. 

코드 한 줄의 변경이 컴파일러 파이프라인을 타고 내려가 CPU의 파이프라인 스톨(Stall)을 유발하는 과정을 추적하는 것은 언제 봐도 짜릿합니다. Kaving의 뛰어난 분석과 기여에 박수를 보냅니다. 여러분도 오늘 작성한 코드에 불필요한 캐스팅이 숨어있지는 않은지, 컴파일러가 내 코드를 어떻게 해석하고 있는지 한 번쯤 어셈블리를 열어 확인해 보시는 건 어떨까요?
