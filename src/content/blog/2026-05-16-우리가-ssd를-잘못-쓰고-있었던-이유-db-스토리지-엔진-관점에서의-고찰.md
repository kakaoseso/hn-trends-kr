최근 Hacker News를 뜨겁게 달군 VLDB 논문 한 편이 있습니다. 제목부터 도발적인 "How to Write to SSDs"입니다. PDF 링크를 클릭했다가 무의식적으로 내 SSD에 다운로드(쓰기)를 해버렸다는 한 HN 유저의 농담 섞인 댓글에 피식 웃음이 났지만, 논문이 던지는 메시지는 결코 가볍지 않습니다.

주니어 시절, 저를 포함한 많은 엔지니어들이 "서버에 NVMe SSD를 달았으니 이제 I/O 병목은 끝났다"고 착각하곤 합니다. 하지만 프로덕션 환경에서 트래픽이 몰릴 때 발생하는 원인 모를 Latency 스파이크와 급격히 깎여나가는 SSD 수명을 마주하고 나면, 우리가 SSD라는 매체를 얼마나 무지하게 다루고 있었는지 깨닫게 됩니다.

오늘은 15년 차 엔지니어의 시선에서, 이 논문이 짚고 있는 핵심과 우리가 데이터베이스를 설계하고 튜닝할 때 반드시 알아야 할 SSD의 민낯을 파헤쳐 보겠습니다.

### SSD는 거대한 거짓말 위에 서 있다

애플리케이션이나 OS 입장에서 SSD는 단순히 '아주 빠른 하드디스크'처럼 보입니다. 하지만 물리적 계층으로 내려가면 SSD는 HDD와 전혀 다른 생물입니다. 핵심은 덮어쓰기(Overwrite)가 불가능하다는 점입니다.

- **Page와 Block:** SSD는 데이터를 읽고 쓸 때는 Page 단위(보통 4KB~16KB)로 처리하지만, 데이터를 지울 때는 Block 단위(수 MB)로만 처리할 수 있습니다.
- **FTL (Flash Translation Layer):** 이 복잡한 제약을 OS로부터 숨기기 위해 SSD 컨트롤러 내부에 존재하는 거대한 맵핑 소프트웨어입니다.
- **WAF (Write Amplification Factor):** 덮어쓰기가 안 되기 때문에, 기존 데이터를 유지한 채 새로운 데이터를 빈 공간에 쓰고 이전 데이터를 '무효(Invalid)' 처리합니다. 찌꺼기가 쌓이면 Garbage Collection(GC)이 도는데, 이때 살아남은 데이터를 다른 Block으로 옮겨 적는 과정에서 내가 의도한 것보다 훨씬 많은 물리적 쓰기가 발생합니다.

즉, 우리가 아무 생각 없이 Random Write를 날리면, 내부에서는 FTL이 미친 듯이 맵핑 테이블을 업데이트하고 GC를 돌리느라 SSD의 수명(TBW)을 갉아먹고 Throughput을 스스로 깎아내립니다.

### B-Tree의 한계와 LSM-Tree의 부상

전통적인 RDBMS(PostgreSQL, MySQL InnoDB)는 HDD 시대에 설계된 B-Tree 구조를 사용합니다. B-Tree는 본질적으로 In-place update(제자리 덮어쓰기)를 수행하며, 이는 SSD에 끔찍한 Random Write 패턴을 만들어냅니다.

```cpp
// B-Tree 스타일의 업데이트 (SSD에 쥐약인 패턴)
void updateRecord(int id, Data data) {
    Node* node = findNode(id);
    node->data = data; 
    writeToDisk(node->offset, node); // Random overwrite!
}

// LSM-Tree 스타일의 업데이트 (SSD 친화적 패턴)
void updateRecord(int id, Data data) {
    LogEntry entry = {id, data};
    appendToFile(currentLogFile, entry); // Sequential append!
}
```

이 문제를 해결하기 위해 등장한 것이 RocksDB나 Cassandra 같은 시스템에서 사용하는 Log-Structured Merge (LSM) Tree입니다. 모든 쓰기를 메모리에 모았다가 디스크에 Sequential하게 내려쓰는(Append-only) 방식이죠. FTL 입장에서는 연속된 Page를 순차적으로 채워나가므로 GC 오버헤드가 극단적으로 줄어듭니다.

### Hacker News의 날카로운 통찰: SMR과 SSD의 평행이론

HN 스레드에서 한 유저가 남긴 댓글이 제 눈길을 끌었습니다. "SMR 하드 드라이브와 SSD의 접근 방식이 얼마나 겹칠지 궁금하다"는 내용이었죠.

정확한 통찰입니다. SMR(Shingled Magnetic Recording) 드라이브는 디스크 트랙을 기와 지붕처럼 겹쳐서 용량을 늘린 HDD입니다. 이 구조 때문에 중간에 있는 데이터를 수정하려면 겹쳐진 다른 트랙까지 다 다시 써야 합니다. 사실상 Random Write가 금지되고 대용량 Sequential Write가 강제되죠. 

물리적 매체(NAND 플래시 vs 마그네틱 플래터)는 전혀 다르지만, 소프트웨어 레벨에서 이를 극복하기 위한 아키텍처(Append-only, Log-structured)는 완벽하게 수렴하고 있습니다.

### Principal Engineer의 시선: 추상화의 붕괴와 ZNS의 등장

솔직히 말해, 저는 현재의 표준 블록 디바이스 인터페이스가 수명을 다해간다고 봅니다. 

우리는 지난 십수 년간 SSD 벤더들이 펌웨어(FTL) 안에 숨겨둔 블랙박스 알고리즘을 리버스 엔지니어링하듯 추측하며 DB 엔진을 최적화해 왔습니다. "이 벤더의 SSD는 GC가 이때쯤 도니까, DB 레벨의 Compaction을 이렇게 튜닝하자" 같은 기형적인 노하우가 난무했죠.

최근 클라우드 하이퍼스케일러들이 ZNS (Zoned Namespaces) SSD에 열광하는 이유가 여기에 있습니다. ZNS는 FTL의 복잡한 맵핑과 GC를 걷어내고, Host(OS나 DB)가 직접 데이터를 순차적으로 기록할 수 있는 'Zone'을 노출합니다. DB 엔진이 스토리지의 물리적 특성을 완벽히 통제하게 되는 것입니다. HN의 다른 유저가 "이런 연구가 새로운 형태의 DB를 만들 것"이라고 말한 것은 결코 과장이 아닙니다.

### 결론: 우리는 무엇을 해야 하는가?

일반적인 웹 백엔드 개발자라면 당장 이 논문을 읽고 비즈니스 로직을 뜯어고칠 필요는 없을 것입니다. 하지만 여러분이 다루는 Postgres, Kafka, Redis 내부에서 데이터가 어떻게 디스크에 내려앉는지 이해하는 것은, 트래픽이 폭주하여 시스템이 비명을 지를 때 아키텍처의 병목을 찾아내는 강력한 무기가 됩니다.

데이터베이스는 마법이 아닙니다. 물리적 한계를 소프트웨어의 기교로 덮어놓은 거대한 타협의 산물일 뿐입니다. 그 타협의 경계를 이해하는 순간, 여러분의 엔지니어링 레벨은 한 단계 도약할 것입니다.
