요즘 우리는 화면에 'Hello World' 텍스트 하나를 띄우기 위해 수십, 수백 메가바이트의 메모리를 소모하는 시대에 살고 있다. Electron이나 거대한 웹 프레임워크 기반의 애플리케이션에 익숙해진 우리에게, 단 16바이트의 코드만으로 화면에 프랙탈 기하학을 렌더링하고 동시에 오디오까지 출력하는 프로그램이 있다면 믿겠는가?

솔직히 처음 해커뉴스(Hacker News)에서 이 글의 제목을 봤을 때는 흔한 어그로성 낚시글이려니 했다. 하지만 첨부된 8줄의 x86 어셈블리 코드를 분석하면서, 나는 오랫동안 잊고 있었던 '플랫폼에 대한 완벽한 이해'가 무엇인지 다시금 깨닫게 되었다. 

이 글에서는 2026년 Outline Demoparty에서 공개된 'wake up! 16b'라는 데모씬(Demoscene) 작품이 어떻게 단 16바이트로 시각적인 매트릭스 레인(Matrix rain) 효과와 사운드를 동시에 만들어내는지, 그 미친 '알고리즘 밀도(Algorithmic density)'를 엔지니어의 시각에서 파헤쳐보려 한다.

### 16바이트의 마법: 코드 분석

먼저 문제의 코드를 살펴보자. MS-DOS 환경에서 구동되는 16비트 리얼 모드(Real-mode) 어셈블리다.

```asm
int 10h          ; 2 bytes - Init Video Mode 0
mov bh, 0xb8     ; 2 bytes - Setup VRAM segment
mov ds, bx       ; 2 bytes
L:
lodsb            ; 1 byte  - Load [SI] to AL, inc SI
sub si, byte 57  ; 3 bytes - Move pointer backward
xor [si], al     ; 2 bytes - Cellular Automaton
out 61h, al      ; 2 bytes - PC Speaker output
jmp short L      ; 2 bytes - Infinite loop
```

단 16바이트. 우리가 흔히 쓰는 UUID 문자열 길이보다 짧은 이 코드가 어떻게 시각과 청각을 동시에 자극할 수 있을까? 핵심은 하드웨어의 물리적 특성을 극한까지 쥐어짜낸 **세 가지 트릭** 에 있다.

#### 1. 비디오 메모리(VRAM)를 연산 캔버스로 사용
첫 줄의 `int 10h`는 BIOS 인터럽트를 호출해 40x25 텍스트 모드(Video Mode 0)를 초기화한다. 이후 `0xB800`이라는 물리적 메모리 주소를 데이터 세그먼트(DS)로 설정한다. 이곳은 VGA/CGA 텍스트 버퍼가 위치한 곳이다.

흥미로운 점은 BIOS가 화면을 지울 때 메모리를 절대적인 '0'으로 채우지 않는다는 것이다. 텍스트 모드에서 각 칸은 2바이트(ASCII 문자 + 색상 속성)를 차지하는데, 초기화 시 ASCII 바이트는 `0x20`(공백 문자), 색상 바이트는 `0x07`(검은 배경에 밝은 회색 텍스트)로 설정된다. 화면은 비어있는 것처럼 보이지만, 실제로는 데이터로 균일하게 채워진 완벽한 캔버스가 준비된 것이다.

#### 2. XOR 연산과 셀룰러 오토마타 (Rule 60)
루프 안의 핵심 연산은 `xor [si], al`이다. 만약 이 연산이 `add`였다면 단순한 이항 누적합(Binomial prefix sum)이 되었을 것이다. 하지만 덧셈 대신 배타적 논리합(XOR)을 사용함으로써, 산술적인 자리올림(Carry)이 무시되고 비트 평면(Bit-plane)이 분리된다.

이는 수학적으로 스티븐 울프럼(Stephen Wolfram)의 기본 셀룰러 오토마타 중 **Rule 60** 과 정확히 일치한다. 초기값이 주어진 상태에서 XOR 연산이 누적되면, 메모리 상에는 자연스럽게 시어핀스키 삼각형(Sierpinski triangle) 형태의 프랙탈 패턴이 그려진다. 

#### 3. PC 스피커 포트(61h)의 기막힌 우연
개인적으로 이 코드에서 가장 경악했던 부분은 `out 61h, al` 명령어다. 포트 `61h`는 전통적으로 PC 내부 스피커를 제어하는 하드웨어 포트다. 이 포트의 **Bit 1** 이 1로 설정되면 스피커 콘이 밀려나고, 0이 되면 제자리로 돌아온다.

앞서 만든 프랙탈 패턴 데이터가 담긴 `AL` 레지스터의 값을 그대로 스피커 포트에 쏴버리는 것이다. 프랙탈 패턴의 Bit 1이 0과 1을 오가며 스피커 콘을 움직이고, 이것이 구형파(Square wave) 형태의 오디오 신호가 된다. 

- **밀도가 높은 구간:** Bit 1이 빠르게 교차하며 고주파(High-frequency) 사운드를 낸다.
- **밀도가 낮은 구간 (프랙탈의 빈 공간):** Bit 1이 0으로 유지되며 리드미컬한 휴지기(Rest)를 만든다.

시각 데이터를 그대로 오디오 포트에 밀어 넣었는데, 하드웨어가 크래시 나지 않고 오히려 프랙탈의 기하학적 구조가 사운드의 리듬으로 변환되는 기적 같은 일이 벌어진 것이다.

### 15년 차 엔지니어의 시선: 우리는 무엇을 잃어버렸나?

루프 내의 `sub si, byte 57`과 `lodsb`의 조합은 포인터를 매번 -56바이트씩 역방향으로 이동시킨다. 56은 64KB 세그먼트를 딱 떨어지게 나누지 못하므로, 전체 메모리를 순회하려면 7번의 랩어라운드(Wrap-around)를 거치며 8,192번의 루프를 돌아야 한다. 

이 기발한 수학적 어긋남은 두 가지 결과를 낳는다. 첫째, 매크로 사이클이 길어지면서 오디오 주파수가 정확히 한 옥타브 낮아져 더 깊고 묵직한 소리가 난다. 둘째, 화면에 렌더링되는 프랙탈이 대각선으로 전단(Shearing)되어, 마치 영화 매트릭스의 '디지털 비(Matrix Rain)'처럼 10개의 수직 기둥이 상승하는 듯한 시각적 효과를 만들어낸다.

현업에서 마이크로서비스 아키텍처(MSA)를 설계하고 클라우드 인프라를 다루는 내 입장에서, 이런 종류의 코드는 사실 '유지보수 불가능한 레거시의 극치'다. BIOS의 초기화 상태에 의존하고, 하드웨어 포트에 쓰레기 값(Bit 1을 제외한 나머지 비트들)을 무작정 던져버리는 이 코드는 철저하게 Undefined Behavior에 기대고 있다. 실제로 해커뉴스 댓글에서도 언급되었듯, 에뮬레이터(DOSBox vs PCem)나 실제 하드웨어의 종류에 따라 화면에 깨져 나오는 ASCII 글리치 텍스처와 사운드의 음색이 미묘하게 달라진다.

하지만 이 코드가 주는 울림은 강력하다. 오늘날 우리 소프트웨어 엔지니어들은 추상화(Abstraction)라는 두꺼운 담요 밑에 숨어 있다. 우리는 가비지 컬렉터가 메모리를 치워주고, OS가 하드웨어를 제어하며, 오케스트레이션 툴이 컨테이너를 관리해 주는 환경에 익숙하다. 

이 16바이트 코드는 개발자가 **'자신이 딛고 서 있는 하드웨어 플랫폼을 바닥 끝까지 이해했을 때'** 어떤 창의적인 결과물이 나올 수 있는지를 보여주는 예술 작품이다. 

### 결론: 실용성은 제로, 영감은 무한대

이 코드를 프로덕션에 쓸 수 있을까? 절대 아니다. 당장 코드 리뷰에서 반려될 것이다.

그러나 기술 블로거이자 엔지니어로서 나는 이런 데모씬 작품들을 볼 때마다 깊은 경외감을 느낀다. 자원 제약이 창의성을 극대화한다는 옛말은 여전히 유효하다. 무한한 컴퓨팅 리소스가 주어지는 클라우드 시대에, 가끔은 우리가 작성하는 수만 줄의 코드가 정말로 그만큼의 가치를 하고 있는지 되돌아볼 필요가 있다.

이번 주말에는 매일 다루던 고수준 프레임워크에서 잠시 벗어나, C나 어셈블리로 로우레벨 메모리를 직접 만져보는 장난을 쳐보는 것은 어떨까? 어쩌면 잊고 있던 엔지니어링의 원초적인 재미를 다시 느낄 수 있을지도 모른다.
