비트코인이 처음 등장했을 때, 우리 모두는 '1 CPU = 1 Vote'라는 낭만적인 비전을 믿었습니다. 하지만 현실은 냉혹했죠. SHA-256 연산만을 미친 듯이 병렬로 처리하는 고정 기능(fixed-function) 실리콘, 즉 ASIC이 등장하면서 채굴판은 소수의 거대 자본과 하드웨어 제조사들의 놀이터로 전락했습니다.

엔지니어로서 저는 지난 10년간 수많은 프로젝트가 'ASIC 저항성(ASIC-resistance)'을 외치며 등장했다가 처참하게 실패하는 것을 지켜봤습니다. 알고리즘을 조금 꼬아봤자, 하드웨어 엔지니어들은 금세 그 패턴을 분석해 더 효율적인 칩을 찍어냈으니까요. 솔직히 말해서 저는 ASIC 저항성이라는 단어 자체를 일종의 마케팅 불릿(BS)으로 취급해 왔습니다.

하지만 모네로(Monero)의 **RandomX** 알고리즘은 다릅니다. 이들은 암호학적 기교로 하드웨어와 싸우는 대신, **하드웨어 경제학의 근본적인 룰** 을 바꿔버렸습니다. ASIC을 막고 싶다고요? 그럼 채굴 알고리즘 자체를 '일반적인 CPU가 처리하는 난잡한 범용 워크로드'로 만들어버리면 됩니다.

오늘은 모네로가 어떻게 RandomX를 통해 이 미친 아이디어를 프로덕션 레벨에서 구현해냈는지, 그리고 그 이면에 숨겨진 엔지니어링적 결단들을 파헤쳐보겠습니다.

## 해시 함수가 아니라 가상 머신(VM)을 돌린다

비트코인 채굴은 단순합니다. 특정 조건을 만족할 때까지 SHA-256 파이프라인에 데이터를 밀어 넣는 무한 루프죠. 하드웨어 설계자 입장에선 너무나 명확한 타겟입니다. 분기(Branch)도 없고, 복잡한 메모리 접근도 없으며, 상태(State)를 유지할 필요도 없습니다.

반면 모네로의 RandomX는 채굴자에게 고정된 해시 함수를 요구하지 않습니다. 대신 **무작위로 생성된 프로그램을 특수한 가상 머신(VM) 위에서 실행** 하도록 강제합니다. 

이 과정은 크게 두 가지 환경 변수에 의해 통제됩니다:
- **K (Key):** 약 2.8일(2048 블록)마다 변경되는 중기적인 환경 변수.
- **H (Hash Input):** 채굴자가 계속해서 논스(Nonce)를 바꿔가며 시도하는 단기 입력값.

### 1. 메모리 병목의 이중화: Dataset과 Scratchpad

RandomX는 CPU의 메모리 계층 구조를 집요하게 괴롭힙니다. 

먼저 $K$ 값을 기반으로 Argon2d를 돌려 256 MiB의 캐시를 만들고, 이를 다시 팽창시켜 약 **2 GiB 크기의 Dataset** 을 생성합니다. 2GB라는 크기는 의도적입니다. 최신 CPU의 L3 캐시를 훌쩍 넘어서기 때문에, 채굴 기기는 무조건 느린 메인 메모리(DRAM)에 접근해야만 합니다. 여기서 메모리 대역폭(Bandwidth) 병목이 발생합니다.

동시에 $H$ 값을 이용해 **2 MiB 크기의 Scratchpad** 를 만듭니다. 이 녀석은 VM이 실행되는 동안 작업 메모리로 사용되는데, 16 KiB, 256 KiB, 2 MiB 단위로 나뉘어 접근됩니다. 눈치채셨나요? 정확히 현대 CPU의 L1, L2, L3 캐시 사이즈를 모방한 것입니다. 

즉, RandomX는 거대한 Dataset으로 DRAM 트래픽을 유발하고, 작은 Scratchpad로 CPU 내부 캐시를 한계까지 긁어댑니다. 커스텀 칩 설계자 입장에선 벌써부터 머리가 아파오는 구조입니다.

### 2. 난잡한 명령어와 분기 예측(Branch Prediction) 파괴

Scratchpad가 준비되면, RandomX는 256개의 명령어로 이루어진 무작위 프로그램을 8개 생성하여 체인으로 연결합니다. 

이 프로그램들은 단순한 정수 연산만 하지 않습니다. 64-bit 곱셈, 128-bit 벡터 연산, 그리고 **IEEE 754 규격을 완벽히 따르는 Double-precision 부동소수점(Floating-point) 연산** 까지 포함합니다. ASIC에 부동소수점 연산 유닛(FPU)을, 그것도 4가지 반올림 모드를 모두 지원하도록 때려 박는 것은 면적과 전력 소모 측면에서 엄청난 낭비입니다. 

여기에 결정타를 날리는 것이 바로 **분기(Branch)** 입니다. 

```text
// RandomX VM Execution Loop (Conceptual)
for (int i = 0; i < 2048; i++) {
    read_write(scratchpad);
    prefetch_load(dataset);
    mix_state(int_registers, float_registers);
    
    if (random_condition == TRUE) { // ~ 1/256 probability
        jump_to_branch(); // CPU Branch Predictors hate this
    }
}
```

코드의 흐름이 일직선이면 하드웨어 파이프라인을 최적화하기 쉽습니다. 하지만 RandomX는 1/256 확률로 실행되는 분기문을 삽입합니다. 대부분은 'Not Taken'으로 예측되도록 설계되어 일반적인 CPU의 분기 예측기(Branch Predictor)에서는 꽤 저렴하게 처리되지만, 정적 분석으로 파이프라인을 깎아내는 ASIC에게는 치명적인 오버헤드를 유발합니다.

### 3. 왜 8개의 프로그램을 체이닝(Chaining)하는가?

이 부분이 제가 RandomX 설계에서 가장 우아하다고 생각하는 대목입니다. 

만약 1개의 프로그램만 실행한다면, 영악한 채굴자는 프로그램을 먼저 분석해서 '자신의 커스텀 하드웨어에서 빨리 도는 쉬운 프로그램'만 골라서 실행(Cherry-picking)할 수 있습니다. 

하지만 RandomX는 8개의 프로그램을 직렬로 연결합니다. 이전 프로그램의 결과 상태(State)가 다음 프로그램의 시드(Seed)가 되기 때문에, 끝까지 돌려보기 전에는 다음 코드가 얼마나 '어려울지' 알 수 없습니다. 중간에 포기하면 그때까지 쓴 전기와 시간은 그대로 매몰 비용이 됩니다. 악의적인 최적화를 게임 이론으로 완벽하게 틀어막은 셈입니다.

## Fast Mode vs Light Mode: 엔지니어링 트레이드오프의 정석

채굴에 2GB 이상의 메모리가 필요하다면, 블록을 검증(Verification)하는 일반 노드들도 2GB를 써야 할까요? 만약 그렇다면 네트워크 참여율이 박살 날 것입니다.

RandomX는 이를 두 가지 모드로 우아하게 해결했습니다.
- **Fast Mode (채굴용):** 2GB Dataset을 미리 메모리에 올려두고 최고 속도로 해싱합니다.
- **Light Mode (검증용):** 256MB 캐시만 유지하고, Dataset의 데이터가 필요할 때마다 실시간(On-the-fly)으로 계산합니다.

결과값은 동일합니다. 검증자는 메모리를 아끼는 대신 CPU 연산을 조금 더 지불하면 됩니다. 반면, 채굴자가 메모리를 아끼려고 Light Mode로 채굴을 시도하면 연산 오버헤드 때문에 절대 경쟁력을 가질 수 없습니다. 시스템 설계 시 '메모리-시간 트레이드오프(Memory-Time Tradeoff)'를 어떻게 활용해야 하는지 보여주는 훌륭한 교과서적 사례입니다.

## Hacker News의 반응: 현실 세계의 피드백

이 아키텍처에 대한 Hacker News 커뮤니티의 반응과 논의도 매우 흥미롭습니다. 몇 가지 주목할 만한 포인트를 짚어보겠습니다.

1. **가장 완벽한 ASIC은 결국 CPU다:** 
   한 유저가 언급했듯, Bitmain이 RandomX용으로 만든 X9 채굴기는 까놓고 보니 **그냥 RISC-V CPU 코어들을 잔뜩 때려 박은 물건** 이었습니다. "RandomX ASIC을 만들고 싶다면, 그냥 CPU를 새로 설계해라"라는 RandomX 개발팀의 호언장담이 현실로 증명된 셈입니다.

2. **노드 운영의 현실적인 문제들 (LMDB Corruption):**
   기술적으로 훌륭한 알고리즘과 별개로, 공식 클라이언트(monerod)의 안정성에 대한 불만도 있었습니다. 프로세스가 강제 종료될 경우 블록체인 DB(LMDB)가 깨지는 현상 때문에 `--db-sync-mode safe` 플래그 사용이 권장된다는 실무적인 팁이 오갔습니다. (SSD 수명을 갉아먹는다는 불평은 덤입니다.)

3. **난방기로 전락한(?) 구형 Xeon 서버들:**
   가장 재밌는 논의는 채굴의 경제성이었습니다. 추운 지역에 사는 유저들은 남는 구형 Xeon 서버로 모네로를 채굴하며 집을 난방하고 있었습니다. 전기 히터(Resistive heating)를 켤 바에야, 똑같이 열을 내면서 모네로까지 캐주는 CPU를 굴리는 게 낫다는 논리죠. (물론 열역학적으로 히트펌프가 낫다는 엔지니어들의 팩트 폭행이 이어지긴 했습니다.)

## 맺음말 (Verdict): PoW는 죽지 않았다

이더리움이 PoS(Proof of Stake)로 성공적으로 전환하면서, 많은 사람들이 PoW의 시대는 끝났다고 말합니다. 스마트 컨트랙트 플랫폼이나 디파이(DeFi) 생태계에서는 확실히 PoS가 더 합리적인 선택일 수 있습니다.

하지만 모네로처럼 '순수한 탈중앙화 익명 화폐'를 지향하거나, Tor 네트워크의 Equi-X(RandomX 파생 알고리즘)처럼 네트워크 스팸을 방어해야 하는 환경에서는 여전히 PoW가 강력한 무기입니다. 

솔직히 저는 여전히 암호화폐 씬에 껴있는 거품을 좋아하지 않습니다. 하지만 RandomX의 아키텍처 문서를 읽고 있으면, 이들이 하드웨어 종속성을 끊어내기 위해 얼마나 깊게 고민하고 훌륭한 엔지니어링을 해냈는지 감탄하게 됩니다. 

결론적으로, RandomX는 단순한 암호학적 트릭이 아닙니다. 컴파일러, 메모리 계층, 컴퓨터 아키텍처, 그리고 경제학까지 모두 융합된 **하나의 거대한 하드웨어 심리전** 입니다. 그리고 현재까지는, 소프트웨어 엔지니어들이 실리콘 벤더들을 상대로 꽤나 성공적인 방어전을 치르고 있는 것 같습니다.
