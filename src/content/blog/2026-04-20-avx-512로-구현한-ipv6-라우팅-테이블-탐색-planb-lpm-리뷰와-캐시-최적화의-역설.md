네트워크 엔지니어링이나 고성능 패킷 처리 시스템을 다뤄본 사람이라면, 라우팅 테이블에서 Longest-Prefix-Match (LPM)를 수행하는 것이 얼마나 까다로운 작업인지 잘 알 것이다. IPv4 시절에는 메모리를 때려 박아서 해결하거나 단순한 Trie 구조로도 충분했지만, IPv6의 128비트 주소 체계로 넘어오면서 상황은 완전히 달라졌다.

최근 Hacker News에서 눈길을 끄는 프로젝트를 하나 발견했다. NSDI '26에 발표될 예정인 "PlanB" 알고리즘을 C++17로 깔끔하게 재구현한 [planb-lpm](https://github.com/esutcu/planb-lpm)이다. 솔직히 말해서 처음에는 흔한 "학술 논문의 SIMD 최적화 장난감" 정도로 생각했다. 하지만 코드를 까보고 벤치마크 결과를 분석하면서, 이 프로젝트가 하드웨어 아키텍처와 실제 데이터 분포(Real-world data distribution)의 상관관계에 대해 아주 중요한 교훈을 던져주고 있다는 것을 깨달았다.

## PlanB LPM의 핵심 아키텍처

이 라이브러리의 핵심은 IPv6 프리픽스 구간을 1차원 배열로 펼친 뒤(Linearized), 이를 9-ary B+-tree로 구성하는 것이다. 왜 하필 9-ary일까? 바로 **Cache-line** 때문이다.

최신 CPU의 캐시 라인 크기는 보통 64바이트다. IPv6 주소의 상위 64비트(보통 라우팅에 사용되는 구간)를 키로 사용할 경우, 8개의 키를 하나의 노드에 담으면 정확히 64바이트 캐시 라인 하나에 정렬(Cache-aligned)된다.

트리를 탐색할 때 각 노드에서 자식 포인터를 찾기 위해 분기해야 하는데, 여기서 AVX-512 SIMD 명령어가 빛을 발한다. 단일 `vpcmpuq` 명령어와 `popcnt`를 조합해 8개의 키를 한 번에 비교하고 다음 자식 노드의 인덱스를 O(1)에 가깝게 찾아낸다.

```python
# Python 바인딩을 통해 제공되는 직관적인 인터페이스
from planb_lpm import Tree, Dynamic

d = Dynamic()
d.load([("::", 0, 1)])
d.insert("2001:db8::", 32, 7)
d.lookup("2001:db8::abcd") # -> 7
```

게다가 원본 논문의 레퍼런스 코드가 리눅스 종속적이고 라이선스도 없는 "전형적인 연구용 코드"였던 반면, 이 구현체는 `std::atomic<std::shared_ptr>`를 활용한 Rebuild-and-swap 모델을 도입해 Wait-free 읽기 경로를 제공한다. 2018년경 대규모 애드테크 플랫폼에서 설정(Config) 데이터를 핫 리로드할 때 우리가 즐겨 쓰던 RCU(Read-Copy-Update) 라이트 버전과 정확히 일치하는 패턴이다. 실무에 바로 투입해도 될 만큼 엔지니어링 완성도가 높다.

## 캐시 최적화의 역설: Real BGP vs Synthetic Data

내가 이 프로젝트에서 가장 감명 깊게 본 부분은 저자의 정직한 벤치마크 분석이다. 보통 이런 류의 프로젝트는 자신의 SIMD 구현체가 기존의 Pointer-chasing 방식(예: Patricia Trie)보다 압도적으로 빠르다는 점만 강조하기 마련이다. 실제로 균등하게 분포된 합성(Synthetic) 데이터셋에서는 PlanB 트리가 Patricia Trie보다 중간값 기준 약 20배나 빠른 Throughput을 보여주었다.

하지만 **실제 RIPE RIS BGP 라우팅 테이블(약 254K 프리픽스)** 을 넣고 테스트하자 충격적인 결과가 나왔다. Patricia Trie의 성능이 무려 40배나 폭증하면서, 오히려 AVX-512를 떡칠한 PlanB 트리와 비등하거나 근소하게 앞서는 결과가 나온 것이다.

이유가 뭘까? 답은 **데이터의 공간적 지역성(Spatial Locality)** 과 **캐시 히트율** 에 있다.

실제 인터넷의 BGP 라우팅 테이블은 주소 공간에 균등하게 퍼져 있지 않다. 특정 짧은 프리픽스(/6, /8, /12 등)에 거대한 트래픽이 몰려 있다. 랜덤한 IPv6 주소를 던졌을 때, Patricia Trie는 1~2번의 포인터 추적(Pointer-chasing)만으로 매칭을 끝내고 L1 캐시 안에서 조기 종료(Early exit)해버린다. 반면, PlanB의 B+-tree는 트리의 깊이가 7로 고정되어 있기 때문에, 최종 승자가 누구든 무조건 7번의 메모리 페치(DRAM 또는 L3 캐시 접근)를 수행해야 한다.

이것은 시니어 엔지니어들이 항상 명심해야 할 아주 고전적인 교훈이다. 알고리즘의 Big-O 표기법이 아무리 훌륭해도, 상수항(Constant factor)이 메모리 병목(Memory bound)에 걸려 있고 실제 데이터 분포가 조기 종료를 허용한다면 멍청해 보이는 Naive 알고리즘이 SIMD 최적화를 이길 수 있다.

## Hacker News의 반응과 실무적 관점

이 글이 HN에 올라왔을 때 커뮤니티의 반응도 꽤 흥미로웠다.

- **RISC-V Vector 명령어 적용 가능성:** 한 유저는 이 알고리즘이 RISC-V의 가변 길이 벡터 확장(RVV)에서는 어떻게 동작할지 궁금해했다. 고정된 64바이트 캐시 라인에 의존하는 AVX-512와 달리, RVV에서는 노드 크기를 동적으로 가져갈 수 있어 아키텍처 설계가 완전히 달라질 수 있다.
- **Linux Kernel FIB와의 비교:** 커널의 라우팅 테이블과 비교하면 어떨까 하는 질문도 있었다. 커널은 극단적인 RCU 오버헤드와 락킹 메커니즘을 동반하기 때문에 유저스페이스의 순수 알고리즘 벤치마크와 직접 비교하긴 어렵다. 하지만 SDN 컨트롤러나 패킷 분석 파이프라인 같은 유저스페이스 툴링에는 이 라이브러리가 당장 훌륭한 대안이 될 수 있다.
- **빌드 시스템 논쟁:** CMake에서 컴파일러 플래그로 AVX-512를 감지하는 방식 대신 소스 코드 내에서 `#ifdef`를 써야 한다는 의견도 있었는데, 개인적으로는 패키징과 크로스 컴파일을 고려할 때 저자의 CMake 방식이 더 현대적인 C++ 프로젝트에 적합하다고 본다.

## 결론 (Verdict)

PlanB-LPM은 단순히 "AVX-512로 IPv6 조회를 빠르게 만들었다"는 수준을 넘어선다. 학계의 아이디어를 실무 레벨의 C++17 코드로 끌어올린 훌륭한 엔지니어링 사례이며, 동시에 벤치마크가 얼마나 기만적일 수 있는지(그리고 어떻게 제대로 분석해야 하는지)를 보여주는 교과서적인 자료다.

당신의 팀이 대규모 네트워크 트래픽 분석기, SDN 컨트롤러, 혹은 커스텀 패킷 시뮬레이터를 개발 중이라면 이 라이브러리는 당장 도입을 검토해 볼 가치가 있다. 단, 도입 전에 **반드시 당신의 실제 프로덕션 트래픽(PCAP)으로 벤치마크를 돌려보라.** 당신의 데이터 분포에 따라서는 단순한 Patricia Trie가 여전히 최고의 선택일지도 모르니까.
