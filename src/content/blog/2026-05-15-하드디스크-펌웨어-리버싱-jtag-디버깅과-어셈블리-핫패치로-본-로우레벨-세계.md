우리 대부분은 하드디스크나 SSD를 그저 데이터를 넣고 빼는 완벽한 블랙박스로 취급한다. 시스템 아키텍처를 설계할 때도 IOPS, Throughput, Latency 같은 지표만 신경 쓸 뿐, 그 드라이브 내부에서 자체적인 RTOS와 수십 개의 ARM 코어가 돌아가고 있다는 사실은 잊기 마련이다.

최근 Xbox 360의 Race condition 취약점을 공략하기 위해 하드디스크 펌웨어를 직접 해킹한 Ryan Miceli의 글을 읽었다. 딜레이를 유발하기 위해 소프트웨어 레벨이 아닌 HDD 펌웨어 레벨을 뜯어고친 이 엄청난 수준의 Yak shaving은 엔지니어로서 묘한 쾌감마저 느끼게 한다. 이번 포스트에서는 이 과정에서 사용된 로우레벨 디버깅 기법과 펌웨어 리버싱 과정을 깊게 파헤쳐보고, 이것이 보안 업계에 던지는 메시지를 논해보려 한다.

## 펌웨어 추출과 얄팍한 난독화

모든 리버싱의 첫 단계는 바이너리를 확보하는 것이다. 저자는 PC-3000 같은 전문 복구 장비나 OEM 펌웨어 업데이트 유틸리티를 역분석하여 펌웨어 이미지를 얻어냈다.

흥미로운 점은 제조사마다 펌웨어를 보호(또는 난독화)하는 방식이다. Western Digital(WD)의 경우 LZHUF 압축 알고리즘을 사용했는데, 표준 알고리즘에서 N 상수와 Threshold 계산 방식을 살짝 비틀어 놓았다. ARM 부트로더의 압축 해제 루틴을 디스어셈블하여 이를 파악해낸 과정은 꽤나 정석적이다.

반면 Samsung SSD(PM871a)의 펌웨어 업데이트 유틸리티에서 추출한 복호화 코드는 실소를 자아낸다.

```c
void DecodeFirmware(unsigned char* pBuffer, unsigned int Length)
{
    for (unsigned int i = 0; i < Length; i++)
    {
        unsigned char nibbleHi = (pBuffer[i] >> 4) & 0xF;
        if ((nibbleHi & 1) != 0)
            nibbleHi >>= 1;
        else
            nibbleHi = 0xF - (nibbleHi >> 1);
        pBuffer[i] = (pBuffer[i] & 0xF) | (nibbleHi << 4);
    }
}
```

이것은 암호화라기보다는 단순한 비트 시프트 장난질이다. Hacker News 스레드에서도 한 유저가 날카롭게 지적했듯, 벤더들이 이런 수준 낮은 난독화를 고집하는 이유는 보안 때문이 아니다. 펌웨어가 유출되었을 때 DMCA(저작권법)를 무기로 법적 제재를 가하기 위한 핑계거리에 불과하다. Security through obscurity의 전형적인, 그리고 최악의 사례다.

## JTAG을 이용한 라이브 HDD 디버깅

가장 인상 깊었던 부분은 하드디스크를 실제로 디버깅하는 과정이다. 저자는 WD 드라이브 기판에 숨겨진 38핀 MICTOR 커넥터에 JTAG을 연결하고 OpenOCD를 통해 MCU에 직접 접근했다. 

단순히 연결만 한다고 끝나는 것이 아니다. 디버깅 중 브레이크포인트에 걸려 드라이브가 응답하지 않으면 Windows는 SATA 타임아웃을 발생시키고 심지어 BSOD를 띄운다. 이를 우회하며 디버깅을 진행한 집념은 정말 대단하다.

디버깅의 핵심은 **VSC** (Vendor Specific Commands)를 악용하는 것이었다. WD 드라이브는 `SMART WRITE LOG` (0xBE) 명령어를 통해 펌웨어나 RAM을 읽고 쓸 수 있는 백도어 인터페이스를 제공한다. 저자는 ATA Passthrough를 이용해 커스텀 VSC 패킷을 드라이브에 전송했다.

```c
// Setup port registers for SMART READ LOG command:
IDEREGS* pRegs = (IDEREGS*)pAtaPassthrough->CurrentTaskFile;
pRegs->bCommandReg = ATA_OP_SMART;
pRegs->bFeaturesReg = SMART_WRITE_LOG;
pRegs->bSectorCountReg = 1;
pRegs->bSectorNumberReg = 0xBE; // Special WD log address
```

메모리 주소에 하드웨어 브레이크포인트를 걸고 이 VSC 명령을 호출하여, 콜스택을 역추적하는 방식으로 ATA 명령어 처리 디스패처를 찾아냈다. 시스템 프로그래밍에서 막막한 블랙박스를 마주했을 때 사용할 수 있는 가장 우아한 Top-down 접근법이다.

## Overlay Modules와 어셈블리 핫패치

디스패처를 찾고 나서 마주한 또 다른 난관은, 실제 `DMA READ EXT` (섹터 읽기) 명령을 처리하는 코드가 펌웨어 이미지 내에 존재하지 않는다는 것이었다. 

하드디스크는 모든 펌웨어를 플래시 메모리에 담지 않는다. 핵심 부트로더만 플래시에 두고, 나머지 확장 기능이나 수리 모듈 등은 디스크 플래터의 숨겨진 영역인 Service Area에 **Overlay Modules** 형태로 저장해둔 뒤 부팅 시 RAM으로 로드한다. 읽기 처리 코드는 바로 이 Overlay Module 0x11에 들어있었다.

저자는 플래시를 덮어쓰는 위험한 방식 대신, 부팅이 완료된 후 RAM에 로드된 코드를 VSC로 직접 수정하는 핫패치(Hot-patch) 방식을 택했다. 읽기 요청을 가로채어 약 200ms 동안 CPU를 공회전시키는 Thumb 어셈블리 Code Cave를 작성했다.

```assembly
Hook_SataDmaRead:
    # Setup delay counter.
    ldr r3, =(MS_DELAY * F_CPU / 1000)
Hook_SataDmaRead_loop:
    # Spin until the counter is 0.
    sub r3, r3, #1
    bne Hook_SataDmaRead_loop
    bx lr
```

결과는 성공적이었다. 캐시 히트를 제외한 실제 물리적 읽기 작업에서 약 450ms의 지연을 만들어냈다. 하드웨어 레벨에서 특정 섹터의 Latency를 조작해버린 것이다.

## 결론: 우리는 하드웨어를 너무 신뢰한다

아이러니하게도, 저자가 이 모든 펌웨어 패치를 마치고 테스트를 진행했을 때, 패치를 적용하지 않은 순정 하드디스크에서도 Xbox 360 익스플로잇이 성공해버렸다. 타이밍 이슈가 우연히 맞아떨어진 것이다. 엔지니어링을 하다 보면 이런 허탈한 순간을 종종 겪지만, 그 과정에서 얻은 지식의 가치는 사라지지 않는다.

이번 사례는 단순한 해킹 튜토리얼이 아니다. 과거 NSA의 Equation Group이 하드디스크 펌웨어에 악성코드를 심어 영구적인 백도어를 구축했던 사건을 기억하는가? Hacker News 커뮤니티에서도 언급되었듯, 하드웨어 펌웨어는 여전히 보안의 거대한 사각지대다. Red Balloon Security 같은 곳에서 하드디스크 펌웨어 해킹을 채용 CTF로 사용하는 것도 같은 맥락이다.

솔직히 말해서, 일반적인 웹 백엔드 개발자나 인프라 엔지니어가 하드디스크 펌웨어를 뜯어볼 일은 없다. 하지만 우리가 의존하는 인프라의 가장 밑바닥이 얼마나 취약하고, 또 얼마나 복잡한 생태계로 이루어져 있는지 이해하는 것은 시스템을 설계하는 시야를 완전히 바꿔놓는다. 벤더가 제공하는 블랙박스를 맹신하지 마라. 그 안에는 언제든 조작 가능한 또 다른 작은 컴퓨터가 숨어있다.
