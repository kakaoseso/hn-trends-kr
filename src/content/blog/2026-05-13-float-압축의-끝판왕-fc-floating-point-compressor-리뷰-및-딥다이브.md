시계열 데이터나 과학/금융 시뮬레이션 데이터를 다뤄본 엔지니어라면, 무한정 쌓이는 64-bit float 데이터의 저장 비용에 골머리를 앓아본 적이 있을 것이다. 보통은 생각 없이 zstd나 lz4를 던져넣고 "이 정도면 됐지" 하고 넘어간다. 나 역시 예전 대규모 센서 데이터를 처리할 때 Gorilla 논문을 보며 직접 구현할까 고민하다가 결국 범용 압축기로 타협했던 기억이 난다.

그런데 최근 흥미로운 프로젝트 하나가 눈에 띄었다. 오직 IEEE-754 64-bit double 데이터만을 위해 만들어진 무손실 압축기, `fc`다.

## 무식하지만 확실한 아키텍처: 코덱 경쟁 (Competition)

`fc`의 핵심 아키텍처는 단순하면서도 무식(?)하다. 입력 스트림을 청크(블록)로 쪼갠 뒤, 무려 50여 개의 특화된 코덱을 **경쟁** 시킨다.

여기에는 Predictor, XOR/Delta, Lempel-Ziv, BWT 같은 변환 코덱들이 전부 포함된다. 각 블록마다 이 코덱들을 전부 돌려보고 가장 크기가 작은 결과를 뱉어내는 방식이다. 범용 압축기가 데이터의 바이트 패턴을 찾는 데 집중한다면, `fc`는 숫자가 가진 '수학적 구조(Smooth curves, Repeated values 등)'를 이해하려고 노력한다.

## 직관적이고 깔끔한 API

코드를 보자. API는 놀라울 정도로 깔끔하다. 15년 넘게 C 코드를 봐왔지만, 이렇게 명확한 인터페이스를 제공하는 라이브러리는 언제나 환영이다.

```c
typedef struct { int p, t, c; } fc_cfg;
size_t fc_enc(const void *src, size_t bytes, void *dst, fc_cfg cfg);
size_t fc_dec(const void *src, size_t bytes, void *dst);
```

인코딩 시 스레드 개수(`t`)와 predictor 테이블 크기(`p`)만 지정하면 끝이다. 디코딩은 아예 설정 구조체조차 필요 없이 시스템의 코어 수를 파악해 알아서 병렬로 처리한다.

## 극단적인 트레이드오프: 인코딩 CPU를 태워 스토리지 I/O를 줄이다

벤치마크 결과는 이 라이브러리의 철학을 명확히 보여준다.

- **Compression Ratio:** 구조화된 float(예: 상수, 포물선 패턴 등)에서는 zstd-9 대비 압축비가 수천 배에 달한다.
- **Encode Throughput:** 약 120 MB/s 수준으로 꽤 느리다. 50개 코덱을 경쟁시키는 비용(CPU 낭비)이다.
- **Decode Throughput:** 1.2 GB/s를 훌쩍 넘기며 zstd-3 디코드 속도의 80% 수준에 근접한다.

왜 디코딩만 빠를까? 인코딩할 때는 모든 코덱을 테스트해야 하지만, 디코딩할 때는 헤더에 적힌 승자(winner) 코덱 하나만 실행하면 되기 때문이다. 게다가 디코딩은 블록 단위로 병렬 처리(Auto-threaded)된다. 전형적인 **Write-once / Read-many** 워크로드에 완벽히 부합한다.

## 치명적인 하드웨어 제약

솔직히 아쉬운 점도 있다. 현재 x86-64 환경에서 AVX2, SSE4.2, BMI, LZCNT 명령어 셋을 강제한다. ARM/NEON 지원은 아예 없다. AWS Graviton이나 Apple Silicon이 대세가 되어가는 요즘 트렌드에는 다소 역행하는 하드웨어 제약이다. 이식성이 중요한 범용 라이브러리로 쓰기엔 아직 무리가 있다.

## Hacker News의 반응과 철학

HN 스레드에서도 꽤 열띤 토론이 있었다. 누군가는 OpenLZ나 pcodec과 비교를 요구했고, Chimp128 같은 최신 논문 알고리즘을 언급하기도 했다.

이에 대한 작성자의 스탠스는 단호했다. "범용 압축기를 대체하려는 게 아니다. 숫자 데이터의 구조적 특성을 파악해 최적의 표현을 찾는 좁은 목적의 도구다." 이런 철학, 개인적으로 아주 좋아한다. 만능 도구(Silver bullet)를 자처하는 프로젝트치고 끝이 좋은 경우를 못 봤다. 자신이 해결하고자 하는 문제의 도메인을 정확히 아는 엔지니어의 코드다.

## 총평 (Verdict)

결론적으로 `fc`는 당장 내일 프로덕션에 투입할 만한 물건은 아니다. 작성자 스스로도 'Research-grade'라고 선을 그었고, 포맷의 하위 호환성도 보장하지 않는다. 노이즈가 많은 순수 랜덤 데이터에서는 여타 압축기처럼 힘을 쓰지 못한다.

하지만 시계열 데이터베이스 엔진을 바닥부터 개발 중이거나, 페타바이트 단위의 과학 데이터를 S3에 저장하며 비용 압박을 받는 CTO라면 반드시 뜯어봐야 할 훌륭한 레퍼런스다. 인코딩에 CPU 자원을 태워서 스토리지와 네트워크 I/O를 극한으로 줄인다는 트레이드오프는, 클라우드 비용 최적화가 화두인 요즘 시대에 꽤 매력적인 아키텍처이기 때문이다.
