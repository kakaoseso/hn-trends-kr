최근 해커뉴스(Hacker News)를 뜨겁게 달군 프로젝트 하나가 눈에 띄었습니다. "Anos"라는 이름의 약 100KiB 크기 x86-64 및 RISC-V용 마이크로커널입니다.

솔직히 말해서, 요즘 같이 AI가 코드를 짜주고 프레임워크가 모든 것을 추상화해버리는 시대에 처음부터 끝까지 손으로 짠 OS를 본다는 것은 꽤나 신선한 충격입니다. 원작자는 리누스 토발즈의 유명한 멘트를 인용하며 그저 취미일 뿐, GNU-Linux처럼 크고 전문적이진 않을 것이라고 못 박았습니다. 하지만 코드를 뜯어보면 단순한 장난감 이상의 깊은 엔지니어링 철학이 담겨 있습니다.

오늘은 15년 차 엔지니어의 시각에서 Anos의 아키텍처를 해부해보고, 왜 이런 프로젝트가 여전히 중요한지 이야기해보려 합니다.

## 마이크로커널의 부활: STAGE3와 SYSTEM

Anos의 핵심 설계는 철저하게 마이크로커널(Microkernel) 철학을 따릅니다. 커널 공간인 **STAGE3** 는 스케줄링, 메모리 관리, 그리고 IPC(Inter-Process Communication)라는 최소한의 기능만 제공합니다. 나머지 모든 OS 수준의 추상화는 유저 모드 슈퍼바이저인 **SYSTEM** 이 담당하죠.

과거 L4 마이크로커널이나 구글의 Fuchsia(Zircon)를 다뤄보신 분들이라면 이 구조가 꽤 익숙하실 겁니다.

- **STAGE3:** CPU, 타이머(HPET, TSC), 인터럽트, 물리/가상 메모리 관리 등 하드웨어와 맞닿은 극소수의 기능만 처리합니다.
- **SYSTEM:** VFS, 프로세스 관리, 하드웨어 드라이버(PCI, AHCI) 등을 유저 스페이스에서 구동합니다.

여기서 가장 흥미로운 부분은 **Capability-based security** 모델입니다. 유저 프로세스는 기본적으로 아무런 권한이 없습니다. 시스템 자원에 접근하려면 SYSTEM을 통해 권한(Capability)을 위임받아야만 합니다.

## IPC와 Syscall: 성능의 병목을 어떻게 뚫었나?

마이크로커널의 영원한 숙제는 Context switch 오버헤드입니다. 모든 드라이버가 유저 스페이스에 있다면, 디스크에서 파일을 하나 읽을 때마다 수많은 IPC 메시지가 오가야 하죠.

Anos는 이를 해결하기 위해 동기식(Synchronous), 제로 카피(Zero-copy) 메시지 패싱을 구현했습니다. x86_64에서는 SYSCALL과 SYSRET 명령어를, RISC-V에서는 ecall을 사용하여 Fast channel을 구축했습니다. 과거에 쓰던 int 0x69 기반의 Slow channel은 과감히 Deprecation 시켰더군요. 아주 올바른 결정입니다.

개념적으로 Anos의 IPC 요청은 다음과 같은 형태를 띨 것입니다. (공식 문서의 설명을 바탕으로 재구성한 수도코드입니다)

```c
// 유저 스페이스의 PCI 드라이버가 SYSTEM에 권한을 요청하는 개념적 흐름
ipc_message_t msg = {
    .target = SYSTEM_SERVER_ID,
    .operation = REQ_MAP_MMIO,
    .payload = { .phys_addr = BAR0_ADDRESS, .size = 0x1000 }
};

// Zero-copy 동기식 IPC 호출 (Fast channel)
status_t res = syscall_ipc_send_recv(&msg);

if (res == OK) {
    // 위임받은 Capability를 통해 MMIO 영역 제어
    uint32_t* pci_regs = (uint32_t*) msg.response.mapped_addr;
    pci_regs[OFFSET] = ENABLE_DEVICE;
}
```

이러한 설계는 드라이버가 크래시 나더라도 전체 시스템이 패닉에 빠지지 않게 해줍니다. 2010년대 후반에 분산 시스템의 장애 격리(Fault isolation) 문제를 해결하기 위해 고군분투했던 기억이 나는데, 단일 노드 OS 레벨에서 이를 우아하게 풀어낸 점이 인상적입니다.

## 아쉬운 점: SMP와 하드웨어 지원의 한계

물론 한계도 명확합니다. 현재 x86_64에서 최대 16코어의 SMP(Symmetric Multiprocessing)를 지원하지만, x2APIC로 전환하지 않아 8코어 이상에서는 불안정하다고 명시되어 있습니다. 최신 서버 환경에서 x2APIC는 선택이 아닌 필수입니다. 이 부분은 향후 반드시 개선되어야 할 병목 지점입니다.

또한 RISC-V 포팅은 아직 실제 하드웨어에서 검증되지 않은 WIP(Work In Progress) 상태입니다. 하지만 GCC 16-experimental과 Newlib을 베이스로 자체 툴체인을 구축해 두 아키텍처의 Feature parity를 맞춰가려는 시도 자체는 박수받아 마땅합니다.

## 총평 (Verdict)

Anos는 당장 프로덕션에 도입할 수 있는 OS가 아닙니다. 하지만 시니어 엔지니어라면, 혹은 시스템 프로그래밍의 근본을 이해하고 싶은 주니어라면 주말에 시간을 내어 이 100KiB짜리 커널 코드를 정독해 보시길 강력히 권합니다.

추상화의 늪에 빠져 우리가 딛고 있는 기반(Hardware)이 어떻게 동작하는지 잊어버리기 쉬운 요즘, Anos는 훌륭한 나침반이 되어줄 것입니다.
