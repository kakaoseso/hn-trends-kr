최근 PyPI의 핵심 딥러닝 프레임워크인 `lightning` 패키지에서 발생한 공급망(Supply Chain) 공격 소식을 접하고, 솔직히 올 것이 왔다는 생각이 들었습니다. 15년 넘게 엔지니어로 일하면서 수많은 패키징 사고를 봤지만, 이번 'Shai-Hulud' 테마의 악성코드는 그 정교함과 타겟팅 방식에서 이전과는 차원이 다릅니다.

단순히 오타를 노린 타이포스쿼팅이 아닙니다. AI 모델을 훈련시키거나 파인튜닝하는 시니어 엔지니어와 연구원들의 로컬 환경, 그리고 CI/CD 파이프라인 전체를 장악하려는 고도로 설계된 웜(Worm)입니다.

## 악성코드의 동작 원리: 단순한 파이썬 스크립트가 아니다

이번에 감염된 버전은 4월 30일에 배포된 `lightning` 2.6.2와 2.6.3입니다. 개발자가 `pip install lightning`을 실행하고 모듈을 임포트하는 순간, 패키지 내부에 숨겨진 `_runtime` 디렉토리의 Python 로더가 동작합니다.

흥미로운 점은 메인 페이로드가 파이썬이 아니라 난독화된 **JavaScript** 라는 것입니다.

- **Dropper:** `setup.mjs`라는 부트스트래퍼가 실행됩니다. 만약 시스템에 Bun 런타임이 없다면, OS와 아키텍처를 파악해 GitHub 릴리즈에서 Bun을 조용히 다운로드합니다.
- **Payload:** 이후 14.8MB에 달하는 거대한 JS 페이로드(`.claude/router_runtime.js`)를 실행합니다.

이 악성코드는 로컬 파일시스템의 토큰, 환경 변수뿐만 아니라 AWS, Azure, GCP의 클라우드 자격 증명을 싹쓸이합니다. 특히 Linux GitHub Actions 러너에서는 Runner.Worker 프로세스의 메모리를 덤프하여 시크릿을 추출해냅니다. 정말 지독할 정도로 철저합니다.

## 4중 탈취 채널과 교묘한 지속성(Persistence)

제가 이 공격에서 가장 놀란 부분은 데이터를 빼돌리는 Exfiltration 채널의 설계입니다. 네트워크 차단에 대비해 4개의 독립적인 채널을 병렬로 돌립니다.

1. C2 서버로의 HTTPS POST 전송
2. GitHub 커밋 검색 API를 활용한 Dead-drop
3. 공격자가 통제하는 공개 GitHub 레포지토리 생성
4. 탈취한 토큰을 이용해 피해자의 레포지토리에 직접 푸시

여기에 더해 개발자 도구를 타겟으로 한 지속성 확보 방식은 혀를 내두르게 합니다. VS Code의 `.vscode/tasks.json`을 변조해 폴더를 열 때마다 악성코드가 실행되게 만드는 것은 흔하지만, **Claude Code** 의 훅 시스템(`.claude/settings.json`)을 악용한 사례는 야생에서 발견된 최초의 케이스 중 하나일 것입니다. 최신 AI 개발 도구의 트렌드를 공격자들도 완벽하게 꿰뚫고 있다는 증거입니다.

## 해커뉴스 커뮤니티의 반응과 파이썬 생태계의 고질병

해커뉴스(Hacker News) 스레드에서는 이 사태를 두고 열띤 토론이 벌어졌습니다. 누군가는 이런 공격이 정말 늘어난 것인지 묻더군요. 데이터는 명확합니다. 공격 빈도는 폭발적으로 증가하고 있습니다.

가장 뼈아픈 지적은 파이썬 커뮤니티의 배포 문화에 대한 비판이었습니다. 한 유저는 프로덕션 머신에서 `pip install`을 실행해 파이썬 프로그램을 배포하는 것이 거의 보편적이라고 꼬집었습니다. 전적으로 동의합니다.

우리는 2024년이 넘도록 여전히 제대로 된 Lockfile 없이 `requirements.txt`에 의존하고, CI 파이프라인이나 프로덕션 서버에서 외부 네트워크를 통해 코드를 즉석에서 빌드하고 실행합니다. Docker를 쓴다고 안심할 수 없습니다. Dockerfile 내부에서 `pip install`을 무방비로 실행한다면, 결국 컨테이너 빌드 시점에 악성코드가 여러분의 빌드 환경 시크릿을 탈취할 테니까요.

의존성을 극단적으로 줄이고 LLM을 이용해 순수 코드로만 작성하자는 의견도 있었지만, 이는 현실적인 대안이 되기 어렵습니다. 브라우저 API 변경이나 타임존 업데이트 같은 수많은 엣지 케이스를 우리가 직접 유지보수하는 것은 불가능에 가깝습니다. 결국 우리는 서드파티 라이브러리를 써야만 합니다.

## Principal Engineer의 시선: 우리는 무엇을 해야 하는가?

이번 사태는 단순한 해프닝이 아닙니다. npm 생태계를 노렸던 공격이 PyPI를 거쳐 다시 npm으로 번져나가는 크로스 에코시스템 웜으로 진화했습니다.

만약 여러분의 팀이 해당 기간에 `lightning` 2.6.2나 2.6.3 버전을 설치했다면, 그 머신은 **완전히 타협된** 것으로 간주해야 합니다. AWS IAM 키, GitHub 토큰, 로컬 환경 변수 등 모든 것을 즉시 로테이션해야 합니다.

장기적으로 우리는 엔지니어링 문화를 바꿔야 합니다. 첫째, 로컬 개발 환경과 CI/CD 환경을 철저히 격리하고 샌드박싱해야 합니다. 둘째, 의존성 패키지를 설치할 때 재현 가능한 빌드(Reproducible builds)를 강제해야 합니다. Nix 같은 도구가 부담스럽다면, 최소한 엄격한 Lockfile 관리와 사내 프라이빗 패키지 미러링이라도 도입해야 합니다.

더 이상 누군가 검수했겠지라는 안일한 생각으로 남의 코드를 내 프로덕션에 밀어 넣는 시대는 끝났습니다. 편리함의 대가는 생각보다 훨씬 가혹합니다.
