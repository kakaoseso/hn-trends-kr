최근 AWS 리전 장애나 예상치 못한 네트워크 파티션 문제로 새벽에 PagerDuty 알람을 받고 깨본 적이 있으신가요? 우리는 흔히 마이크로서비스 아키텍처를 설계하며 '고가용성(High Availability)'과 '이중화(Redundancy)'를 입버릇처럼 말합니다. Load Balancer 뒤에 인스턴스 몇 개를 더 띄워두고, Multi-AZ로 데이터베이스를 구성한 뒤 안심하곤 하죠.

하지만 생명과 직결된, 그리고 수조 원의 예산이 투입된 우주 환경에서의 장애 허용(Fault Tolerance)은 우리가 아는 웹 인프라의 이중화와는 차원이 다릅니다. 최근 Communications of the ACM에 올라온 [NASA Artemis II의 결함 허용 컴퓨터 아키텍처](https://cacm.acm.org/news/how-nasa-built-artemis-iis-fault-tolerant-computer/)에 대한 글과 Hacker News의 열띤 토론을 보며, 시니어 엔지니어로서 우리의 시스템 설계를 되돌아보게 하는 몇 가지 깊은 인사이트를 얻었습니다.

이번 포스트에서는 NASA가 우주 방사선과 하드웨어 결함이라는 극한의 환경 속에서 어떻게 시스템을 설계했는지, 그리고 그 이면에 숨겨진 엔지니어링 철학을 파헤쳐보겠습니다.

## 8개의 코어, Fail-Silent 설계와 Lockstep 프로세서

Artemis II의 Orion 우주선은 총 2개의 Vehicle Management Computer를 사용하며, 각각 2개의 Flight Control Module(FCM)을 가집니다. 즉, 총 4개의 FCM이 존재하죠. 여기서 끝이 아닙니다. 각 FCM은 다시 '자가 점검(self-checking)'을 수행하는 한 쌍의 프로세서로 구성됩니다. 결과적으로 8개의 CPU가 병렬로 비행 소프트웨어를 실행합니다.

Hacker News에서 한 유저가 날카로운 질문을 던졌습니다.
> "누가 8개가 적당한 숫자라고 결정했을까요? 왜 4개나 16개가 아닐까요?"

이에 대한 답은 철저한 '허용 가능한 손실률(acceptable total loss rate)' 계산에서 나옵니다. 우주 공간의 백그라운드 방사선이 비트를 뒤집는 현상(Bit Flip)에 대해 NASA는 지난 50년간의 방대한 데이터를 가지고 있습니다. 이를 기반으로 확률적 모델링을 통해 최적의 중복도를 찾아낸 것이죠.

기술적으로 가장 흥미로운 부분은 이들이 **Fail-Silent** 철학을 채택했다는 점입니다. 두 개의 프로세서가 Lockstep 방식으로 정확히 동일한 연산을 수행합니다. 

```c
// Lockstep 아키텍처의 개념적 동작
void execute_instruction(Instruction inst) {
    Result res1 = cpu1.execute(inst);
    Result res2 = cpu2.execute(inst);
    
    // 하드웨어 레벨에서의 즉각적인 결함 탐지
    if ((res1 ^ res2) != 0) {
        trigger_hardware_fault();
        halt_and_isolate_module(); // Fail-Silent
    }
}
```

방사선으로 인해 하나의 CPU에서 잘못된 계산이 발생하면, 시스템은 즉시 이를 감지하고 해당 FCM을 '침묵(Silent)'시킵니다. 잘못된 데이터를 전파하느니 차라리 죽어버리는 것을 택하는 것입니다. 웹 환경에서 우리가 흔히 겪는 '오염된 캐시'나 '비정상 상태로 동작하며 에러 로그만 뿜어내는 좀비 프로세스'를 생각해보면, 이 Fail-Silent 접근법이 얼마나 깔끔하고 안전한지 알 수 있습니다.

놀라운 점은 이렇게 죽어버린 FCM이 짐짝이 되지 않는다는 것입니다. 시스템은 즉시 모듈을 리셋하고, 정상 동작 중인 다른 모듈들과 상태를 재동기화(re-synchronize)하여 비행 중 다시 클러스터에 합류시킵니다. NASA에 따르면 22초 안에 3개의 FCM을 잃더라도 남은 1개로 안전하게 비행을 유지할 수 있다고 합니다.

## 결정론적(Deterministic) 에러 체크와 하드웨어 보강

분산 시스템에서 가장 해결하기 어려운 문제 중 하나는 상태의 일관성(Consistency)을 유지하는 것입니다. NASA는 이를 해결하기 위해 극단적인 결정론(Determinism)을 강제합니다.

모든 FCM은 동일한 입력을 받고, 동일한 코드를 실행하며, 동일한 출력을 내야 합니다. 매 초마다 각 FCM의 로컬 클럭은 네트워크의 '진짜(True)' 시간과 비교되어 캘리브레이션 됩니다. 만약 애플리케이션이 엄격한 Deadline을 맞추지 못하면, 예외 없이 해당 모듈은 즉시 차단되고 리셋됩니다. Latency Spike를 그저 '일시적인 네트워크 지연'으로 치부하고 넘어가는 우리의 일반적인 서버 환경과는 다릅니다.

하드웨어 자체도 무장되어 있습니다.
- **TMR Memory:** Triple-Modular-Redundant 메모리를 사용하여 읽기 작업마다 발생하는 Single-bit 에러를 자체적으로 수정합니다.
- **Dual-lane NIC:** 네트워크 인터페이스 카드조차 두 개의 트래픽 레인을 지속적으로 비교하여, 통신 패브릭에서 Bit Flip이 발생하면 즉각 Fail-Silent 처리합니다.

## 이기종 이중화(Dissimilar Redundancy)와 N-Version Programming의 환상

아무리 하드웨어를 완벽하게 이중화하더라도, 소프트웨어 버그 하나면 4개의 FCM이 동시에 다운될 수 있습니다. 이를 Common Mode Failure라고 부릅니다. 이를 방지하기 위해 Orion은 완전히 독립적인 Backup Flight Software(BFS)를 탑재합니다.

이 BFS는 메인 시스템과 다른 하드웨어, 다른 OS에서 구동되며, 완전히 독립적으로 개발된 '단순화된' 비행 소프트웨어를 실행합니다. 

Hacker News의 토론 중 가장 제 눈길을 끌었던 것은 N-Version Programming에 대한 비판이었습니다. 보통 사람들은 "두 개의 다른 팀이 다른 언어로 똑같은 스펙을 구현하면 버그가 안 겹치겠지?"라고 생각합니다. 하지만 1986년 Knight and Leveson의 유명한 논문이 증명했듯, 이는 환상에 불과합니다. 버그는 보통 '도메인의 복잡성'이나 '스펙 자체의 결함'에서 발생하기 때문에, 다른 팀이 만들더라도 결국 비슷한 엣지 케이스에서 똑같이 무너집니다.

NASA가 이 문제를 해결한 방식은 '똑같은 스펙'을 구현하는 것이 아니라, 백업 시스템의 스펙 자체를 **극도로 단순화** 하는 것이었습니다. BFS는 메인 시스템의 모든 화려한 기능을 지원하지 않습니다. 오직 우주선을 안정화하고 승무원을 살리는 핵심 기능에만 집중합니다. 복잡성을 줄이는 것만이 소프트웨어 결함을 피하는 유일한 은통알(Silver Bullet)임을 NASA도 알고 있는 것입니다.

## 완벽한 실패를 위한 설계: Dead Bus

가장 인상 깊었던 부분은 모든 전력이 상실되는 'Dead Bus' 시나리오에 대한 대비입니다. 전력이 복구되면 우주선은 Safe Mode로 진입하여 스스로 자세를 제어하고, 태양광 패널을 태양 쪽으로 향하게 하여 전력을 회복합니다. 그 후 열 안정을 위해 꼬리를 태양으로 향하게 한 뒤 지구와의 통신 재개를 시도합니다.

이것이 진정한 Graceful Degradation(우아한 성능 저하)입니다. 시스템이 완전히 붕괴되었을 때, 어떤 순서로 시스템을 복구하고 가장 크리티컬한 리소스(전력, 통신, 생명 유지)를 어떻게 확보할 것인지 하드코딩된 생존 본능을 심어둔 것입니다.

## 결론: 우리의 시스템에 적용할 점

솔직히 말해서, 우리가 개발하는 일반적인 B2C/B2B 서비스에 이런 수준의 결함 허용(Fault Tolerance)을 구축하는 것은 미친 짓입니다. 비용(Cost)과 복잡도(Complexity)가 비즈니스 가치를 아득히 초과할 테니까요. 

하지만 이 아키텍처에 담긴 엔지니어링 철학은 반드시 배워야 합니다.

1. **Fail-Silent의 원칙:** 잘못된 상태로 시스템을 억지로 굴리지 마세요. 차라리 빠르게 죽이고(Crash-only software design), 깨끗한 상태로 재시작하는 것이 연쇄 장애(Cascading Failure)를 막는 길입니다.
2. **복잡성의 통제:** 완벽한 백업 시스템은 메인 시스템의 복제본이 아닙니다. 가장 핵심적인 기능만 남긴 단순화된 시스템이어야 합니다.
3. **최악의 시나리오 대비:** AWS 리전 전체가 날아가거나 DB가 완전히 삭제되었을 때, 시스템이 어떻게 '우아하게' 죽고 복구될지 설계되어 있나요?

Apollo 시절 우주비행사들은 컴퓨터가 고장 나면 직접 수리해야 할지도 모른다는 사실에 "차라리 뇌수술을 배우는 게 낫겠다"고 농담을 했다고 합니다. 반세기가지난 지금, NASA의 엔지니어들은 하드웨어와 소프트웨어의 경계를 넘나들며 우주비행사들이 뇌수술을 배울 필요가 없도록 완벽에 가까운 시스템을 만들어냈습니다.

오늘 당신이 작성한 코드와 아키텍처는 예상치 못한 결함 앞에서 얼마나 우아하게 대처할 수 있습니까? Artemis II의 아키텍처는 우리에게 그 무거운 질문을 던지고 있습니다.
