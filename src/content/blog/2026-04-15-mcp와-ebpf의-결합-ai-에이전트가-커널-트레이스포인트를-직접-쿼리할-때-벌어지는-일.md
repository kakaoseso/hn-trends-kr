최근 인프라스트럭처와 AI 씬이 만나는 교차점에서 굉장히 흥미로운 패턴이 하나 떠오르고 있다. 바로 **MCP** (Model Context Protocol)를 단순한 챗봇용 컨텍스트 주입 도구가 아니라, **Observability** (관측성)의 핵심 인터페이스로 사용하려는 움직임이다.

지난달, 이 흐름을 보여주는 세 가지 시그널이 거의 동시에 발생했다. 첫째, Datadog이 자사 대시보드와 AI 에이전트를 연결하는 MCP 서버를 출시했다. 둘째, 보안 업체 Qualys가 MCP 서버를 "AI 시대의 새로운 섀도우 IT(Shadow IT)"로 규정하며 정적 시크릿 키 관리의 위험성을 경고했다. 셋째, Microsoft Retina가 eBPF를 활용해 애플리케이션 변경 없이 커널 레벨의 네트워크 텔레메트리를 수집하는 방식을 선보였다.

이 세 가지 사건을 관통하는 하나의 명확한 트렌드가 있다. **AI 에이전트는 이제 인프라스트럭처의 텔레메트리 데이터에 직접 접근하길 원하며, MCP가 그 표준 통로로 자리 잡고 있다는 것** 이다.

하지만 15년 넘게 백엔드와 인프라를 굴려본 엔지니어 입장에서, 현재 시장의 접근 방식에는 꽤 뚜렷한 한계가 보인다. 오늘은 Ingero 팀이 블로그를 통해 제시한 'MCP-native Observability' 아키텍처를 바탕으로, 우리가 왜 기존의 대시보드 기반 모니터링에서 벗어나야 하는지, 그리고 해커뉴스(Hacker News) 커뮤니티에서 제기된 현실적인 우려들은 무엇인지 깊게 파헤쳐 보겠다.

## 두 가지 MCP 접근법: Wrapper vs Native

현재 Observability 데이터를 AI 에이전트(Claude 등)에 연결하는 방식은 크게 두 갈래로 나뉜다.

### 1. Wrapper 방식 (Datadog 스타일)
기존에 구축된 메트릭, 로그, 트레이스 파이프라인 위에 MCP 어댑터를 얹는 방식이다. AI 에이전트는 이미 집계(Aggregation)되고 전처리된 데이터를 API를 통해 가져온다. 

이 방식은 나쁘지 않다. "지난 1시간 동안 서비스 X의 p99 Latency는 어땠어?" 같은 질문에는 훌륭하게 답한다. 하지만 Root cause(근본 원인)를 파고들어야 하는 상황에서는 무용지물이 된다. 데이터가 집계되는 순간, 개별 트랜잭션의 컨텍스트와 인과관계는 증발해 버리기 때문이다.

### 2. MCP-Native 방식 (Ingero 스타일)
이 글에서 주목하고 싶은 아키텍처다. 기존 모니터링 플랫폼을 감싸는 대신, eBPF 에이전트를 통해 CUDA 런타임 및 드라이버 API를 `uprobe`로 직접 트레이싱한다. 수집된 로우 레벨 데이터는 SQLite에 저장되고, 7개의 MCP 도구(Tool)를 통해 AI 에이전트에게 **날것 그대로** 노출된다.

이 구조에서 MCP는 단순한 어댑터가 아니라 **Primary Interface** 가 된다.

## 집계(Aggregation)가 숨겨버린 진실을 AI가 찾는 법

Ingero 팀의 vLLM TTFT(Time To First Token) 회귀 분석 사례는 이 아키텍처의 강력함을 완벽하게 보여준다. 첫 번째 토큰 생성 시간이 베이스라인 대비 14.5배나 느려진 상황이었다.

기존 대시보드였다면 GPU 사용률이 튀었다는 것 정도만 알 수 있었을 것이다. 하지만 eBPF로 떠낸 로우 트레이스 데이터를 SQLite에 담아 Claude에게 MCP로 연결하자, Claude는 다음과 같은 도구들을 자율적으로 호출하며 원인을 추적했다.

- **get_trace_stats:** 트레이스 요약 확인 (12,847개의 CUDA 이벤트, 총 GPU 시간 등)
- **run_sql:** 로우 이벤트 데이터에 대한 커스텀 SQL 쿼리 실행 ("100ms 이상 걸린 cudaMemcpyAsync 호출을 모두 보여줘")
- **get_stacks:** 플래그가 지정된 이벤트의 콜 스택 검사

결과는 놀라웠다. 단 30초 만에 Claude는 `logprobs` 연산이 디코드 루프를 블로킹하여 Critical path에서 256배의 병목을 일으키고 있다는 사실을 찾아냈다. **이러한 인과관계는 메트릭을 집계(Aggregation)하는 순간 완전히 사라지는 정보들이다.**

## 코드로 보는 MCP 연결

이 모든 과정이 복잡한 연동 코드 없이, 단순한 JSON 설정 파일 하나로 끝난다는 점이 MCP의 진정한 파괴력이다. Claude Desktop이나 CLI 도구에 아래와 같은 설정만 주입하면 된다.

```json
{
  "mcpServers": {
    "ingero": {
      "command": "./bin/ingero",
      "args": ["mcp", "--db", "investigations/pytorch-dataloader-starvation.db"]
    }
  }
}
```

이후 `claude --mcp-config /tmp/ingero-mcp-dataloader.json` 명령어를 실행하고 "GPU 성능 저하의 원인이 뭐야?"라고 묻기만 하면, 모델이 알아서 7개의 도구를 탐색하고 SQL을 작성해 원인을 분석한다.

## Hacker News의 시선: 혁신과 보안의 딜레마

이 아키텍처에 대한 해커뉴스(HN)의 반응은 뜨거우면서도 날카로웠다. 시니어 엔지니어들이 모인 곳답게 칭찬보다는 뼈 때리는 비판과 우려가 주를 이뤘다.

**1. 레거시 API의 껍데기일 뿐이라는 비판**
> "현재 대부분의 MCP 서버와 앱들은 디자인적으로 너무 빈약합니다. 초기 스마트폰 시절 앱들이 웹사이트를 그대로 복붙해놓은 것과 같아요. B2B MCP 서버들은 그저 레거시 API를 래핑할 뿐, 경험을 밑바닥부터 재창조하려 하지 않습니다."

전적으로 동의한다. Datadog의 접근 방식이 딱 이렇다. AI 에이전트의 진정한 가치는 인간이 할 수 없는 속도로 수만 줄의 커널 이벤트를 스캔하고 인과관계를 조립하는 데 있다. 대시보드용으로 예쁘게 깎아놓은 요약 데이터를 AI에게 먹이는 것은, 페라리를 사놓고 골목길에서만 타는 격이다.

**2. AI에게 SQL 실행 권한을 주는 것에 대한 공포**
> "AI 에이전트에게 커스텀 SQL 쿼리를 실행할 수 있는 MCP 엔드포인트를 준다고요? 만약 AI가 성능 버그의 원인을 못 찾으면, 그냥 DB 데이터를 수정(UPDATE/DELETE)해버리고 '버그를 해결했습니다!'라고 거짓말하면 어떡하죠?"

가장 많은 공감을 받은 댓글 중 하나다. LLM의 환각(Hallucination) 현상을 고려할 때 충분히 가능한 시나리오다. 다른 유저가 "SQLite를 Read-only 모드로 열면 된다"고 답하긴 했지만, 이는 단편적인 해결책일 뿐이다. 엔터프라이즈 환경에서 AI 에이전트에게 인프라 접근 권한을 줄 때는 **Least Privilege(최소 권한의 원칙)** 를 어떻게 시스템적으로 강제할 것인지에 대한 깊은 고민이 필요하다. Qualys가 지적한 MCP 서버의 보안 취약점 문제와도 직결되는 부분이다.

**3. 동적 eBPF 프로빙의 가능성**
> "MCP eBPF 모듈을 만들어서, AI가 상황에 따라 동적으로 프로브(Probe) 포인트를 생성하게 할 수는 없을까요?"

기술적으로는 가능하다. 실제로 에러 로그를 보고 AI가 스스로 판단해 필요한 커널 함수에 동적으로 트레이스포인트를 꽂아 넣는 시나리오는 상상만 해도 짜릿하다. 하지만 프로덕션 환경의 인프라를 책임지는 Principal Engineer 입장에서는 식은땀이 흐르는 이야기다. 검증되지 않은 eBPF 프로그램이 커널 패닉을 일으키거나 성능을 갉아먹을 리스크를 컴플라이언스 팀에게 어떻게 설득할 것인가? 아직은 시기상조다.

## 결론: 그래서 우리는 무엇을 준비해야 하는가?

솔직히 말해, 나는 여전히 대시보드를 사랑한다. 시스템의 전반적인 건강 상태를 한눈에 파악하는 데는 시계열 그래프만 한 것이 없다. 

하지만 장애가 발생하고 p99 Latency가 미쳐 날뛸 때, 우리는 결국 대시보드를 닫고 터미널을 연다. 분산 트레이싱 로그를 뒤지고, 쓰레드 덤프를 뜨고, 심할 경우 `strace`나 `bpftrace`를 켜서 커널 레벨의 이벤트를 눈으로 쫓는다. 

Ingero가 보여준 **MCP-native Observability** 는 바로 이 '고통스러운 디버깅의 시간'을 AI에게 위임하는 완벽한 청사진이다. 메트릭 파이프라인을 거치며 정보가 손실되기 전, 커널과 하드웨어에서 발생하는 날것의 이벤트를 AI 에이전트에게 직접 연결하는 패러다임 전환이다.

아직 보안(SQL 인젝션, 권한 제어)이나 생태계 성숙도 측면에서 해결해야 할 과제가 산더미처럼 쌓여있다. 하지만 방향성은 명확하다. 앞으로의 모니터링 시스템은 '인간이 보기 좋게 요약하는 기능'과 'AI가 분석하기 좋게 원본을 보존하는 기능'으로 철저히 분리될 것이다. 그리고 후자의 중심에는 의심의 여지 없이 MCP와 eBPF가 자리 잡을 것이다.
