최근 구글 Project Zero에서 픽셀 10을 타겟으로 한 제로클릭 익스플로잇 체인을 공개했습니다. 15년 넘게 시스템 엔지니어링과 보안 아키텍처를 다뤄오면서 수많은 취약점을 봐왔지만, 이번 케이스는 여러모로 시사하는 바가 큽니다. 우리가 아무리 유저스페이스 샌드박스를 견고하게 만들고 최신 완화 기법을 도입해도, 결국 커널 레벨의 하드웨어 드라이버 하나가 뚫리면 모든 것이 무용지물이 된다는 뼈아픈 교훈을 다시 한번 상기시켜 줍니다.

## 픽셀 9에서 10으로: 방어 기법의 진화와 우회

Project Zero 팀은 기존 픽셀 9에서 사용했던 Dolby 제로클릭 취약점(CVE-2025-54957)을 픽셀 10에 맞춰 업데이트했습니다. 흥미로운 부분은 픽셀 10이 `-fstack-protector` 대신 RET PAC을 사용한다는 점입니다. 스택 버퍼 오버플로우를 잡기 위한 `__stack_chk_fail` 덮어쓰기가 불가능해졌죠. 하지만 그들은 디코더 초기화 시 딱 한 번 호출되고 마는 `dap_cpdp_init` 코드를 덮어쓰는 방식으로 이를 우회했습니다. 방어 기법이 발전하면 공격자들은 그저 다른 실행 흐름을 찾을 뿐입니다.

## The Holy Grail: 5줄로 끝나는 커널 장악

이번 리포트의 핵심은 텐서 G5 칩에 탑재된 새로운 VPU(Video Processing Unit) 드라이버에서 발견된 로컬 권한 상승(LPE) 취약점입니다. 기존 BigWave 드라이버 대신 Chips&Media의 WAVE677DV 칩을 위한 드라이버가 도입되었는데, 이 드라이버는 리눅스 표준인 V4L2 API를 따르지 않고 칩의 하드웨어 인터페이스를 유저스페이스에 직접 노출하는 과감한(그리고 위험한) 결정을 내렸습니다.

문제의 코드를 한번 보시죠.

```c
static int vpu_mmap(struct file *fp, struct vm_area_struct *vm)
{
    unsigned long pfn;
    struct vpu_core *core =
        container_of(fp->f_inode->i_cdev, struct vpu_core, cdev);
    
    vm_flags_set(vm, VM_IO | VM_DONTEXPAND | VM_DONTDUMP);
    /* This is a CSRs mapping, use pgprot_device */
    vm->vm_page_prot = pgprot_device(vm->vm_page_prot);
    pfn = core->paddr >> PAGE_SHIFT;
    
    return remap_pfn_range(vm, vm->vm_start, pfn, vm->vm_end-vm->vm_start, vm->vm_page_prot) ? -EAGAIN : 0;
}
```

시스템 프로그래밍을 해보신 분들이라면 이 코드에서 등골이 서늘해지는 것을 느끼셨을 겁니다. `vpu_mmap` 함수는 VPU 하드웨어의 MMIO 레지스터 영역을 유저스페이스 가상 주소 공간에 매핑하기 위해 만들어졌습니다.

하지만 치명적인 실수가 있습니다. `remap_pfn_range`를 호출할 때 매핑할 크기(`vm->vm_end - vm->vm_start`)를 검증하지 않습니다. 즉, 유저스페이스에서 `mmap` 시스템 콜을 호출할 때 레지스터 영역보다 훨씬 큰 사이즈를 요청하면, 커널은 VPU 물리 주소부터 시작해서 원하는 만큼의 물리 메모리를 전부 유저스페이스에 매핑해버립니다.

더 최악인 것은 픽셀의 커널 물리 주소가 고정되어 있다는 점입니다. VPU 메모리 영역과 커널 사이의 오프셋이 항상 동일하기 때문에, 공격자는 커널의 위치를 찾기 위해 메모리를 스캔할 필요조차 없습니다. 그저 VMA 길이를 충분히 크게 잡아서 `mmap`을 호출하면 커널의 `.text`와 `.data` 영역을 내 마음대로 읽고 쓸 수 있는 완벽한 Arbitrary Read/Write 권한을 얻게 됩니다. 익스플로잇 코드를 작성하는 데 하루도 걸리지 않았다는 Project Zero의 코멘트가 전혀 과장으로 들리지 않습니다.

## 커뮤니티의 반응과 나의 생각

솔직히 이 코드를 처음 봤을 때 헛웃음이 나왔습니다. 이건 10년 전 초창기 리눅스 DRM 드라이버들에서나 보이던 101 수준의 실수입니다. 해커뉴스(Hacker News)에서도 이 이슈를 두고 열띤 토론이 벌어졌는데, 몇 가지 흥미로운 관점들이 있었습니다.

- **AI와 보안의 아이러니:** 한 유저가 이 코드 스니펫을 그대로 GPT-5.5에 넣고 문제점을 찾아보라고 질문했더니, 외부 검색 없이도 정확히 이 바운드 체크 누락 취약점을 짚어냈다고 합니다. AI가 취약점을 이렇게 쉽게 찾아낼 수 있다면, 애초에 코드 리뷰 단계에서는 왜 걸러지지 않았을까요? 일각에서는 최근 AI 기반 기능이 모바일 기기에 대거 탑재되면서(메시지 미디어 자동 디코딩 등) 제로클릭 공격 표면 자체가 넓어지고 있다고 지적합니다.
- **CVE의 인플레이션:** 최근 하루에 발표되는 CVE 수가 200개를 돌파하며 급증하고 있습니다. 이것이 AI가 생성한 저품질 코드가 많아져서인지, 아니면 AI 도구를 활용한 버그 헌팅이 쉬워져서인지에 대한 논쟁이 뜨겁습니다. 제 생각에는 둘 다입니다. 생산성이라는 명목 하에 검증되지 않은 코드가 쏟아지고 있고, 공격자들은 이를 자동화된 툴로 쓸어 담고 있습니다.
- **생태계의 파편화와 벤더의 책임:** 구글이 이 버그를 리포트 받고 71일 만에 패치한 것은 확실히 칭찬받을 만한 발전입니다. 하지만 안드로이드 생태계 전체를 보면 암울합니다. 수많은 서드파티 제조사들이 여전히 수개월에서 수년 전의 보안 패치에 머물러 있습니다. iOS의 Lockdown Mode와 비교하는 의견들도 있었지만, 본질적인 문제는 하드웨어 벤더(이 경우 칩셋 및 드라이버 제조사)가 보안을 최우선으로 두지 않는다는 점입니다.

## 결론: 우리는 여전히 모래성 위에 코드를 짠다

이번 픽셀 10 익스플로잇은 아주 명확한 교훈을 줍니다. 보안은 결코 소프트웨어 스택 상단의 화려한 샌드박싱 기술만으로 완성되지 않습니다. 커널과 직접 맞닿아 있는 서드파티 하드웨어 드라이버의 코드 품질이 보장되지 않는다면, 모든 방어 기법은 한순간에 무력화됩니다.

개발팀이 빠르게 출시하라는 압박 속에서 V4L2 같은 표준 인터페이스 연동을 포기하고 유저스페이스 직접 매핑이라는 숏컷을 택했을 때, 그 기술적 부채는 결국 유저의 보안 위협으로 돌아왔습니다. AI가 코드를 짜고 취약점을 찾는 시대가 도래했지만, 기본을 지키는 엔지니어링의 중요성은 오히려 더욱 커지고 있습니다. 프로덕션 레벨에 도입하기 전, 특히 하드웨어와 맞닿는 로우레벨 코드는 철저한 오딧이 필수적입니다.
