솔직히 말해서, 15년 넘게 백엔드와 데이터베이스 아키텍처를 설계해 오면서 제5정규형(5NF)을 의식하며 테이블을 설계한 적은 단 한 번도 없다. 스키마 리뷰 회의에서 누군가 "이 테이블은 4NF나 5NF를 위반하지 않나요?"라고 묻는다면, 십중팔구 그는 실무 경험이 부족한 주니어이거나 학술적인 완벽함에 심취한 아키텍트일 것이다.

최근 Alexey Makhotkin이 작성한 5NF and Database Design이라는 글과 이에 대한 Hacker News의 열띤 토론을 보며, 나 역시 오랫동안 느껴왔던 데이터베이스 정규화 교육의 맹점에 대해 이야기해보고자 한다. 우리는 왜 여전히 쓸데없이 복잡한 정규화 이론으로 고통받고 있는가?

## 5NF 교육의 근본적인 문제점: 쪼개기(Splitting)의 환상

대부분의 교과서나 Wikipedia에서 5NF를 설명하는 방식은 항상 똑같다. 먼저 직관적으로 이해하기 힘든 이상한 3개짜리 컬럼을 가진 테이블을 던져준다. 그런 다음 "특정 비즈니스 룰이 추가되면 이상 현상이 발생하므로, 이를 3개의 2-컬럼 테이블로 쪼개야 한다"고 가르친다.



Wikipedia의 예시를 보자. 영업사원(Salesman), 브랜드(Brand), 상품 유형(Product type)이 한 테이블에 묶여 있다. 그리고 "A 브랜드와 B 브랜드가 모두 P 상품을 만들고, 영업사원이 두 브랜드를 모두 취급한다면 반드시 두 브랜드의 P 상품을 모두 팔아야 한다"는 기괴한 비즈니스 룰을 가정한다.

현업에서 이런 식의 요구사항을 본 적이 있는가? 나는 없다. 애초에 설계가 잘못된 테이블을 가져와서 억지스러운 룰을 붙인 뒤 그것을 분해하는 과정을 보여주는 것은 엔지니어링적으로 아무런 도움이 되지 않는다. 실무에서는 테이블을 '쪼개는' 방식으로 설계하지 않는다. 우리는 비즈니스 요구사항(Logical model)에서 출발하여 물리적 테이블(Physical schema)을 '생성'한다.

## 실무적인 접근: 논리적 모델에서 물리적 테이블로

Makhotkin의 글에서 가장 마음에 드는 부분은 5NF라는 학술적 용어를 버리고, 두 가지 실용적인 설계 패턴으로 접근했다는 점이다.

### 1. AB-BC-AC Triangle 패턴 (아이스크림 예제)

아이스크림 브랜드, 맛(Flavour), 그리고 친구들의 선호도를 모델링한다고 가정해보자. 요구사항은 다음과 같다.
- 한 브랜드는 여러 맛을 생산한다. (M:N)
- 친구는 여러 브랜드를 좋아한다. (M:N)
- 친구는 여러 맛을 좋아한다. (M:N)

이 구조는 자연스럽게 3개의 Anchor(엔티티)와 이들을 연결하는 3개의 Junction table로 귀결된다. 교과서에서 말하는 5NF 분해 과정 따위는 필요 없다. 처음부터 `brand_flavours`, `friend_brands`, `friend_flavours`라는 교차 테이블을 만들면 그만이다. 이는 매우 직관적이며 데이터 중복이나 이상 현상도 발생하지 않는다.

### 2. ABC+D Star 패턴 (뮤지션 예제)

조금 더 복잡한 예시를 보자. 콘서트(Concert), 뮤지션(Musician), 악기(Instrument)가 있다. "어떤 뮤지션이 특정 콘서트에서 특정 악기를 연주했다"는 사실을 기록해야 한다. 초보자들은 이 3가지를 묶어 하나의 테이블로 만들려고 하겠지만, 숙련된 엔지니어라면 여기에 숨겨진 개념인 **Performance(공연)** 라는 앵커를 도출해낸다.



Performance 테이블은 Concert, Musician, Instrument와 각각 1:N 관계를 맺는다. 여기서 물리적 설계의 핵심은 Primary Key를 어떻게 잡느냐로 넘어간다.

만약 "한 뮤지션은 같은 콘서트에서 특정 악기를 한 번만 연주한다"는 유니크 제약조건이 비즈니스 요구사항에 있다면, 우리는 두 가지 선택을 할 수 있다.

**Synthetic ID를 사용하는 경우:**
```sql
CREATE TABLE performances (
    id INTEGER NOT NULL PRIMARY KEY,
    concert_id INTEGER NOT NULL,
    musician_id INTEGER NOT NULL,
    instrument_id INTEGER NOT NULL,
    UNIQUE (concert_id, musician_id, instrument_id)
);
```

**Composite Primary Key를 사용하는 경우:**
```sql
CREATE TABLE performances (
    concert_id INTEGER NOT NULL,
    musician_id INTEGER NOT NULL,
    instrument_id INTEGER NOT NULL,
    PRIMARY KEY (concert_id, musician_id, instrument_id)
);
```

내 경험상 대부분의 모던 애플리케이션에서는 ORM과의 호환성 및 확장성을 위해 첫 번째 방식(Synthetic ID + Unique Constraint)을 선호한다. 이 역시 5NF를 몰라도 상식적인 수준에서 도출할 수 있는 설계다.

## Hacker News의 반응과 나의 시선

이 글이 Hacker News에 올라왔을 때 커뮤니티의 반응은 매우 뜨거웠다. 특히 인상 깊었던 몇 가지 의견과 내 생각을 덧붙인다.

- **"Normalize till it hurts, then denormalize till it works!"**
이보다 데이터베이스 설계를 완벽하게 요약한 문장이 있을까? 정규화는 데이터 무결성을 위해 필수적이지만, 4NF 이상의 과도한 정규화는 필연적으로 수많은 Junction table과 복잡한 JOIN을 낳는다. OLTP 환경에서는 데이터의 정합성이 최우선이므로 정규화가 중요하지만, 조회 성능이 중요한 상황(또는 OLAP 환경)에서는 의도적인 반정규화(Denormalization)가 훨씬 실용적인 해결책이다.

- **학술적 번호 매기기의 무용론**
한 유저는 1NF부터 5NF까지 번호를 매기는 방식이 그저 '객관식 시험 문제를 내기 좋게 만들기 위함'이라고 꼬집었다. 전적으로 동의한다. 1NF~3NF(또는 BCNF)까지는 "데이터를 중복 저장하지 마라"는 상식적인 엔지니어링 원칙으로 귀결되지만, 그 이상은 B-Tree 인덱스 비용과 쿼리 복잡도만 기하급수적으로 높일 뿐이다.

- **UUID와 Synthetic Key 논쟁**
복합키(Composite Key) 대신 UUID나 BigInt 기반의 단일 PK를 사용하는 것에 대한 논쟁도 있었다. 어떤 이들은 이것이 정규화를 망친다고 비판하지만, 분산 시스템과 마이크로서비스 아키텍처(MSA)가 지배적인 현대 백엔드 환경에서 UUID v7 같은 정렬 가능한 Synthetic Key의 사용은 선택이 아닌 생존에 가깝다.

## 최종 결론 (Verdict)

그렇다면 제5정규형(5NF)은 완전히 쓰레기인가? 그렇지는 않다. 정규형 이론은 우리가 설계한 스키마에서 발생할 수 있는 데이터 이상(Anomaly)을 감지하는 훌륭한 '어휘(Vocabulary)'를 제공한다. 

하지만 이를 엔지니어링의 절대적인 '스펙(Specification)'으로 삼아서는 안 된다. Alexey Makhotkin의 지적처럼, 이해할 수 없는 테이블을 쪼개는 방식의 교육은 이제 멈춰야 한다. 우리는 비즈니스 요구사항을 정확히 파악하고, 명확한 논리적 엔티티(Anchor)와 관계(Link)를 정의한 뒤, 성능과 정합성 사이의 트레이드오프를 고려하여 물리적 테이블을 설계해야 한다.

데이터베이스는 학자들의 장난감이 아니다. 수천만 건의 트래픽을 견뎌내야 하는 프로덕션 환경에서는 학술적 완벽함보다 직관성과 성능이 훨씬 더 가치 있다는 사실을 잊지 말자.
