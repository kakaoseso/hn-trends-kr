웹 브라우저에서 하드웨어 센서에 접근하는 Web API가 발전하면서 재미있는 프로젝트들이 많이 나오고 있습니다. 최근 Hacker News에서 흥미로운 프로젝트를 하나 발견했습니다. 마이크 대신 스마트폰의 **가속도계(Accelerometer)** 를 이용해 기타를 튜닝하는 웹앱입니다.

## 기발한 접근법: 소리가 아닌 진동
원리는 꽤 직관적입니다. 스마트폰을 기타 바디에 꽉 누르고 줄을 튕기면, 기타의 진동이 스마트폰의 IMU 센서로 전달됩니다. 이 진동 데이터를 분석해 주파수를 찾아내고 피치를 판별하는 것이죠. 마이크를 쓸 때 발생하는 주변 소음 문제를 물리적인 진동으로 우회하겠다는 발상 자체는 엔지니어로서 꽤 박수를 쳐줄 만합니다. 저 역시 과거에 오디오 프로세싱 파이프라인의 노이즈 캔슬링 문제로 골머리를 앓았던 적이 있는데, 아예 입력 소스를 바꿔버린다는 접근은 신선했습니다.

## 나이퀴스트 정리와 하드웨어의 한계
하지만 실무를 뛰는 엔지니어라면 여기서 바로 데이터의 **Sample Rate(샘플링 속도)** 를 의심해야 합니다. 나이퀴스트-섀넌 샘플링 정리(Nyquist-Shannon sampling theorem)에 따르면, 특정 주파수를 정확히 측정하려면 최소 그 주파수의 2배 이상의 샘플링 속도가 필요합니다.

기타의 가장 낮은 줄인 E2의 주파수는 약 82Hz입니다. 즉, 가속도계가 최소 164Hz 이상으로 데이터를 뽑아줘야 한다는 뜻이죠. HN 댓글의 한 유저도 정확히 이 부분을 지적했습니다. 브라우저에서 제공하는 가속도계의 샘플링 주파수가 50Hz에 불과해 가장 낮은 음조차 측정할 수 없었다는 것입니다. 작성자는 Alias-correction 기법을 사용했다고 언급하지만, 근본적인 데이터 해상도의 부족을 완벽히 메우기는 어렵습니다.

## 보안과 프라이버시: 브라우저가 센서를 제한하는 이유
왜 브라우저는 센서 업데이트 주기를 이렇게 낮게 잡을까요? 하드웨어 스펙이 부족해서가 아닙니다. 바로 **보안(Security)** 때문입니다.

가속도계나 자이로스코프 데이터를 고해상도로 수집하면, 사용자가 책상 위에서 타이핑하는 키보드의 미세한 진동을 분석해 비밀번호를 추론하는 사이드 채널 공격(Side-channel attack)이 가능해집니다. 이 때문에 현대 브라우저들은 의도적으로 센서 API의 폴링 레이트를 낮추거나 데이터에 노이즈를 섞습니다. HN 스레드에서도 이 앱이 얼마나 민감한 정보를 유출할 수 있는지 보여주는 좋은 사례라는 의견이 있었습니다.

## 총평: 프로덕션 레벨인가, 장난감인가?
결론부터 말하자면 이 프로젝트는 기타 튜너라기보다는 기타 디튜너(Detuner)에 가깝습니다. 실제 튜닝용으로 쓰기엔 한계가 명확하죠.

솔직히 처음 이 글을 봤을 때는 '이게 될 리가 없는데'라며 회의적이었습니다. 하지만 이런 사이드 프로젝트는 우리에게 꽤 중요한 인사이트를 줍니다. 브라우저 샌드박스 안에서 하드웨어를 다룰 때 발생하는 Trade-off, 즉 **기능성(Capability)과 보안(Security)의 충돌** 을 아주 명확하게 보여주기 때문입니다. 가끔은 이런 엉뚱한 해킹이 기술의 근본적인 제약을 다시 한번 상기시켜주는 좋은 교보재가 됩니다.
