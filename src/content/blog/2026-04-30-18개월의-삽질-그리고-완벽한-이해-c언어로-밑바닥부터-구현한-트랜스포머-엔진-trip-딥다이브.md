요즘 개발 씬을 보면 다들 OpenAI API를 호출하거나 `pip install transformers` 한 줄 치고는 자신이 LLM을 다룬다고 말하는 경우가 많다. 우리는 어느새 트랜스포머 아키텍처를 거대한 블랙박스처럼 취급하고 있다. 하지만 Residual stream에서 Gradient가 정확히 어떻게 흐르는지, Attention 행렬 곱셈이 메모리에서 어떻게 처리되는지 진짜로 이해하고 있는 엔지니어가 몇이나 될까?

이런 갈증을 느끼던 차에, 해커뉴스에서 굉장히 흥미로운 프로젝트를 하나 발견했다. 이름은 **TRiP** 이다. 한 명의 개발자가 18개월 동안 점심시간과 주말을 갈아 넣어, 오직 C언어만으로 밑바닥부터 짜올린 트랜스포머 엔진이다.

## 왜 굳이 C언어로 바퀴를 다시 발명했을까?

이 프로젝트는 llama.cpp와 경쟁하려는 것이 아니다. 철저하게 **교육과 본질적 이해** 를 목적으로 한다. 파이토치(PyTorch)나 텐서플로우(TensorFlow) 같은 프레임워크 없이, 심지어 외부 선형대수 라이브러리도 없이 오직 C언어 배열과 포인터만으로 Inference, Training, Tokenizer, 심지어 Vision 처리까지 모두 구현했다.

솔직히 처음엔 '또 누군가 Karpathy의 llma2.c를 포크해서 적당히 기능 몇 개 붙였겠지'라고 생각하며 회의적이었다. 하지만 코드를 뜯어보고 나니 내 생각이 완전히 틀렸음을 인정할 수밖에 없었다. 이 개발자는 BPE Tokenizer의 Vocabulary 생성부터 AdamW 옵티마이저, Backpropagation까지 전부 직접 짰다.

## TRiP의 핵심 기술적 디테일

이 프로젝트가 기술적으로 우아한 이유는 14,000줄 남짓한 코드가 단 7개의 파일로 깔끔하게 모듈화되어 있다는 점이다.

- **Architecture:** Llama2, Gemma 1.0/1.1, GPT-2는 물론이고, Vision-Language 모델인 PaliGemma까지 지원한다.
- **Memory Management:** 제한된 하드웨어 환경을 위해 `mmap` 을 활용하여 거대한 가중치(Weights)를 RAM에 효율적으로 매핑한다.
- **No Dependencies:** CMake도, 파이썬도 필요 없다. 그저 `make` 명령어 하나면 끝난다.

가장 인상 깊었던 부분은 `math.c` 파일의 구조다. 저자는 모든 수학 연산을 **Forward와 Backward 쌍** 으로 나란히 배치했다.

```c
// math.c 내부의 개념적 구조
void rmsnorm(float* out, float* in, float* weight, int size) {
    // Forward pass 구현
}

void rmsnorm_backward(float* din, float* dout, float* in, float* weight, int size) {
    // Backward pass (Gradient 계산) 구현
}
```

이 구조는 그 어떤 교과서보다 훌륭하다. 코드를 위에서 아래로 읽는 것만으로도 Chain rule이 어떻게 적용되어 Gradient가 역전파되는지 직관적으로 이해할 수 있다. 2018년쯤 우리가 처음 커스텀 메모리 할당자를 C로 작성하며 OS의 메모리 관리를 깨우치던 때의 희열이 떠올랐다.

## FP32 vs BF16: 하드웨어의 냉혹한 현실

저자가 README에 남긴 흥미로운 코멘트가 있다. 바로 **"CPU에서는 bfloat16이나 float16보다 float32가 항상 더 높은 성능을 낸다"** 는 사실에 본인도 놀랐다는 것이다.

이건 시니어 엔지니어라면 한 번쯤 겪어봤을 하드웨어 아키텍처의 현실이다. 오늘날의 일반적인 CPU들은 AVX512-BF16 같은 최신 인스트럭션 셋이 없는 한, BF16 연산을 네이티브로 처리하지 못한다. 결국 내부적으로 캐스팅(Casting) 오버헤드가 발생하기 때문에 메모리 대역폭에서 이득을 보더라도 연산 속도에서 깎아먹어 전체 Throughput 이 떨어지는 것이다. 이런 디테일은 직접 밑바닥부터 프로파일링을 해보지 않으면 절대 알 수 없는 귀중한 인사이트다.

## AI가 짜준 코드가 없는 순수한 엔지니어링

해커뉴스 스레드에서 가장 많은 공감을 받은 댓글은 다음과 같다.

> "thats super cool congrats on the nice project and for not using some ai bot for it" (정말 멋진 프로젝트입니다. AI 봇을 쓰지 않고 직접 만들었다니 축하합니다.)

나 역시 이 의견에 격하게 동의한다. 저자는 JSON 파서나 X11 보일러플레이트 같은 귀찮은 작업에만 일부 AI의 도움을 받았을 뿐, 코어 로직은 100% 직접 타이핑했다고 밝혔다. 요즘처럼 주니어 개발자들이 Copilot의 탭 완성에 의존해 자신이 무엇을 짜는지도 모른 채 PR을 올리는 시대에, 이런 '무식하지만 확실한' 하드코어 엔지니어링은 진정한 근육 기억(Muscle memory)을 만들어준다.

## 결론: 프로덕션용은 아니지만, 최고의 교과서

결론을 내리자면, 이 프로젝트를 프로덕션 환경에 가져다 쓰는 것은 미친 짓이다. Inference 속도나 최적화를 원한다면 vLLM, TensorRT-LLM, 혹은 llama.cpp를 쓰는 것이 맞다.

하지만 **TRiP** 은 장난감이 아니다. 이것은 살아 숨 쉬는 '트랜스포머 해부학 교과서'다. 만약 당신이 매일 LLM API를 호출하면서도 정작 그 내부에서 어떤 행렬 곱셈이 일어나는지 찜찜함이 남아있는 연차 있는 엔지니어라면, 이번 주말에는 넷플릭스 대신 이 프로젝트의 `math.c` 와 `backward.c` 를 정독해 보기를 강력히 권한다. 

기본으로 돌아가는 것만큼 엔지니어의 가슴을 뛰게 하는 것은 없다.
