15년 넘게 엔지니어링 씬에 구르면서 뼈저리게 느낀 진리가 하나 있다. 바로 JIT 컴파일러는 현대 소프트웨어 공학의 정수이자, 동시에 유지보수의 거대한 블랙홀이라는 점이다.

우리가 흔히 쓰는 Python, Lua, Ruby 같은 동적 타입 언어들의 레퍼런스 구현체는 대부분 C 인터프리터로 작성되어 있다. C 인터프리터는 직관적이고 표준을 완벽히 준수하지만, 속도가 느리다. 이를 극복하기 위해 V8, HotSpot, LuaJIT 같은 괴물들이 탄생했지만, 이런 JIT 컴파일러를 밑바닥부터 짜고 유지보수하는 데는 수백, 수천 인년(Person-years)의 비용이 든다. 게다가 언어 스펙이 업데이트될 때마다 JIT 구현체는 뒤처지기 일쑤다.

최근 Hacker News를 뜨겁게 달군 Laurie Tratt의 글 [Retrofitting JIT Compilers into C Interpreters](https://tratt.net/laurie/blog/2026/retrofitting_jit_compilers_into_c_interpreters.html)는 이 딜레마에 대한 매우 우아하고 실용적인 해답을 제시한다. 기존의 낡은 C 인터프리터를 거의 수정하지 않고도 자동으로 JIT 컴파일러로 탈바꿈시키는 프레임워크, **yk** 프로젝트다.

솔직히 처음 제목만 봤을 때는 학계에서 흔히 나오는 그저 그런 장난감 프로젝트인 줄 알았다. 하지만 아키텍처를 파고들수록, 이들이 실무적인 트레이드오프를 얼마나 치열하게 고민했는지 알 수 있었다.

## Meta-tracing: 인터프리터를 추적하는 인터프리터

JIT를 자동으로 생성하려는 시도가 이번이 처음은 아니다. PyPy의 RPython이나 GraalVM의 Truffle 같은 훌륭한 선행 사례들이 있다. 하지만 이들의 치명적인 단점은 새로운 호스트 언어(RPython, Java)로 인터프리터를 처음부터 다시 짜야 한다는 것이다. 기존 C 확장 모듈들과의 호환성이 깨지는 순간, 현업 도입은 물 건너간다.

yk는 이 문제를 Meta-tracing 기법으로 해결한다. 사용자의 코드를 추적(Trace)하는 것이 아니라, 사용자의 코드를 실행하고 있는 **C 인터프리터 자체의 동작** 을 추적한다. 

이 마법의 핵심은 LLVM을 포크한 `ykllvm`이다. C 인터프리터를 이 커스텀 컴파일러로 빌드하면, 컴파일러는 바이너리 내부에 LLVM IR(Intermediate Representation)을 직렬화하여 숨겨둔다. 그리고 런타임에 핫 루프(Hot loop)를 감지하면, 인터프리터가 실행한 C 코드의 베이직 블록(Basic block) ID들을 추적하여 해당하는 LLVM IR 조각들을 이어붙인다. 이렇게 모인 IR을 런타임에 최적화하고 기계어로 컴파일하는 방식이다.

## 50줄의 코드 수정으로 얻어낸 기적

내가 이 프로젝트에서 가장 감탄한 부분은 인터프리터 작성자가 JIT 최적화기에게 힌트를 주는 방식이다. C 인터프리터의 코드는 범용적이기 때문에 JIT가 섣불리 최적화하기 어렵다. yk는 단 몇 줄의 어노테이션으로 이 한계를 돌파한다.

대표적인 것이 `yk_promote`와 `yk_idempotent`다.

```c
// 기존 C 코드
case OP_INT:
    push(constant_pool[GET_OPVAL(i)]);
    pc++;
    break;
```

위 코드를 아래와 같이 살짝 수정해보자.

```c
// yk 힌트가 추가된 코드
case OP_INT:
    push(yk_promote(constant_pool[GET_OPVAL(i)]));
    pc++;
    break;
```

`yk_promote`는 런타임 값을 추적(Trace) 단계에서 상수로 박아버리고(Burn-in) Guard를 세우는 역할을 한다. 여기에 더해 명령어(Opcode)를 읽어오는 함수를 `yk_idempotent` (멱등성 보장)로 선언하면, JIT 컴파일러는 "아, 이 주소에서 읽어오는 명령어는 항상 같구나"라고 판단하여 명령어 디코딩 로직 전체를 상수 폴딩(Constant folding)으로 날려버린다.

저자들에 따르면 PUC Lua 코드베이스에서 단 50줄을 수정하고 400줄을 추가한 것만으로, 벤치마크에서 평균 2배, 특정 연산에서는 4배 이상의 성능 향상을 이끌어냈다고 한다.

## LuaJIT의 유산: Backwards Code Generation

엔지니어로서 짜릿함을 느꼈던 또 다른 포인트는 기계어 생성 방식이다. yk는 LuaJIT의 아이디어를 차용하여 Trace를 **뒤에서부터 앞으로(Backwards)** 순회하며 기계어를 생성한다.

일반적인 포워드(Forwards) 방식에서는 레지스터 할당(Register allocation)이 끔찍하게 복잡해진다. 변수의 수명을 예측하기 위해 별도의 패스가 필요하기 때문이다. 하지만 뒤에서부터 코드를 생성하면, 변수가 처음 등장하는 순간이 곧 그 변수의 마지막 사용 시점이 된다. 

- **명확한 레지스터 수명:** 레지스터 할당이 직관적이고 빨라진다.
- **암묵적인 Dead Code Elimination:** 부수 효과(Side effect)가 없는 명령어가 뒤에서 사용되지 않았다면, 기계어를 생성할 때 그냥 무시하면 된다. 단 10줄의 코드로 훌륭한 DCE가 구현된다.

이런 실용적인 엔지니어링 결단이 프로젝트의 격을 높여준다.

## Hacker News의 반응과 현실적인 비판

HN 스레드([링크](https://news.ycombinator.com/item?id=47777897))를 보면 커뮤니티의 반응도 뜨겁다. 많은 엔지니어들이 이 기술이 유저의 코드를 건드리는 것인지, 인터프리터를 건드리는 것인지 헷갈려 했는데, 이는 Meta-tracing의 개념이 그만큼 생소하기 때문이다.

가장 날카로운 지적 중 하나는 성능과 유지보수에 관한 것이었다. "겨우 2배 빨라진다고? LuaJIT은 5~10배 빠른데?"라는 비판이다. 

맞는 말이다. 순수 성능만 보면 장인정신으로 한 땀 한 땀 어셈블리를 깎아 만든 LuaJIT을 이길 수 없다. 하지만 간과해선 안 될 사실이 있다. **LuaJIT은 13년 전 버전인 Lua 5.1에 갇혀 있다.** 반면 yk의 메인테이너는 Lua 5.4에서 5.5로 버전을 올리는데 단 2시간이 걸렸다고 한다. 

현업에서 CTO나 리드 엔지니어로 일해본 사람이라면 알 것이다. 레거시 호환성을 100% 유지하면서 최신 언어 스펙을 즉각적으로 따라갈 수 있는 2배 빠른 런타임은, 호환성을 포기해야 하는 10배 빠른 런타임보다 비즈니스 가치가 훨씬 높다.

다만, LLVM을 포크해서 `ykllvm`을 유지해야 한다는 점은 심각한 기술 부채 리스크다. HN 유저 한 명도 "왜 LLVM 패스로 만들지 못했나?"라고 질문했는데, Stackmap의 구조적 한계와 호환성 문제 때문이라고 답했다. LLVM 업스트림이 빠르게 변하는 상황에서 이 포크 버전을 소규모 팀이 계속 유지보수할 수 있을지는 미지수다.

## 총평: 아직은 Alpha, 하지만 언어 구현의 미래

현재 yk는 Alpha 단계다. Deoptimization 과정에서 AOT 컴파일된 C 코드로 돌아가기 위해 Shadow stack을 쓰고 Stackmap을 억지로 맞추는 등, 엣지 케이스에서 터질 만한 시한폭탄들이 아직 존재한다. 당장 프로덕션의 CPython이나 Ruby에 적용하기엔 무리가 있다.

하지만 이 프로젝트가 증명한 가치는 명확하다. 수많은 도메인 특화 언어(DSL)나 레거시 C 인터프리터들이 큰 비용 없이 JIT의 혜택을 누릴 수 있는 새로운 설계 공간(Design space)을 열어젖혔다.

매번 "AI가 코드를 다 짜준다"는 식의 피상적인 뉴스가 범람하는 요즘, 컴파일러 최적화와 레지스터 할당의 밑바닥을 파고드는 이런 딥 다이브 프로젝트를 보는 것은 엔지니어로서 큰 즐거움이다. C 인터프리터를 다루거나 언어 런타임에 관심이 있는 시니어 엔지니어라면, 주말에 시간을 내어 yk의 소스코드를 훑어보길 강력히 권한다.
