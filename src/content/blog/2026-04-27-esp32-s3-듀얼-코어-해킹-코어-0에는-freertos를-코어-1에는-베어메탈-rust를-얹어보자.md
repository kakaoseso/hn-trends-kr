IoT 하드웨어나 임베디드 프로젝트를 리드하다 보면 항상 딜레마에 빠지게 됩니다. Wi-Fi나 BLE 같은 복잡한 통신 스택은 제조사가 제공하는 C 기반의 SDK(예: Espressif의 ESP-IDF)가 압도적으로 안정적입니다. 수년간 수많은 엣지 케이스를 거치며 다듬어졌기 때문이죠. 반면, 타이밍에 민감한 오디오 처리나 복잡한 비즈니스 로직은 메모리 안전성과 추상화가 뛰어난 Rust로 작성하고 싶어집니다.

일반적인 해결책은 C SDK 위에 Rust 래퍼를 씌우는 것입니다. 하지만 이 방식은 생태계가 성숙하지 않은 칩셋에서는 새벽 2시에 알 수 없는 Undefined Behavior와 싸워야 하는 지름길이기도 합니다. 

최근 Tingou Wu의 블로그에 올라온 [Running Bare-Metal Rust Alongside ESP-IDF on the ESP32-S3's Second Core](https://tingouw.com/blog/embedded/esp32/run_rust_on_app_core)라는 글은 이 문제에 대해 아주 우아하고 '해커다운' 접근법을 제시했습니다. 바로 ESP32-S3의 **듀얼 코어** 를 물리적으로 분리해서 사용하는 것입니다.

## 왜 그냥 FreeRTOS Task를 코어 1에 고정하지 않는가?

이 글을 읽고 가장 먼저 든 생각은 '그냥 `xTaskCreatePinnedToCore` API를 써서 Rust 함수를 코어 1에 할당하면 되는 거 아닌가?' 였습니다. Hacker News 스레드에서도 한 유저가 정확히 이 점을 지적했죠. 우선순위를 최대로 높이고 다른 태스크를 코어 0으로 몰아넣으면 되지 않느냐고 말입니다.

하지만 하드 리얼타임(Hard Real-time) 시스템을 다뤄본 엔지니어라면 이게 말처럼 쉽지 않다는 걸 압니다. FreeRTOS 스케줄러가 코어 1에서 돌고 있는 이상, 시스템 틱(Tick)이나 OS 레벨의 인터럽트로 인해 마이크로초 단위의 Jitter가 발생할 수밖에 없습니다. 

저자는 `CONFIG_FREERTOS_UNICORE` 옵션을 활성화하여 FreeRTOS를 코어 0에 완전히 가둬버립니다. 그리고 OS가 전혀 없는 텅 빈 코어 1을 하드웨어 레지스터 조작으로 직접 깨워 베어메탈 `no_std` Rust 코드를 실행시킵니다. OS의 간섭이 0%인 완벽한 격리 환경을 구축한 것이죠.

## 기술적 구현: 어떻게 두 세계를 연결하는가

이 아키텍처를 구현하기 위한 핵심 단계들은 꽤나 흥미롭습니다. 임베디드 시스템의 로우레벨 동작을 이해하는 데 아주 좋은 교보재입니다.

- **메모리 예약:** 코어 1은 OS가 없으므로 힙 할당자도 없습니다. C 코드 쪽에서 `SOC_RESERVE_MEMORY_REGION` 매크로를 사용해 ESP-IDF가 절대 건드리지 못하는 128KB의 물리적 SRAM 영역을 강제로 떼어냅니다.
- **어셈블리 트램펄린:** 코어가 리셋에서 깨어나면 스택 포인터가 없습니다. 아주 짧은 Xtensa 어셈블리 코드(`app_core_trampoline.S`)를 작성해 레지스터 `a1`에 예약된 메모리의 최상단 주소를 꽂아주고 Rust의 엔트리 포인트로 점프합니다.
- **원자적(Atomic) 공유 메모리:** 두 코어는 완전히 다른 패러다임으로 돌고 있지만, 물리적인 메모리 공간은 공유합니다. 뮤텍스(Mutex) 같은 OS 레벨의 락은 사용할 수 없으므로, `AtomicU32`를 이용해 Lock-free 방식으로 데이터를 주고받습니다.

## 하이라이트: 런타임 Hot-Swappable Rust 바이너리

이 글에서 제가 가장 감탄한 부분은 파트 1의 정적 링킹을 넘어, 파트 2에서 구현한 **런타임 로딩** 입니다.

단순히 C와 Rust를 컴파일 타임에 링킹하는 것은 펌웨어를 수정할 때마다 전체를 다시 빌드해야 하는 불편함이 있습니다. 저자는 Rust 코드를 완전히 독립적인 실행 파일(Standalone Binary)로 빌드하고, 이를 플래시 메모리의 별도 파티션(`rust_app`)에 저장합니다.

이후 코어 0의 C 부트로더가 ESP32의 MMU(Memory Management Unit)를 제어하여 이 플래시 파티션을 가상 메모리 주소(`0x42400000`)에 매핑한 뒤 코어 1을 깨웁니다. 

이게 무슨 뜻일까요? **메인 시스템(Wi-Fi, BLE, FreeRTOS)을 재부팅하거나 펌웨어 업데이트를 하지 않고도, 코어 1에서 돌아가는 핵심 비즈니스 로직(Rust)만 런타임에 교체(Hot-swap)할 수 있다는 의미입니다.** 클라우드 환경의 마이크로서비스 아키텍처나 FaaS(Function-as-a-Service)를 3달러짜리 마이크로컨트롤러 위에서 구현한 셈입니다.

## 커뮤니티의 반응과 나의 생각

이 접근법은 사실 업계에서 완전히 새로운 것은 아닙니다. Hacker News의 한 코멘터가 언급했듯, 많은 실무자들은 `ESP-Hosted-MCU` 펌웨어를 올린 ESP32를 단순한 '라디오 코프로세서(Radio Co-processor)'로 쓰고, 메인 로직은 STM32 같은 별도의 칩에서 베어메탈이나 RTOS로 돌리는 방식을 선호합니다. 

저 역시 2018년도에 비슷한 구조로 통신 칩과 DSP 칩을 SPI로 연결해 아키텍처를 분리했던 경험이 있습니다. 하지만 Tingou Wu의 방식은 이 물리적인 두 개의 칩을 ESP32-S3라는 단일 실리콘 위에서 논리적으로 완벽히 분리해 냈다는 점에서 아키텍처적으로 훨씬 우아하고 비용 효율적입니다.

물론 비판적인 시각도 존재합니다. 통신 스택을 별도 코어나 칩으로 분리하는 트렌드가 칩 제조사들(예: STMicroelectronics의 STM32WB 등)로 하여금 네트워크 스택을 암호화된 클로즈드 소스 바이너리 블롭(Blob)으로 숨기기 좋은 핑계거리가 되고 있다는 지적은 뼈아픕니다. 오픈소스를 지향하는 엔지니어들에게는 블랙박스가 하나 더 생기는 셈이니까요.

## 결론: 프로덕션 레벨에서 쓸만한가?

솔직히 말해서, 이 구조를 당장 내일 출시할 상용 프로덕트에 적용하겠냐고 묻는다면 저는 주저할 것입니다. 

가장 큰 문제는 **디버깅과 에러 복구** 입니다. 코어 1의 베어메탈 Rust 로직에서 패닉이 발생하면 현재 코드는 무한 루프에 빠집니다. OS가 없으니 코어 0에게 "나 죽었어"라고 우아하게 알려줄 방법이 기본적으로는 없습니다. 상용화하려면 코어 0과 코어 1 사이에 견고한 락프리 링 버퍼(Ring Buffer) 채널을 구축하고, 하드웨어 왓치독(Watchdog)을 세밀하게 조율해야 할 것입니다.

하지만 기술적인 가능성 측면에서는 별 5개를 주고 싶습니다. 임베디드 환경에서 C의 안정성(통신 스택)과 Rust의 성능 및 안전성(비즈니스 로직)을 타협 없이 모두 가져가고 싶은 시니어 엔지니어라면, 주말 장난감 프로젝트로 이보다 더 흥미로운 주제는 없을 것입니다. 

결국 훌륭한 엔지니어링이란 주어진 제약(단일 칩, 부족한 Rust 생태계) 안에서 하드웨어의 특성(듀얼 코어, MMU)을 영리하게 비틀어 최적의 해답을 찾아내는 과정이니까요.
