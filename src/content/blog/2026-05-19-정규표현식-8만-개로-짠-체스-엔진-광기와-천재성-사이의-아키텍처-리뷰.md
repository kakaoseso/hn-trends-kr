"어떤 문제에 직면했을 때 '아, 정규표현식을 쓰면 되겠군'이라고 생각한다면, 이제 당신에겐 두 개의 문제가 생긴 것이다."

이 낡은 농담을 모르는 시니어 엔지니어는 없을 겁니다. 우리는 로그를 파싱하거나 입력값을 검증할 때 정규표현식(Regex)을 씁니다. 그리고 종종 그 복잡성에 혀를 내두르죠. 그런데 최근 보안 연구자이자 기행(?)으로 유명한 Nicholas Carlini가 무려 84,688개의 정규표현식을 순차적으로 실행해 동작하는 2-ply Minimax 체스 엔진을 공개했습니다.

처음 이 소식을 접했을 때 제 반응은 "도대체 왜?"였습니다. 하지만 그의 블로그 포스트를 끝까지 읽고 난 후, 저는 이 프로젝트가 단순한 장난을 넘어 컴파일러 설계, State Machine, 그리고 성능 최적화에 대한 깊은 통찰을 담고 있다는 것을 깨달았습니다. 오늘은 이 미친 프로젝트가 도대체 '어떻게' 동작하는지, 그리고 우리가 여기서 무엇을 배울 수 있는지 Deep Dive 해보겠습니다.

## Regex를 Instruction Set으로 탈바꿈시키다

정규표현식으로 체스를 둔다니, 대체 어떻게 상태(State)를 저장하고 연산(Compute)을 할까요?

Carlini의 접근법은 매우 우아하면서도 무식합니다. 컴퓨터의 모든 상태(스택과 변수들)를 거대한 **단일 문자열** 로 표현하는 것입니다. 그리고 각 정규표현식은 이 상태 문자열을 입력받아 변형을 가한 뒤 반환하는 하나의 '명령어(Instruction)' 역할을 합니다.

가장 기본적인 `push` 명령어의 구현을 보시죠.

```python
def push(const):
    return [(r"(%%\n#stack:\n)",
             r"\g<1>"+const+r"\n")]
```

단순합니다. 상태 문자열의 맨 위에 있는 `%%\n#stack:\n` 앵커를 찾아서 그 바로 아래에 상수값을 밀어 넣습니다. `pop`이나 변수를 읽어오는 `lookup`도 마찬가지 방식으로 동작합니다.

하지만 할당(Assignment)부터는 골치가 아파집니다. 변수가 이미 존재하는 경우와 새로 만들어야 하는 경우를 분기(Branch)해야 하기 때문이죠. 여기서 그는 정규표현식의 매칭 실패 메커니즘을 이용해 **Branch-Free Conditional Execution** 을 구현합니다. 마치 ARM 아키텍처의 Predicated Execution을 보는 듯한 느낌이었습니다. 모든 명령어는 조건부로 실행되며, 상태 문자열에 특정 마커(예: `%%`)가 있을 때만 정규표현식이 동작하게 만듭니다.

## SIMD를 Regex로 구현하다니

이 글을 읽으며 제가 가장 감탄했던 부분은 바로 다중 스레드(Multi-threading)와 SIMD(Single-Instruction Multiple-Data)의 구현입니다.

체스 엔진은 수많은 경우의 수(Board States)를 탐색해야 합니다. 보통은 재귀적인 루프를 돌며 이를 처리하지만, 정규표현식의 순차적 실행만으로는 루프를 흉내 내기 어렵습니다. 대신 그는 정규표현식의 **Global Replace** 특성을 기가 막히게 활용합니다.

상태 문자열 내에 여러 개의 스레드(각각 다른 체스판 상태를 나타냄)를 동시에 텍스트로 늘어놓습니다. 그리고 `%%` 마커를 가진 모든 스레드에 대해 정규표현식 치환을 한 번에 실행합니다.

```python
def fork_inactive(tag):
    return [(r"%%\n([^%]*)",
             r"%%\n\1" + "%"+tag+r"\n\1")
    ]
```

이 `fork` 명령어가 실행되면, 현재 활성화된 모든 상태가 복제되어 하나는 계속 실행되고 다른 하나는 대기 상태(`%waiting`)로 빠집니다. 폰(Pawn) 8개의 이동 가능성을 계산할 때 루프를 도는 대신, 상태를 8개로 Fork한 뒤 단 한 번의 이동 검사 Regex를 전체 상태에 동시에 때려버립니다. 이건 정말이지 변태적일 정도로 아름다운 발상입니다.

## Symbolic Execution으로 컴파일러 만들기

당연히 84,688개의 정규표현식을 손으로 짠 것은 아닙니다. Carlini는 Python으로 일종의 매크로 어셈블러를 만들었습니다.

일반적인 AST 기반 파서 대신 **Symbolic Execution(기호 실행)** 을 사용했습니다. Python 코드를 실행하되, 실제 값을 계산하는 대신 어떤 연산이 수행되는지를 추적(Trace)하여 CallTree를 구성합니다. `if`문을 만나면 두 갈래의 분기를 모두 추적한 뒤, 이를 Regex의 조건부 마커(`%true`, `%false`)로 변환하고 나중에 `reactivate` 명령어로 합칩니다(Merge).

제가 2018년에 복잡한 분산 트랜잭션의 보상 로직(Saga pattern)을 설계할 때 상태 머신을 이런 식으로 펼쳐서 로깅했던 기억이 나더군요. 복잡한 Control Flow를 Data Flow로 치환하여 해결하는 전형적이고 강력한 패턴입니다.

## 시니어 엔지니어의 시선: 최적화가 진짜 예술

이 프로젝트가 단순한 장난감에서 그치지 않는 이유는 성능 최적화 과정에 있습니다. 초기 버전은 한 수를 계산하는 데 30분이 걸렸지만, 최종 버전은 1~10초로 줄였습니다. 무려 100배의 성능 향상입니다.

- **Garbage Collection:** 변수를 계속 복제하면 상태 문자열이 10GB까지 팽창합니다. 그는 사용이 끝난 변수를 즉각적으로 삭제하여 메모리를 300MB로 억제했습니다. O(n) 스캔 비용이 드는 Regex 환경에서 상태 크기를 줄이는 것은 핵심적인 튜닝입니다.
- **Regex Fail-Fast 유도:** 조건문에서 `True`/`False`를 매칭할 때, 앞에 `\n#stack:\n`을 명시적으로 요구하는 사소한 변경(`\n` 하나 추가)으로 성능을 2배 끌어올렸습니다. 정규표현식 엔진이 무의미한 변수 값 내부의 True/False 문자열을 빠르게 건너뛰게(Backtracking 최소화) 만든 것이죠. 프로덕션 환경에서 Regex 성능 이슈를 겪어본 분이라면 이 쾌감을 아실 겁니다.
- **Loop Unrolling & Instruction Fusion:** 폰을 찾는 등 반복적인 스캔 작업을 여러 개의 명령어로 쪼개지 않고, 하나의 거대한 Regex로 병합(Fusion)하여 처리 속도를 극대화했습니다.

## Hacker News의 반응과 한계

[Hacker News 스레드](https://news.ycombinator.com/item?id=48136909)에서도 반응은 폭발적입니다. 누군가는 "마치 열병을 앓을 때 꾸는 꿈(fever dream) 같다"고 표현했고, "NFA(비결정적 유한 오토마타)가 텍스트북 밖에서 이렇게 구현된 건 처음 본다"며 경악하는 사람도 있었습니다.

물론 한계는 명확합니다. 2-ply(자신과 상대의 수 딱 한 번씩만 예측) 엔진이기 때문에 실력은 처참합니다. HN의 한 유저가 지적했듯, 엔진은 체크메이트를 방어하는 3-ply 이상의 수를 내다보지 못해 아주 단순한 함정(Mate-in-1)에도 쉽게 무너집니다. 또한 불법적인 수(Illegal move)를 생성하지 않기 위해 탐색 공간을 제한한 탓에, 온전한 Minimax 알고리즘이라고 보기도 어렵습니다.

## 결론

솔직히 말해, 이 기술을 현업 프로덕션에 쓸 일은 영원히 없을 겁니다. (만약 여러분의 팀 누군가가 Regex로 비즈니스 로직 분기를 태우려 한다면 당장 키보드를 뺏으십시오.)

하지만 Carlini의 Regex Chess는 우리에게 중요한 사실을 일깨워줍니다. 우리가 매일 사용하는 평범한 도구(Regex)의 극한을 밀어붙였을 때, 그 기저에 깔린 컴퓨터 과학의 원리(상태 기계, SIMD, 심볼릭 실행)가 얼마나 아름답게 교차하는지를요.

때로는 아무런 실용성이 없는 '순수한 잉여력'이 엔지니어의 시야를 가장 크게 넓혀주기도 합니다. 주말에 시간 내어 그의 [GitHub 저장소](https://github.com/carlini/regex-chess)를 둘러보시길 강력히 권합니다. 아마 당분간 정규표현식을 볼 때마다 체스판이 아른거리는 부작용은 감수하셔야 할 겁니다.
