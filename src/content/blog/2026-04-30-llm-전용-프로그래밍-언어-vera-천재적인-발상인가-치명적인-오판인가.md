요즘 주변 시니어 엔지니어들이나 CTO들과 이야기해보면, 다들 코드 작성 시간의 상당 부분을 Cursor나 GitHub Copilot 같은 AI 에이전트와 씨름하는 데 쓰고 있습니다. LLM이 짜준 코드를 리뷰하고, 사이드 이펙트를 잡고, 미묘한 버그를 고치는 게 일상이 되었죠. 이런 상황에서 최근 Hacker News를 뜨겁게 달군 프로젝트 하나가 제 눈길을 끌었습니다. 바로 **오직 기계(LLM)가 작성하도록 설계된 프로그래밍 언어**, Vera입니다.

처음 이 프로젝트의 설명을 읽었을 때, 저는 신선한 충격과 동시에 깊은 의구심을 느꼈습니다. 15년 넘게 코드를 짜고 시스템을 설계해 온 제 관점에서, Vera가 제시하는 비전은 절반의 천재성과 절반의 순진함이 섞여 있었습니다.

## Vera가 제시하는 새로운 패러다임

Vera의 철학은 꽤 명확합니다. 하드웨어의 제약에서 Assembly가 나왔고, 생산성의 필요성에서 Python이 나왔듯, 이제 LLM이 코드를 작성하는 시대라면 언어 자체도 LLM에 맞춰 진화해야 한다는 것입니다.

Vera의 창시자는 LLM이 겪는 가장 큰 문제가 '문법'이 아니라 '규모에 따른 일관성(Coherence over scale)'이라고 주장합니다. LLM은 패턴 매칭 기계이기 때문에 로컬 컨텍스트에서는 그럴싸한 코드를 만들어내지만, 시스템 전체의 상태나 불변성(Invariants)을 유지하는 데는 취약하다는 것이죠.

백문이 불여일견, Vera로 작성된 안전한 나눗셈 함수를 살펴보겠습니다.

```vera
public fn safe_divide(@Int, @Int -> @Int)
requires(@Int.1 != 0)
ensures(@Int.result == @Int.0 / @Int.1)
effects(pure)
{
  @Int.0 / @Int.1
}
```

이 짧은 코드에서 Vera의 가장 극단적인 두 가지 특징을 확인할 수 있습니다.

- **변수명의 부재:** 코드 어디에도 `dividend`나 `divisor` 같은 이름이 없습니다. 대신 `@Int.0`(가장 최근에 바인딩된 Int)나 `@Int.1`(그 이전에 바인딩된 Int) 같은 타입 기반 De Bruijn 인덱스를 사용합니다.
- **강력한 Contract 시스템:** `requires`(사전 조건), `ensures`(사후 조건), `effects`(사이드 이펙트 명시)가 필수입니다. 이 조건들은 Z3 SMT 솔버에 의해 컴파일 타임에 정적으로 검증됩니다.

## 시니어 엔지니어의 시선: 훌륭한 점과 치명적인 오판

이 프로젝트의 코어 디자인을 뜯어보며, 저는 두 가지 상반된 감정을 느꼈습니다.

### 1. Effect 타입과 Z3 검증: 에이전트 시대의 필수 요소

이 부분은 정말 박수를 쳐주고 싶습니다. LLM이 작성한 코드를 Production 환경에 배포할 때 가장 두려운 것은 "이 코드가 내가 모르는 사이에 DB를 날려버리거나 외부 API로 데이터를 유출하지 않을까?" 하는 점입니다.

Vera는 기본적으로 Pure 함수를 강제하며, I/O나 HTTP 통신이 필요할 경우 `effects(<Http>)`처럼 시그니처에 명시해야 합니다. 이는 LLM이 작성한 코드의 Capability를 컴파일러 단에서 수학적으로 증명할 수 있다는 뜻입니다. 에이전트가 작성한 코드를 안전한 샌드박스에서 실행하거나 권한을 제어하는 데 있어, 이런 Effect 타입 시스템과 Contract 기반 설계는 미래의 프로그래밍 언어가 반드시 차용해야 할 훌륭한 패턴입니다.

### 2. 변수명 제거 (De Bruijn indices): 최악의 설계 미스

하지만 변수명을 없애고 구조적 참조(`@Int.0`)를 강제한다는 대목에서는 솔직히 고개를 저을 수밖에 없었습니다. Hacker News 커뮤니티에서도 이 부분에 대한 비판이 쏟아졌죠.

작성자는 논문을 인용하며 LLM이 잘못된 변수명 때문에 혼란을 겪는다고 주장합니다. 하지만 현업에서 수많은 시간을 LLM과 페어 프로그래밍해 본 제 경험은 정반대입니다. LLM은 명시적이고 시맨틱한 변수명(Semantic names)이 존재할 때 훨씬 더 코드를 잘 이해하고 로직을 유지합니다. Go 언어가 LLM에게 유독 강점을 보이는 이유도 모든 것이 명시적으로 텍스트로 드러나기 때문입니다.

게다가 인덱스 기반 참조는 치명적인 물리적 약점이 있습니다. 만약 함수 상단에 변수 하나를 새로 추가한다고 가정해 봅시다. 기존 코드에 있던 모든 `@Int.n`은 이제 `@Int.n+1`로 바뀌어야 합니다. 의미론적으로는 아무것도 변하지 않은 하단 로직들이 전부 수정되어야 한다는 뜻입니다. 이는 LLM의 Context Window를 낭비하게 만들고, Diff 기반의 코드 수정을 지옥으로 만듭니다.

### 3. "학습 데이터"라는 거대한 장벽

HN의 한 유저가 아주 날카로운 지적을 했습니다. LLM이 특정 언어(Python, TypeScript 등)를 잘 다루는 이유는 그 언어의 구조가 LLM 친화적이라서가 아니라, 그저 인터넷에 깔린 수십억 줄의 코드를 학습했기 때문입니다.

Vera처럼 완전히 새로운 패러다임의 언어를 도입하면, LLM은 Zero-shot 상태에서 시작해야 합니다. 매 프롬프트마다 언어의 스펙과 예제 코드를 통째로 Context에 밀어 넣어야 하죠. 이는 토큰 최적화 관점에서 매우 비효율적입니다.

## 결론 (Verdict)

그래서 Vera는 쓸만한가요? 솔직히 말씀드리면, 지금 당장 Production에 도입할 수 있는 도구는 아닙니다. 변수명을 없앤 결정은 LLM의 Attention 메커니즘이 어떻게 동작하는지 깊이 이해하지 못한 결과라고 생각합니다.

하지만 Vera를 단순한 장난감으로 치부해서는 안 됩니다. **AI가 코드를 작성하는 시대에는 '인간이 읽기 좋은 코드'와 '기계가 검증하기 좋은 코드'의 기준이 다를 수 있다** 는 화두를 던졌기 때문입니다. 특히 Z3를 활용한 Contract 검증과 Effect 타입 시스템은, 향후 우리가 TypeScript나 Rust 생태계에서 AI 에이전트를 위한 메타 프레임워크를 설계할 때 반드시 참고해야 할 귀중한 레퍼런스입니다.

기계가 코드를 짜는 시대, 우리는 기계를 통제하기 위해 더 엄격한 수학적 증명(Contracts)으로 돌아가야 할지도 모릅니다. Vera는 비록 완벽한 해답은 아닐지언정, 우리가 던져야 할 올바른 질문을 보여주고 있습니다.
