15년 넘게 엔지니어로 일하면서 수많은 기발한(또는 끔찍한) 해킹 사례들을 봐왔지만, 이번 건은 정말 혀를 내두를 수준입니다. 보통 웹 개발자나 시스템 엔지니어에게 '폰트'란 그저 브라우저가 알아서 렌더링해 주는 정적인 리소스에 불과합니다. 하지만 누군가는 그 안에서 3D 1인칭 슈팅 게임을 돌릴 생각을 했습니다.

최근 Hacker News를 뜨겁게 달군 **TTF-DOOM** 프로젝트는 1991년 Apple이 만든 TrueType 폰트의 Hinting VM(가상 머신) 위에서 Raycasting 엔진을 구현해 냈습니다. 단순히 WASM을 폰트 렌더러에 얹은 `llama.ttf` 같은 프로젝트와는 결이 다릅니다. 이건 순수한 레거시 바이트코드와의 처절한 사투입니다. 

이 프로젝트가 도대체 어떻게 동작하는지, 그리고 이 미친 아이디어를 구현하기 위해 개발자가 어떤 기상천외한 우회 방법(Workaround)을 사용했는지 엔지니어의 시각에서 파헤쳐 보겠습니다.

### 폰트가 튜링 완전(Turing-complete)하다고?

먼저 배경지식이 필요합니다. TrueType 폰트(.ttf)는 단순히 벡터 그래픽의 집합이 아닙니다. 저해상도 모니터 시절, 픽셀 그리드에 글자를 예쁘게 맞추기 위해(Grid-fitting) 폰트 파일 내부에는 일종의 미니 프로그램이 들어갔습니다. 이 Hinting 프로그램은 스택, 저장 공간, 산술 연산, 조건문, 함수 호출을 모두 지원하는 완전한 가상 머신입니다. 네, **튜링 완전** 합니다.

개발자는 이 점을 이용해 단 6.5KB짜리 폰트 파일 안에 DDA(Digital Differential Analyzer) Raycasting 엔진을 집어넣었습니다.

### 아키텍처: 어떻게 폰트가 3D를 렌더링하는가

전체적인 파이프라인은 생각보다 우아하게(?) 분리되어 있습니다.

- **JS (제어 및 로직):** 플레이어의 이동, 적의 AI, 충돌 처리 등을 담당합니다.
- **Font Variation (상태 전달):** JS는 매 프레임 플레이어의 X, Y 좌표와 시야각을 `font-variation-settings`의 커스텀 축(`MOVX`, `MOVY`, `TURN`)을 통해 폰트에 전달합니다.
- **TrueType VM (렌더링):** 폰트 엔진은 전달받은 좌표를 바탕으로 Raycasting을 수행하여 벽의 원근감을 계산합니다.

가장 재밌는 부분은 화면에 그리는 방식입니다. 알파벳 `A`라는 글리프(Glyph)는 16개의 수직 막대(Vertical bar)로 구성되어 있습니다. 폰트 엔진은 Raycasting 결과에 따라 `SCFS` 명령어를 사용해 이 막대들의 위치와 높이를 재배치합니다. 즉, 브라우저가 글자 'A'를 화면에 그리는 순간, 우리는 3D 원근감이 적용된 벽을 보게 되는 것입니다.

개발자는 이 바이트코드를 직접 짜는 대신 C언어와 유사한 커스텀 DSL을 만들고 컴파일러를 작성했습니다.

```c
func raycast(col: int) -> int:
    var ra: int = player_angle + col * 3 - FOV_HALF
    var dx: int = get_cos(ra)
    var dy: int = get_sin(ra)
    ...
```

이 코드는 파서와 코드 제너레이터를 거쳐 `FDEF`, `CALL`, `RS`, `WS`, `SCFS` 같은 TT 바이트코드로 변환되어 `.ttf` 파일에 주입됩니다.

### 1991년산 VM과 싸우기 (진짜 광기)

이 글을 읽는 시니어 엔지니어라면 벌써부터 제약 조건들이 눈에 밟힐 겁니다. 개발자가 직면한 문제들은 과거 임베디드나 J2ME 시절 모바일 게임을 개발할 때 겪던 고통과 정확히 일치합니다.

**1. 부서진 곱셈 명령어와 Fixed-point의 저주**
내부적으로 모든 연산은 F26Dot6 Fixed-point(고정 소수점)로 처리됩니다. 문제는 `MUL` 명령어가 `a * b`가 아니라 `(a * b) / 64`를 반환한다는 점입니다. 즉, `1 * 4`를 하면 `0`이 나옵니다. 개발자는 이틀을 날린 끝에 기가 막힌 우회로를 찾았습니다. `DIV(a, 1)`을 호출하면 `a * 64`가 반환되는 버그성 동작을 이용해, `MUL(DIV(a, 1), b)`로 정상적인 곱셈 결과를 얻어냈습니다. 레거시 시스템에서 흔히 볼 수 있는 전형적인 버그를 버그로 덮는 패턴입니다.

**2. WHILE 루프의 부재와 Call Stack Limit**
TT 바이트코드에는 `WHILE` 명령어가 없습니다. 반복문을 돌리려면 함수를 재귀 호출(`FDEF`)해야 합니다. 그런데 폰트 렌더러인 FreeType은 무한 루프를 막기 위해 Call Stack을 약 64 프레임으로 제한합니다. 16개의 컬럼에 대해 14번의 Ray step을 계산하면 스택이 꽉 차버립니다. 극한의 최적화와 타협이 필요했을 겁니다.

**3. 브라우저 캐싱 무력화**
크롬 같은 모던 브라우저는 Hinting된 글리프를 캐싱합니다. 좌표가 조금 바뀌었다고 매 프레임 비싼 Hinting VM을 다시 돌리지 않으려 하죠. 개발자는 이 캐싱을 깨기 위해 매 프레임마다 미세한 Jitter(떨림) 값을 추가해 강제로 브라우저가 렌더링을 다시 하도록 만들었습니다. 

### 커뮤니티의 반응과 한계

Hacker News의 반응은 뜨거웠지만, 엔지니어다운 날카로운 지적들도 많았습니다.

- **Doom vs Wolfenstein:** 한 유저가 지적했듯, 엄밀히 말해 Doom은 BSP(Binary Space Partitioning) 트리를 사용한 폴리곤 렌더링 방식이고, 이 프로젝트처럼 수직선 기반의 Raycasting을 사용하는 것은 Wolfenstein 3D 방식입니다. 기술적으로는 TTF-Wolfenstein이 맞습니다.
- **플랫폼 의존성과 버그:** macOS의 Chrome이나 Linux의 Brave 브라우저에서는 3D 벽이 렌더링되지 않고 그냥 '초록색 수직 막대'들만 보인다는 리포트가 많습니다. 브라우저마다 폰트 Hinting 엔진(FreeType, CoreText 등)의 구현이 다르고, 최적화 방식이 다르기 때문에 발생하는 문제입니다. 표준 스펙의 엣지 케이스를 극한으로 악용한 프로젝트가 가지는 숙명적인 한계입니다.

### 결론: 우리는 왜 이런 짓을 하는가?

솔직히 말해서 이 기술을 프로덕션에 쓸 일은 영원히 없을 겁니다. 퍼포먼스는 브라우저의 변덕에 기대어 겨우 30-60fps를 유지하고, 플랫폼 호환성은 엉망이며, 유지보수는 불가능에 가깝습니다.

하지만 저는 이런 프로젝트를 볼 때마다 코딩의 본질적인 즐거움을 느낍니다. 우리는 매일 React, Kubernetes, AWS 같은 거대한 추상화 레이어 위에서 안전하게 일합니다. 하지만 가끔은 그 밑바닥으로 내려가, 1991년에 만들어진 레거시 바이트코드의 제약 조건과 싸우며 1바이트를 쥐어짜는 경험이 우리를 더 나은 엔지니어로 만듭니다.

이 프로젝트는 시스템의 한계를 이해하고, 문서를 뒤지며, 우회로를 찾아내는 엔지니어링의 정수를 보여줍니다. 주말에 시간이 나신다면 저장소를 클론해서 브라우저의 폰트 렌더러가 비명을 지르는 소리를 직접 들어보시길 권합니다.
