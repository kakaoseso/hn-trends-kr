클라우드 환경에서 GPU 인스턴스를 대여해 쓰는 것은 이제 일상입니다. 우리는 하이퍼바이저나 컨테이너가 완벽한 격리를 제공할 것이라고 믿죠. 하지만 하드웨어 레벨의 취약점은 이런 얄팍한 믿음을 산산조각 냅니다. 최근 Nvidia GPU를 타겟으로 한 새로운 Rowhammer 공격이 공개되며, 공유 GPU 환경의 보안에 큰 경고등이 켜졌습니다.

## Rowhammer의 진화: CPU에서 GPU로

Rowhammer는 새로운 개념이 아닙니다. 2014년에 처음 CPU DRAM(DDR3)에서 비트 플립(Bit-flip)을 일으킬 수 있다는 게 증명되었죠. 메모리 셀에 빠르고 반복적으로 접근하여 전기적 간섭을 일으키고, 인접한 셀의 데이터를 물리적으로 변조하는 기법입니다.

지난 10년간 이 공격은 DDR4, ECC 우회, 심지어 네트워크를 통한 공격으로까지 진화했습니다. 하지만 이번 타겟은 CPU가 아니라 GPU의 GDDR 메모리입니다. 작년에 GDDR이 취약하다는 첫 연구가 있었지만, 당시엔 신경망 출력을 약간 망가뜨리는 수준에 그쳤습니다. 그러나 이번 공격은 차원이 다릅니다. 공격자는 GPU 메모리 쓰기 작업을 교묘하게 조작하여 호스트 CPU 메모리에 대한 임의 접근 권한을 얻어냈고, 결과적으로 호스트 머신의 **루트(root)** 권한까지 탈취했습니다.

## 엔지니어의 시선: 왜 이것이 충격적인가?

솔직히 이 소식을 처음 접했을 때 제 반응은 "결국 올 것이 왔군" 이었습니다. GPU는 본질적으로 거대한 병렬 메모리 접근 머신입니다. Rowhammer 공격에 필요한 '빠르고 반복적인 메모리 접근'을 하기에 GPU만큼 완벽한 하드웨어가 또 있을까요?

Hacker News에서도 이 주제로 열띤 토론이 벌어졌습니다. 흥미로운 점은 커뮤니티의 반응입니다.

- **Target:** 현재 취약점이 확인된 것은 Ampere 아키텍처 기반의 RTX 3060과 RTX 6000입니다.
- **Impact:** 일반 컨슈머보다는 Multi-tenant 클라우드 환경이나 데이터센터에 치명적입니다.

가장 충격적인 부분은 제한된 GPU 메모리 조작이 어떻게 호스트 CPU 메모리 접근으로 이어졌는가 하는 점입니다. 이는 GPU와 CPU 간의 메모리 매핑과 DMA(Direct Memory Access) 구조의 맹점을 정확히 찌른 것입니다. 샌드박스나 가상화 환경이 소프트웨어적으로 아무리 견고하더라도, 하드웨어 레벨에서 비트가 뒤집히면 모든 방어막이 무용지물이 됩니다.

## 방어책: IOMMU와 레거시의 딜레마

다행히 완화책(Mitigation)은 존재합니다.

- **ECC:** GPU의 ECC 기능을 켭니다.
- **IOMMU:** BIOS에서 IOMMU를 켭니다.

IOMMU(Input-Output Memory Management Unit)는 디바이스가 호스트 메모리에 DMA로 접근할 때 이를 격리하고 통제하는 핵심 보안 기능입니다. 문제는 여전히 수많은 메인보드 벤더들이 레거시 OS나 오래된 애플리케이션과의 호환성을 핑계로 IOMMU를 기본으로 꺼두고 있다는 점입니다.

최신 컴퓨팅 환경에서 이런 식의 보수적인 디폴트 설정은 엔지니어로서 정말 답답한 노릇입니다. 보안과 하위 호환성 사이의 트레이드오프에서, 이제는 보안을 디폴트로 가져가야 할 때가 한참 지났습니다. IOMMU를 켜면 약간의 Latency나 Throughput 저하가 있을 수 있지만, 루트 권한을 통째로 넘겨주는 것과는 비교할 수 없는 비용입니다.

## 결론 (Verdict)

당장 집에서 게임을 돌리는 개인 PC의 전원 플러그를 뽑을 필요는 없습니다. 하지만 AWS, GCP 같은 퍼블릭 클라우드 인프라를 설계하거나 사내에서 AI 학습용 GPU 클러스터를 운영하는 인프라 팀이라면 이야기가 다릅니다.

하드웨어 취약점은 소프트웨어 패치처럼 코드 몇 줄로 간단히 해결되지 않습니다. 지금 당장 운영 중인 시스템의 BIOS 설정과 IOMMU 활성화 여부를 감사(Audit)하시기 바랍니다. 클라우드 벤더가 알아서 해줄 것이라고 맹신하지 마세요. 인프라의 밑바닥을 이해하고 통제하는 것, 그것이 시니어 엔지니어의 진짜 역할입니다.
