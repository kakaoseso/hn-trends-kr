과거 C 언어 시절부터 우리는 32비트(float)에서 64비트(double)로 넘어가는 것을 축복으로 여겼습니다. 메모리가 허락하는 한 더 높은 정밀도를 가지는 것은 수치 해석에서 발생하는 수많은 부동소수점 오차 문제를 마법처럼 해결해 주었으니까요. 저 역시 과거 금융 데이터 파이프라인이나 물리 엔진을 설계할 때, 고민 없이 `double`을 기본 타입으로 박아 넣고 시작하곤 했습니다.

하지만 딥러닝, 특히 거대 언어 모델(LLM)의 시대가 도래하면서 하드웨어 아키텍처의 패러다임이 완전히 뒤집혔습니다. 이제 병목은 연산 장치(ALU)가 아니라 **메모리 대역폭** 과 **VRAM 용량** 입니다. 수천억 개의 파라미터를 메모리에 올리기 위해 우리는 FP16(16비트)에 만족하지 못하고 FP8(8비트)을 넘어, 이제는 고작 4비트 공간에 부동소수점을 욱여넣는 **FP4** 를 진지하게 프로덕션 레벨에서 다루고 있습니다.

오늘은 John D. Cook의 블로그 글과 해커뉴스(Hacker News)의 열띤 토론을 바탕으로, 이 4비트 부동소수점이 도대체 어떻게 동작하는지, 그리고 왜 이것이 단순한 장난감이 아닌 엔지니어링의 정수인지 깊게 파헤쳐 보겠습니다.

## 고작 4비트로 부동소수점을 표현한다고?

4비트는 단 16개의 상태값만 가질 수 있습니다. 이 좁은 공간에 부호(Sign), 지수(Exponent), 가수(Mantissa)를 모두 구겨 넣어야 합니다.

일반적으로 부호에 1비트를 할당하고 나면 우리에겐 3비트가 남습니다. 이를 지수(x)와 가수(y)에 어떻게 분배하느냐에 따라 E3M0, E2M1, E1M2, E0M3 네 가지 조합이 나옵니다. 현재 Nvidia를 비롯한 업계 표준으로 굳어지고 있는 것은 **E2M1** (지수 2비트, 가수 1비트) 포맷입니다.

E2M1 포맷에서 실제 값을 구하는 공식은 다음과 같습니다.

`(-1)^s * 2^(e-b) * (1 + m/2)`

여기서 `b`는 편향(bias) 값입니다. 단, 지수 `e`가 0일 때는 예외 처리가 들어갑니다. `m=0`이면 0을, `m=1`이면 0.5를 나타냅니다. 

이 16개의 값을 Python의 `Pychop` 라이브러리를 통해 나열해 보면 다음과 같은 흥미로운 테이블이 나옵니다.

```python
import pychop

# Pull the format metadata from Pychop.
spec = pychop.MX_FORMATS["mxfp4_e2m1"]
assert (spec.exp_bits, spec.sig_bits) == (2, 1)

def e2m1_value(s: int, e: int, m: int) -> float:
    sign = -1.0 if s else 1.0
    # Subnormal / zero
    if e == 0:
        return sign * (m / 2.0)
    # Normal
    return sign * (2.0 ** (e - 1)) * (1.0 + m / 2.0)

def display_value(bits: int, x: float) -> str:
    if bits == 0b0000:
        return "+0"
    if bits == 0b1000:
        return "-0"
    return f"{x:+g}"

rows = []
for bits in range(16):
    s = (bits >> 3) & 0b1
    e = (bits >> 1) & 0b11
    m = bits & 0b1
    x = e2m1_value(s, e, m)
    rows.append({
        "Bits": f"{bits:04b}",
        "s": s,
        "exp_bits": f"{e:02b}",
        "m": m,
        "Value": display_value(bits, x),
    })

# Pretty-print the table.
header = f"{'Bits':<4} {'s':>1} {'exp':>3} {'m':>1} {'Value':>6}"
print(header)
print("-" * len(header))
for row in rows:
    print(
        f"{row['Bits']:<4} "
        f"{row['s']:>1} "
        f"{row['exp_bits']:>3} "
        f"{row['m']:>1} "
        f"{row['Value']:>6}"
    )
```

출력된 값들을 보면: `+0, +0.5, +1, +1.5, +2, +3, +4, +6` (그리고 음수 대칭) 입니다.

솔직히 처음 이 테이블을 봤을 때 제 반응은 "이게 진짜 모델 학습이나 추론에 쓸모가 있다고?" 였습니다. 심지어 16개밖에 안 되는 슬롯 중에 2개를 `+0`과 `-0`에 할당해 버렸습니다. IEEE 754의 유산을 고집스럽게 따르는 이 모습이 비효율적으로 보일 수도 있습니다.

## 엔지니어의 시선: 왜 정수(INT4)가 아니라 부동소수점(FP4)인가?

여기서 주니어 엔지니어들이 흔히 하는 질문이 있습니다. "4비트면 그냥 0부터 15까지 정수(Integer)로 표현하고, 적당히 bias를 빼서 -7 ~ 8로 쓰는 게 낫지 않나요?"

하지만 실제 신경망 모델의 가중치(Weight) 분포를 분석해 보면 이야기가 다릅니다. 가중치들은 0 근처에 정규분포 형태로 빽빽하게 몰려 있고, 꼬리(tail) 쪽으로 갈수록 듬성듬성해집니다. 

정수형(INT4)은 선형적인 간격을 가집니다. 반면, 부동소수점 방식은 지수부 덕분에 **Dynamic range** 를 가집니다. 즉, 0 근처에서는 매우 세밀한 간격(0, 0.5, 1, 1.5)을 표현하고, 값이 커질수록 간격이 넓어집니다(4, 6). 이는 LLM 가중치의 실제 분포와 훨씬 잘 맞아떨어집니다. 엔비디아가 굳이 하드웨어 레벨에서 E2M1을 지원하는 데에는 다 철저한 데이터 기반의 이유가 있는 것입니다.

## 해커뉴스 반응과 진짜 '비밀 무기': 룩업 테이블(LUT)

이번 해커뉴스 스레드에서 가장 제 무릎을 탁 치게 만든 것은 연산 최적화에 대한 토론이었습니다.

- **값의 범위:** 4비트는 단 16개의 상태를 가집니다.
- **연산 복잡도:** 두 FP4 숫자를 곱하는 연산을 생각해 봅시다.

기존 FP32 환경에서는 복잡한 부동소수점 곱셈기(Multiplier) 회로를 거쳐야 합니다. 하지만 FP4에서는 그럴 필요가 전혀 없습니다. `16 x 16 = 256`개의 결과값만 존재하기 때문입니다. 

해커뉴스의 한 유저가 지적했듯, `a * b` 연산은 그저 256바이트짜리 1차원 배열(LUT)을 만들어두고 `lut[a << 4 + b]`로 읽어오면 끝납니다. 덧셈도 마찬가지입니다. 이는 압축을 통한 메모리 대역폭 절약뿐만 아니라, **연산 Throughput 측면에서 압도적인 이점** 을 가져다줍니다. 모델의 추론 속도가 메모리 바운드에서 벗어나 날개를 다는 순간입니다.

또한, 스레드에서는 CPU의 FP64 vs FP32 성능에 대한 격렬한 논쟁도 있었습니다. 최신 아키텍처(Zen4 등)에서 스칼라 연산 자체의 Latency는 비슷할지 몰라도, SIMD(AVX) 레벨로 가면 레지스터에 태울 수 있는 데이터 양이 두 배 차이 나기 때문에 Throughput은 명확히 2배 이상 벌어집니다. 하물며 FP4라면 어떨까요? 대역폭 효율은 상상을 초월하게 됩니다.

## 한계점과 OCP 표준 (MXFP4)

물론 순수 FP4 자체로는 한계가 명확합니다. 해커뉴스 유저들이 지적했듯 이 포맷에는 `NaN`이나 `Infinity`를 표현할 슬롯조차 없습니다.

그래서 현업에서는 개별 가중치를 순수 FP4로만 두지 않습니다. 대신 **MXFP4** 나 **NVFP4** 같은 블록 스케일링(Block Scaling) 기법을 사용합니다. 예를 들어 32개의 FP4 값을 하나의 블록으로 묶고, 이 블록 전체에 적용되는 고정밀도(FP16 등)의 공통 Scaling Factor를 하나 두는 방식입니다. 이렇게 하면 메모리 사용량은 극적으로 줄이면서도, 모델의 전체적인 표현력과 안정성은 유지할 수 있습니다.

## 결론: FP4는 프로덕션 레벨인가?

제 결론부터 말씀드리면, **FP4는 절대 장난감이 아닙니다. 다가오는 엣지 AI와 대규모 LLM 추론의 핵심 인프라입니다.**

과거 80비트 x87 코프로세서 시절의 엔지니어들이 지금의 4비트 모델을 본다면 기겁을 하겠지만, 시대가 요구하는 최적화의 방향이 달라졌습니다. 우리는 이제 '개별 파라미터의 절대적인 정밀도'보다 '거대한 통계적 모델의 거시적인 표현력과 메모리 대역폭의 극한 활용'이 더 중요하다는 것을 깨달았습니다.

앞으로 양자화(Quantization) 기술은 단순히 모델 사이즈를 줄이는 후처리 작업을 넘어, 하드웨어 아키텍처와 모델 학습 단계(QAT)부터 깊게 관여하는 형태로 발전할 것입니다. 시니어 엔지니어라면 지금 당장 Pychop 같은 라이브러리를 켜고, 저비트 부동소수점의 세계가 어떻게 동작하는지 직접 감을 잡아보시기를 강력히 권합니다.
