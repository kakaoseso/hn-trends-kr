최근 LLM 씬을 보면 온통 수백억, 수천억 파라미터 이야기로 도배되어 있다. 하지만 현업에서 엔지니어링을 하다 보면 가끔 현타가 온다. "거실 전등 꺼줘" 같은 단순한 명령 하나 처리하자고 매번 클라우드 API를 태우거나 수 GB에 달하는 로컬 모델을 메모리에 올려둬야 할까? 

이런 고민을 하던 차에 Hacker News에서 매우 흥미로운 프로젝트를 발견했다. Gemini 3.1의 Tool Calling 능력을 단 26M(2600만) 파라미터로 증류(Distillation)한 **Needle** 이라는 모델이다.

## Needle: 14MB짜리 인공지능 라우터

Needle은 범용 대화형 AI가 아니다. 철저하게 'Single-shot Tool Calling'에만 타겟팅된 특수 목적 모델이다. 제작팀은 Gemini 3.1의 능력을 추출해 자체 설계한 Simple Attention Network에 우겨넣었다. 

가장 충격적인 것은 사이즈다. INT4 양자화를 적용하면 모델 크기가 고작 14MB에 불과하다. 스마트폰은 물론이고 스마트워치, 안경, 심지어 라즈베리파이 같은 극단적인 Edge 환경의 메모리에도 상시 상주시킬 수 있는 크기다.

### 아키텍처 톺아보기

단순히 파라미터만 줄인 것이 아니라, 구조적으로 꽤 흥미로운 선택들을 했다.

- **Dimension & Heads:** d=512, 8H/4KV 구조를 채택했다. GQA(Grouped Query Attention)를 통해 KV 캐시 메모리 사용량을 극단적으로 줄였다.
- **ZCRMSNorm:** 일반적인 LayerNorm이나 RMSNorm 대신 Zero-Centered RMSNorm을 채택하여 정규화의 안정성을 높였다.
- **Asymmetric Blocks:** FFN(Feed-Forward Network)을 과감히 제거한 Encoder 블록 12개와, Cross Attention을 수행하는 Decoder 블록 8개로 구성되어 있다.

이 모델은 16개의 TPU v6e에서 200B 토큰으로 27시간 동안 Pretrain을 거친 후, 2B 토큰의 단일 샷 함수 호출 데이터셋으로 45분간 Post-train 되었다. 그 결과 자체 런타임인 Cactus 위에서 **Prefill 6000 toks/sec, Decode 1200 toks/sec** 라는 경이로운 Throughput을 뽑아낸다. 이 정도면 Latency가 사실상 '0'에 가깝게 느껴지는 수준이다.

### 코드로 보는 직관성

제공되는 API도 매우 직관적이다. Python 환경에서 로컬 모델을 불러와 바로 테스트해 볼 수 있다.

```python
from needle import SimpleAttentionNetwork, load_checkpoint, generate, get_tokenizer

params, config = load_checkpoint("checkpoints/needle.pkl")
model = SimpleAttentionNetwork(config)
tokenizer = get_tokenizer()

result = generate(
    model, params, tokenizer,
    query="What's the weather in San Francisco?",
    tools='[{"name":"get_weather","parameters":{"location":"string"}}]',
    stream=False,
)
print(result)
# [{"name":"get_weather","arguments":{"location":"San Francisco"}}]
```

단순한 자연어 쿼리와 도구(Tools)의 JSON 스키마를 던져주면, 정확하게 파싱된 함수 호출 인자를 반환한다. 

## Principal Engineer의 시선: 이거 진짜 쓸만한가?

솔직히 처음엔 의심했다. "26M짜리 모델이 파싱을 해봤자 얼마나 정확하겠어? 환각(Hallucination) 파티가 열리거나 엉뚱한 파라미터를 뱉어내겠지." 

하지만 아키텍처와 목적성을 뜯어보니 제작팀의 영리한 선택에 박수를 보내게 된다. 대화형(Conversational) 능력을 완전히 버렸기 때문에 가능한 성능이다. 모델의 역할을 "자연어 입력 -> JSON 형태의 Tool Call 출력"이라는 단일 파이프라인으로 제한함으로써, 제한된 Capacity(26M)를 파싱과 라우팅에만 100% 쏟아부은 것이다.

과거 마이크로서비스 아키텍처(MSA)를 설계할 때, API Gateway 단에 복잡한 정규식 기반의 룰 엔진을 짜넣던 시절이 있었다. 유지보수는 지옥 같았고 예외 케이스는 끝이 없었다. 이제는 그 룰 엔진 자리에 14MB짜리 Needle 모델을 박아넣고 자연어 기반의 라우팅을 시키는 패턴이 실무에서 꽤 유효해질 것이라 확신한다.

## Hacker News의 반응과 커뮤니티의 통찰

HN 스레드에서도 나와 비슷한 생각을 하는 엔지니어들의 열띤 토론이 이어졌다.

- **Home Assistant의 구원자:** 가장 많은 공감을 얻은 유스케이스다. 스마트홈 구축 시 "컴퓨터! 불 꺼!" 같은 명령을 처리하기 위해 굳이 무거운 LLM을 돌릴 필요가 없다. Needle은 로컬에서 프라이버시 침해 없이 즉각적으로 반응하는 훌륭한 컨트롤러가 될 수 있다.
- **자연어 CLI 툴:** 한 유저는 CLI 프로그램에 Needle을 번들링하여 `toolcli what can you do`라고 치면 내부적으로 `toolcli --help summary`로 매핑해주는 아이디어를 제안했다. 14MB의 용량 추가가 부담스럽다는 반론도 있었지만, 사용자 경험(UX) 측면에서는 대단히 매력적인 접근이다.
- **WebGPU와 Edge AI:** 모델이 워낙 작다 보니 Transformers.js나 WebGPU를 이용해 브라우저 단에서 돌리자는 의견도 즉각적으로 나왔다. 실제로 스레드가 진행되는 동안 한 유저가 Hugging Face Space에 Docker 기반의 Playground를 띄워버리는 오픈소스 생태계 특유의 추진력도 볼 수 있었다.

재미있게도 26M(Million)을 26B(Billion)로 잘못 읽고 "이게 왜 작은 거냐"고 따지는 사람들도 있었다. 요즘 모델 사이즈 인플레이션이 얼마나 심한지 보여주는 씁쓸한 해프닝이다.

## 결론: LLM 만능주의에 던지는 돌직구

Needle은 모든 문제에 GPT-4o나 Claude 3.5 Sonnet을 들이밀 필요가 없다는 것을 증명하는 훌륭한 사례다. 

물론 프로덕션 레벨에서 복잡한 도구 체이닝(Chaining)을 요구하거나, 컨텍스트가 긴 대화 속에서 함수를 호출해야 하는 상황이라면 Needle은 적합하지 않다. 그럴 때는 더 큰 용량의 모델이 필요하다. 

하지만 IoT 디바이스, 모바일 앱의 온디바이스 AI, 혹은 백엔드의 가벼운 자연어 라우터 역할로는 이보다 더 날카로운 도구를 찾기 힘들 것이다. 15년 넘게 엔지니어링을 해오며 배운 진리가 하나 있다면, **"가장 좋은 도구는 가장 강력한 도구가 아니라, 목적에 가장 알맞은 도구"** 라는 것이다. Needle은 그 철학에 완벽하게 부합한다. 당장 이번 주말에 사이드 프로젝트로 Home Assistant에 연동해 볼 생각이다.
