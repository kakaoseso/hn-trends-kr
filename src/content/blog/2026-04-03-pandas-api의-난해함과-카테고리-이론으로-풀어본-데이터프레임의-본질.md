솔직히 말해봅시다. Pandas API는 정말 끔찍합니다.

데이터 엔지니어링이나 분석을 조금이라도 해본 사람이라면 공감할 겁니다. `pivot`과 `melt`는 어떻게 다른가요? `apply`, `map`, `transform`, `agg`, `applymap`의 정확한 차이를 안 보고 설명할 수 있나요? Pandas에는 DataFrame 객체에만 200개가 넘는 메서드가 존재합니다. 일관된 구조나 철학이 없다 보니, 우리는 본질적인 데이터의 형태를 이해하기보다는 API 문서를 외우고 StackOverflow를 복붙하는 데 시간을 쏟게 됩니다.

최근 Hacker News에서 흥미로운 글을 하나 읽었습니다. 한 개발자가 자신만의 데이터프레임 라이브러리를 만들면서, 이 복잡한 200여 개의 연산들을 '카테고리 이론(Category Theory)'을 통해 단 몇 개의 수학적 원시 연산으로 압축해낸 과정을 담은 글입니다. 

단순히 수학적 유희로 끝나는 글이 아닙니다. 우리가 매일 사용하는 데이터 도구들이 왜 이렇게 설계되었는지, 그리고 앞으로 어떤 방향으로 나아가야 하는지에 대한 깊은 통찰을 줍니다. 15년 넘게 시스템을 설계해 온 엔지니어의 관점에서, 이 글이 시사하는 바와 커뮤니티의 반응을 깊이 파헤쳐 보겠습니다.

## 200개의 API를 15개로 압축하기

글의 저자는 먼저 UC Berkeley 연구진이 만든 Modin 프로젝트의 논문(Towards Scalable Dataframe Systems)을 인용합니다. Modin 팀은 100만 개의 Jupyter 노트북을 분석하여 사람들이 Pandas를 실제로 어떻게 사용하는지 조사했고, 200개가 넘는 API의 85% 이상이 단 15개의 '데이터프레임 대수(Dataframe Algebra)'로 표현된다는 것을 증명했습니다.

- **Relational:** SELECTION, PROJECTION, UNION, DIFFERENCE, JOIN, GROUPBY 등 (SQL과 동일)
- **Dataframe Specific:** TRANSPOSE, MAP, TOLABELS, FROMLABELS 등

이것만으로도 훌륭한 압축이지만, 저자는 여기서 한 걸음 더 나아갑니다. "이 15개의 연산 밑바탕에 깔린 더 근본적인 원시 연산(Primitive)이 있지 않을까?"

## 카테고리 이론이 설명하는 3가지 스키마 마이그레이션 (Σ ⊣ Δ ⊣ Π)

저자는 Fong과 Spivak의 저서 *Seven Sketches in Compositionality*를 인용하여, 스키마를 변경하는 모든 관계형 연산이 카테고리 이론의 세 가지 '마이그레이션 함자(Migration Functors)'로 완벽하게 묶인다는 것을 보여줍니다.

이 세 가지는 수학적으로 수반(Adjoint) 관계를 가집니다 (Σ ⊣ Δ ⊣ Π).

- **Delta (Δ) - Restructuring:** 데이터를 건드리지 않고 스키마의 형태만 바꿉니다. SQL의 `SELECT name, salary`나 Pandas의 열 이름 변경이 여기에 속합니다. 입력 스키마만 보면 출력 스키마를 100% 예측할 수 있습니다.
- **Sigma (Σ) - Merging:** 여러 행을 하나의 키로 병합합니다. 다수의 소스 행이 하나의 타겟을 가리킬 때 데이터를 모으는(Collect) 과정입니다. `GROUPBY` 후 `SUM`이나 `COUNT`를 하는 집계 연산이 여기에 해당합니다.
- **Pi (Π) - Pairing:** 두 스키마의 공유 키를 바탕으로 데이터를 꿰매어 더 넓은 행을 만듭니다. 데이터베이스 프로그래머들이 흔히 말하는 `JOIN`입니다.

그렇다면 `DIFFERENCE`(차집합)나 `DROP DUPLICATES`(중복 제거)는 어디에 속할까요? 이들은 스키마를 변경하지 않습니다. 카테고리 이론에서는 이를 스키마 간의 이동이 아니라, 단일 스키마 내에서의 부분집합(Subset)을 다루는 **Topos** 구조로 설명합니다.

결과적으로 200개의 혼란스러운 API는 **3개의 스키마 마이그레이션(Δ, Σ, Π)** 과 **2개의 Topos 연산** 으로 깔끔하게 정리됩니다. 저자는 이 수학적 기반을 바탕으로 Haskell에서 스키마 전환이 컴파일 타임에 100% 검증되는 강력한 타입 세이프 데이터프레임 라이브러리를 구현했습니다. 존재하지 않는 컬럼을 `select` 하거나 타입이 맞지 않는 `join`을 시도하면 아예 컴파일이 되지 않는 것이죠.

## Hacker News의 냉혹한 현실 감각

이론은 아름답습니다. 하지만 현업에 있는 시니어 엔지니어들의 생각은 어떨까요? HN 커뮤니티의 반응은 꽤나 날카롭고 현실적이었습니다.

### 1. Pandas API는 애초에 잘못된 기반이다
가장 많은 공감을 얻은 의견 중 하나는 "Pandas의 사용 패턴을 연구해서 데이터프레임의 우아한 핵심을 찾으려는 시도 자체가 넌센스"라는 지적이었습니다. 

Pandas는 애초에 헤지펀드에서 금융 시계열 데이터(Panel data)를 다루기 위해 만들어졌습니다. 그놈의 지긋지긋한 `Index` 집착이나, `index=False`를 매번 써줘야 하는 기괴한 기본값들은 모두 '모든 데이터에는 의미 있는 타임스탬프 인덱스가 있다'는 초창기 가정에서 비롯된 기술 부채입니다. 

누군가 바퀴라는 걸 한 번도 본 적이 없어서 '7각형 바퀴'를 발명했고, 이제 수백만 명이 그 7각형 바퀴에 맞춰 살아가고 있는 셈입니다. 카테고리 이론가가 와서 "이걸 육각형이나 사각형으로 만들면 더 낫다"고 증명하는 게 무슨 소용일까요? 그냥 원형 바퀴(SQL이나 Relational Algebra)를 쓰면 되는데 말이죠.

### 2. R의 dplyr과 Polars가 이미 해결한 문제
데이터프레임은 본질적으로 데이터베이스 테이블과 크게 다르지 않습니다. R 생태계의 `dplyr`이나 `data.table`은 이미 10년 전에 SQL의 철학을 존중하면서 이 문제를 우아하게 해결했습니다. 최근 Python 생태계에서 Pandas를 대체하며 급부상 중인 **Polars** 역시 마찬가지입니다. Polars가 직관적이고 빠른 이유는 단순히 Rust로 작성되어서가 아니라, 관계형 대수(Relational Algebra)를 기반으로 API를 설계하고 쿼리 최적화(Pushdown 등)를 수행하기 때문입니다.

### 3. 'Drop Duplicates' 논쟁: 상아탑 vs 진흙탕
흥미로웠던 또 다른 토론은 `DROP DUPLICATES` 연산에 대한 시각차였습니다. 

순수주의자들은 "중복 데이터는 업스트림의 데이터 모델링이 잘못되었다는 악취(Smell)일 뿐이며, 이를 무지성으로 drop 하는 것은 안티패턴"이라고 비판합니다. SQL의 `SELECT DISTINCT`처럼 처리하는 것이 맞다는 것이죠.

하지만 현업 데이터 엔지니어들의 항변도 만만치 않습니다. "우리가 항상 완벽한 업스트림 데이터를 받는 건 아닙니다. 영업팀 공유 드라이브에 굴러다니는 엑셀 파일을 긁어온 CSV를 처리해야 할 때, 직관적인 `df.drop_duplicates()`는 현장에서 피 흘리는 엔지니어들에게 꼭 필요한 구명줄입니다." 

저 역시 후자에 깊이 공감합니다. 이론적으로 완벽한 파이프라인을 구축하는 것도 중요하지만, 현실의 데이터는 언제나 지저분하고 우리는 당장 내일 아침까지 리포트를 뽑아내야 하니까요.

## 나의 결론: 이론은 컴파일러에게, 엔지니어에겐 SQL을

저자가 카테고리 이론을 통해 도출해낸 Σ ⊣ Δ ⊣ Π 구조는 훌륭합니다. 특히 쿼리 옵티마이저나 분산 처리 엔진, 타입 세이프한 라이브러리를 직접 밑바닥부터 설계해야 하는 시스템 엔지니어라면 반드시 숙지해야 할 깊은 통찰입니다. 연산의 순서를 안전하게 바꾸고(Commutativity), 불필요한 연산을 제거하는 최적화의 수학적 근거가 되기 때문입니다.

하지만 일반적인 데이터 엔지니어나 백엔드 개발자라면 어떨까요? 굳이 카테고리 이론까지 갈 필요는 없습니다. 우리가 얻어야 할 교훈은 훨씬 단순합니다.

**"수학적으로 검증되지 않은, 편의성 위주로 덧붙여진 Ad-hoc API(Pandas)를 멀리하고, 오랜 시간 검증된 관계형 대수 기반의 도구(SQL, Polars, dplyr)를 가까이 하라."**

Pandas의 시대는 저물고 있습니다. 데이터가 메모리에 다 올라가지 않을 만큼 커지고, 파이프라인의 복잡도가 증가할수록, 우리는 MapReduce 식의 절차적 사고에서 벗어나 관계형 대수가 제공하는 선언적(Declarative) 사고로 넘어가야 합니다. 

다음 프로젝트를 시작한다면, 습관적으로 `import pandas as pd`를 타이핑하기 전에 Polars를 한번 검토해 보시길 권합니다. 카테고리 이론을 몰라도, 잘 설계된 API가 주는 평화를 누릴 수 있을 겁니다.
