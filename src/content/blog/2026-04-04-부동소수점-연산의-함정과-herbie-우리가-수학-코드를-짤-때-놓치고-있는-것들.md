15년 넘게 소프트웨어 엔지니어링을 하면서 가장 디버깅하기 끔찍했던 버그들을 꼽으라면, 십중팔구는 멀티스레딩 이슈 아니면 **부동소수점(Floating-point) 연산 오차** 였습니다.

`0.1 + 0.2 != 0.3`이라는 사실은 주니어 시절에 누구나 배우지만, 막상 프로덕션 환경에서 복잡한 수식 모델이나 물리 엔진, 금융 계산 모듈을 짤 때 이 IEEE 754 표준의 한계를 엄밀하게 고려하는 엔지니어는 드뭅니다. 대부분은 'double을 쓰면 대충 맞겠지' 하고 넘어가죠. 그러다 특정 엣지 케이스에서 값이 폭주(Overflow)하거나 완전히 소실(Catastrophic Cancellation)되면서 장애가 터집니다.

최근 Hacker News에서 다시금 주목받고 있는 [Herbie](https://herbie.uwplse.org/)라는 도구를 보며, 과거에 이 문제로 밤을 새웠던 기억이 떠올랐습니다. 워싱턴 대학교(UW) PLSE 연구실에서 만든 이 툴은, 우리가 작성한 불안정한 부동소수점 수식을 분석하고 수학적으로 동치이면서도 **오차가 훨씬 적은 형태** 로 코드를 자동 재작성해 줍니다.

단순한 장난감이 아닙니다. 이번 포스팅에서는 Herbie가 정확히 어떻게 동작하는지, 그리고 실제 오픈소스 라이브러리의 버그를 어떻게 해결했는지 Deep-dive 해보겠습니다.

## 부동소수점 버그의 실체: math.js의 사례

Herbie 튜토리얼에서 다루는 JavaScript 수학 라이브러리인 `math.js`의 [Bug 208](https://github.com/josdejong/mathjs/pull/208) 사례를 살펴봅시다. 복소수의 제곱근을 구하는 로직에서 심각한 오차가 발생하는 버그였습니다.

문제가 된 수학적 코어(Kernel)는 다음과 같습니다.

```javascript
// x.im >= 0 일 때의 복소수 제곱근 실수부 계산
0.5 * sqrt(2.0 * (sqrt(xre * xre + xim * xim) + xre))
```

수학적으로는 전혀 문제가 없어 보입니다. 피타고라스 정리를 이용해 크기를 구하고, 반각 공식을 적용한 전형적인 구현체죠. 하지만 이를 부동소수점으로 실행하면 두 가지 치명적인 문제가 발생합니다.

1. **Overflow / Underflow:** `xre * xre`를 계산할 때, `xre`가 적당히 큰 값(예: 1e160)이면 제곱하는 순간 double의 최대 범위(약 1.79e308)를 넘어 Infinity가 되어버립니다.
2. **Catastrophic Cancellation (대규모 상쇄):** `xre`가 매우 큰 음수일 경우, `sqrt(...)`의 결과는 양수이고 여기에 `xre`(음수)를 더하게 됩니다. 크기가 비슷한 두 수를 빼는 연산이 되어버리면서 유효 숫자가 모조리 날아가 버립니다.

## Herbie의 마법: 수식을 어떻게 뜯어고치는가?

이 수식을 Herbie에 넣고 돌리면, 내부적으로 수만 번의 랜덤 샘플링과 수학적 변환(테일러 급수 전개, 항 재배치 등)을 거쳐 대안을 제시합니다. 

Herbie가 제안한 가장 정확도 높은(Accuracy 99.7%) 코드는 다음과 같습니다.

```javascript
// Herbie가 최적화한 결과 (JavaScript로 번역 및 리팩토링 후)
var r = Math.hypot(x.re, x.im);
var re;
if (xre + r <= 0) {
  re = 0.5 * Math.sqrt(2 * (x.im / (x.re / x.im) * -0.5));
} else {
  re = 0.5 * Math.sqrt(2 * (x.re + r));
}
```

이 코드를 처음 봤을 때, 저는 두 가지 포인트에서 감탄했습니다.

### 1. Math.hypot의 활용
Herbie는 `sqrt(xre * xre + xim * xim)`를 `Math.hypot(xre, xim)`으로 교체했습니다. `hypot` 함수는 내부적으로 중간 계산 과정에서의 Overflow나 Underflow를 방지하도록 구현되어 있습니다. 시니어 엔지니어라면 당연히 알아야 할 기본기지만, 수식을 그대로 코드로 옮기는 데 급급하다 보면 놓치기 쉬운 부분입니다.

### 2. 조건부 분기(Branch)의 삽입
제가 가장 놀란 부분입니다. Herbie는 원본에 없던 `if (xre + r <= 0)` 분기를 만들어냈습니다. 

솔직히 처음엔 '수학 연산 코어에 Branch를 넣으면 Branch Prediction 실패 시 파이프라인 스톨(Stall)이 발생해서 Throughput이 깎이는 거 아닌가?'라는 의심이 들었습니다. 하지만 성능보다 **정확도** 가 중요한 상황에서는 이 분기가 핵심입니다. 앞서 언급한 **Catastrophic Cancellation** 을 회피하기 위해, `xre`가 음수여서 상쇄가 일어날 위험이 있는 구간과 그렇지 않은 구간을 나누어 완전히 다른 대수적 동치 수식을 적용한 것입니다.

## 엔지니어링 관점에서의 Trade-off

Herbie가 훌륭한 이유는 단순히 '가장 정확한 식' 하나만 던져주고 끝내지 않는다는 점입니다. 결과 페이지를 보면 다양한 대안(Alternatives)을 보여줍니다.

- **Alternative 1:** 정확도 99.7%, 원본 대비 속도 0.8x (가장 정확하지만 분기문 때문에 약간 느림)
- **Alternative 5:** 정확도 84.6%, 원본 대비 속도 1.9x (정확도는 타협했지만 기존보단 낫고, 속도는 두 배 빠름)

이 지점이 바로 우리 같은 엔지니어들이 개입해야 할 곳입니다. 게임 엔진의 파티클 시스템처럼 1프레임 내에 수만 번 실행되어야 하고 약간의 오차가 시각적으로 티 나지 않는다면 Alternative 5를 선택할 것입니다. 반면, 금융권의 이자 계산 모듈이나 정밀한 CAD 소프트웨어라면 주저 없이 Alternative 1을 선택해야 합니다.

## 결론: 그래서 프로덕션에 쓸 만한가?

결론부터 말하자면, **수학적 계산이 코어 비즈니스 로직인 시스템** 에서는 반드시 도입을 검토해 봐야 할 툴입니다.

물론 Herbie가 만능은 아닙니다. 수식이 복잡해질수록 탐색 시간이 기하급수적으로 늘어나며, 데이터 구조체(Struct/Class)나 제어 흐름(Control Flow) 전체를 이해하지는 못합니다. 튜토리얼에서도 나오듯, 개발자가 직접 '수학적 코어'만 발라내어 Herbie에 먹여야 하고, 나온 결과를 다시 기존 코드베이스의 컨텍스트에 맞게 리팩토링하는 과정이 필요합니다.

하지만 우리가 관성적으로 작성하던 `a * a + b * b` 같은 코드 이면에 얼마나 거대한 오차의 싱크홀이 도사리고 있는지 시각화해서 보여준다는 것만으로도 이 툴의 가치는 충분합니다. 

다음번 스프린트에서 복잡한 수식이 들어간 PR을 리뷰하게 된다면, 리뷰 코멘트를 달기 전에 그 수식을 Herbie에 한 번 돌려보시길 권합니다. 아마 당신의 동료는 당신이 엄청난 수학적 통찰력을 가졌다고 착각하게 될지도 모릅니다.
