최근 몇 년간 업계는 마치 'RAG(Retrieval-Augmented Generation) 병'에 걸린 것 같았습니다. 문서 검색이나 질의응답 시스템을 구축한다고 하면, 아무런 의심 없이 문서를 Chunking하고, 임베딩 모델을 돌려 Vector DB에 쑤셔 넣는 아키텍처를 기본값으로 채택하곤 했죠. 

하지만 실무에서 이 RAG 아키텍처를 운영해 본 엔지니어라면 누구나 공감할 만한 치명적인 한계가 있습니다. 바로 **문맥의 파편화** 입니다. 문서를 기계적으로 자르다 보니, 문서가 원래 가지고 있던 계층 구조나 디렉토리 기반의 도메인 컨텍스트가 완전히 날아가 버리는 것이죠.

최근 Hacker News를 뜨겁게 달군 [Mintlify의 블로그 글](https://www.mintlify.com/blog/how-we-built-a-virtual-filesystem-for-our-assistant)은 이 문제를 아주 흥미로운 방식으로 해결했습니다. 그들은 AI 문서 어시스턴트에서 전통적인 RAG를 버리고, 에이전트가 직접 탐색할 수 있는 **가상 파일 시스템(Virtual Filesystem)** 을 도입했습니다. 

솔직히 처음 제목만 봤을 때는 '또 무슨 오버엔지니어링인가' 싶었지만, 아키텍처를 뜯어보고 HN 커뮤니티의 토론을 지켜보면서 이 접근법이 가진 실용적인 우아함에 꽤 깊은 인상을 받았습니다. 오늘은 이 아키텍처가 어떻게 동작하는지, 그리고 왜 이런 선택을 했는지 제 시각에서 파헤쳐보겠습니다.

## 샌드박스의 딜레마: Latency와 비용

최근 AI 에이전트들은 점점 파일 시스템을 기본 인터페이스로 채택하는 추세입니다. `grep`, `cat`, `ls`, `find` 같은 기본적인 UNIX 명령어만 있으면 에이전트가 스스로 코드를 탐색하고 문서를 읽어낼 수 있기 때문입니다. 

가장 직관적인 방법은 에이전트에게 실제 샌드박스 환경을 제공하고 전체 저장소를 Clone하는 것입니다. 하지만 프론트엔드 어시스턴트처럼 사용자가 로딩 스피너를 쳐다보고 있는 환경에서 이 방식은 재앙에 가깝습니다.

- **Latency:** 세션 생성(GitHub Clone 포함)의 p90 시간이 무려 46초에 달했습니다.
- **Cost:** 월 85만 건의 대화를 처리하기 위해 최소한의 마이크로 VM(1 vCPU, 2 GiB RAM)을 띄운다고 가정해도, 인프라 비용만 연간 $70,000 이상이 발생합니다.

백그라운드 비동기 작업이라면 모를까, 실시간 챗봇에서 46초의 부팅 시간은 프로덕트의 죽음을 의미합니다. Mintlify 팀은 이 문제를 해결하기 위해 '진짜' 파일 시스템 대신 '가짜' 쉘을 만들기로 결정합니다.

## ChromaFs: 가짜 쉘과 데이터베이스의 기막힌 동거

핵심 아이디어는 에이전트에게 파일 시스템의 '환상(Illusion)'만 제공하는 것입니다. 

Mintlify는 이미 검색을 위해 문서를 인덱싱하고 청크로 나누어 ChromaDB에 저장하고 있었습니다. 이들은 Vercel Labs에서 만든 TypeScript 기반의 bash 재구현체인 [just-bash](https://github.com/vercel-labs/just-bash)를 활용해 **ChromaFs** 라는 가상 파일 시스템을 구축했습니다.



에이전트가 `ls`나 `cat` 같은 명령어를 입력하면, ChromaFs가 이를 가로채서 백엔드의 ChromaDB 쿼리로 변환합니다. 그 결과는 놀랍습니다.

- **Boot Time:** 46초에서 약 100ms로 감소
- **Marginal Cost:** 기존 DB 인프라를 재사용하므로 추가 컴퓨팅 비용 $0

### 디렉토리 트리 부트스트래핑과 RBAC

에이전트가 명령을 내리기 전에 파일 시스템의 구조를 알아야 합니다. 이를 위해 전체 파일 트리를 gzipped JSON 형태(`__path_tree__`)로 Chroma에 저장해 둡니다.

```json
{
  "auth/oauth": { "isPublic": true, "groups": [] },
  "auth/api-keys": { "isPublic": true, "groups": [] },
  "internal/billing": { "isPublic": false, "groups": ["admin", "billing"] },
  "api-reference/endpoints/users": { "isPublic": true, "groups": [] }
}
```

초기화 시 서버는 이 문서를 메모리에 로드하여 트리 구조를 만듭니다. 여기서 제가 감탄한 부분은 **Access Control(RBAC)** 의 우아한 구현입니다. 

실제 리눅스 샌드박스에서 사용자별 권한 제어를 하려면 `chmod`를 만지거나 별도의 컨테이너 이미지를 관리해야 하는 등 인프라 레벨의 복잡도가 급증합니다. 하지만 ChromaFs에서는 트리를 빌드하기 전에 사용자의 세션 토큰을 기반으로 접근 불가능한 노드를 필터링해버리면 끝납니다. 에이전트 입장에서는 권한이 없는 파일은 아예 존재하지 않는 것처럼 보이게 됩니다.

### 청크 재조립 (Reassembling Pages)

전통적인 RAG의 가장 큰 문제는 문맥 단절입니다. ChromaFs는 에이전트가 `cat /auth/oauth.mdx`를 실행하면, 해당 페이지 슬러그를 가진 모든 청크를 가져와 `chunk_index` 순으로 정렬한 뒤 하나의 완전한 페이지로 결합하여 반환합니다. 

기존 RAG가 '질문과 의미상 비슷한 파편화된 텍스트 쪼가리'를 던져줬다면, 이 방식은 에이전트가 원본 문서의 전체 문맥을 온전히 읽을 수 있게 해줍니다.

### Grep 최적화: 이 아키텍처의 백미

개인적으로 이 시스템에서 가장 엔지니어링적으로 훌륭하다고 느낀 부분은 `grep -r`의 처리입니다. 가상 파일 시스템에서 네트워크를 타고 모든 파일을 스캔하는 건 미친 짓이죠.

ChromaFs는 `grep` 명령을 가로채서 정규식이나 문자열 검색을 ChromaDB 쿼리(`$contains`, `$regex`)로 변환합니다. DB 레벨에서 일차적으로 거친 필터링(Coarse filter)을 수행하여 일치할 가능성이 있는 파일들을 찾아내고, 이 청크들을 Redis 캐시로 Bulk Prefetch 합니다. 그런 다음 매칭된 파일들만 대상으로 인메모리에서 정교한 필터링(Fine filter)을 수행합니다.

이런 식의 Two-pass 필터링은 대규모 분산 검색 시스템에서 흔히 쓰이는 패턴인데, 이를 가상 쉘의 `grep` 명령어 뒤에 숨겨놓은 설계는 매우 실용적이고 훌륭합니다.

## 왜 FUSE가 아니라 Fake Shell인가?

Hacker News 댓글 창에서 가장 뜨거웠던 논쟁 중 하나는 "왜 FUSE(Filesystem in Userspace)를 써서 진짜 마운트를 하지 않고, TS로 가짜 bash를 흉내 내는 오버엔지니어링을 했는가?"였습니다. 저 역시 글을 읽으며 가장 먼저 든 의문이었습니다.

이에 대해 Mintlify 엔지니어의 답변과 커뮤니티의 통찰이 핵심을 찔렀습니다.

1. **성능과 인프라 오버헤드:** FUSE를 통해 ChromaDB를 연동하는 프로토타입을 만들었으나 너무 느렸다고 합니다. 게다가 FUSE를 쓰려면 결국 에이전트가 실행될 EC2 노드나 샌드박스 인프라를 유지해야 하므로 비용 절감의 의미가 퇴색됩니다.
2. **에이전트의 특성 (가장 중요한 포인트):** "왜 그냥 LLM에게 ChromaDB나 SQL을 직접 쿼리하는 커스텀 도구(Tool)를 주지 않느냐?"는 질문에 대한 답입니다. 최신 LLM들은 코드 학습 데이터의 상당 부분이 `bash`, `grep`, `ls` 같은 표준 도구 사용법에 편중되어 있습니다. 

한 HN 유저의 표현을 빌리자면, **"에이전트에게 커스텀 도구를 쥐여주면 즉시 IQ가 50포인트 하락하지만, 파일 시스템과 표준 CLI 도구를 주면 날아다닌다"** 는 것입니다. 에이전트는 '도구 호출(Tool-calling)용 grep'이 아니라 'bash의 grep' 자체에 최적화되어 학습되었기 때문입니다.

## 총평: 프로덕션 레벨인가, 장난감인가?

이 아키텍처는 명확한 제약 조건(읽기 전용 문서, 제한된 bash 명령어 셋) 하에서 극단적인 효율을 뽑아낸 훌륭한 프라그마틱(Pragmatic) 엔지니어링의 사례입니다.

물론 모든 조직에 이 방식이 은탄환이 될 수는 없습니다. 사내 위키나 문서가 계층 구조 없이 스파게티처럼 엉켜있는 조직이라면, 가상 파일 시스템을 씌워봤자 '예쁘게 포장된 쓰레기통'이 될 뿐입니다. 이런 경우에는 오히려 전통적인 검색 엔진이나 지식 그래프 기반의 접근이 나을 수 있습니다.

하지만 이 사례가 우리에게 주는 진짜 교훈은 **"RAG = Vector Search + Chunking"이라는 맹목적인 공식을 깨부쉈다는 것** 입니다. 정보 검색의 본질은 사용자의 의도와 데이터의 구조를 연결하는 것입니다. 때로는 수백만 차원의 벡터 연산보다, 잘 정리된 디렉토리 구조와 `grep` 명령어 하나가 더 강력한 문맥을 제공할 수 있습니다.

AI 에이전트를 위한 시스템을 설계하고 계신가요? 그렇다면 무작정 임베딩 모델부터 돌리기 전에, 우리가 가진 데이터가 어떤 구조를 띠고 있는지, 그리고 에이전트가 가장 익숙해하는 인터페이스가 무엇인지 다시 한번 고민해 보시길 바랍니다.
