컴파일러 툴체인을 다뤄본 엔지니어라면 누구나 공감할 만한 업계의 격언이 하나 있다. "LLVM 위에서 충분히 오래 살아남은 언어는 결국 자신만의 커스텀 IR을 만들게 된다." 심지어 Clang조차도 이 과정을 겪고 있으니 말이다.

최근 C++ 및 LLVM 기반의 Clojure 방언인 Jank가 자신들만의 커스텀 IR(Intermediate Representation)을 도입했다는 흥미로운 글을 발표했다. 목표는 명확하다. 오픈소스 생태계에서 가장 강력한 런타임 중 하나인 JVM과 정면승부를 벌이겠다는 것이다. 

개인적으로 수많은 'JVM 킬러'들을 보아왔지만, 대부분은 마이크로 벤치마크에서만 반짝하고 실제 프로덕션의 복잡도 앞에서는 무너지는 경우가 많았다. 하지만 이번 Jank의 접근 방식은 컴파일러 아키텍처의 근본적인 한계를 짚고 넘어갔다는 점에서 상당히 인상적이다. 이들이 왜 LLVM IR을 버리고(혹은 그 앞단에) 커스텀 IR을 두기로 결정했는지, 그리고 어떻게 피보나치 벤치마크에서 JVM을 압도했는지 엔지니어의 시각에서 파헤쳐보자.

## 왜 LLVM IR만으로는 부족했을까?

흔히들 LLVM을 사용하면 수많은 아키텍처 백엔드와 강력한 최적화를 '공짜로' 얻는다고 생각한다. C나 Rust 같은 정적 타입 언어들에게는 사실이다. 하지만 Clojure처럼 극도로 동적이고 다형성이 높은 언어에게 LLVM IR은 너무 저수준(low-level)이다.

LLVM은 Clojure의 `var`, 영속적 자료구조(persistent data structures), 지연 시퀀스(lazy sequences) 같은 고수준의 의미론(semantics)을 전혀 이해하지 못한다. Jank 컴파일러가 생성한 LLVM IR을 LLVM 옵티마이저에 던져봤자, LLVM 입장에서는 그저 수많은 포인터 역참조와 동적 함수 호출의 난해한 조합으로 보일 뿐이다. 의미를 모르니 과감한 최적화(DCE, 인라이닝 등)를 수행할 수 없다.

이 한계를 극복하기 위해 Jank는 **Clojure의 의미론을 이해하는 고수준 IR** 을 도입했다. 이 IR은 SSA(Single Static Assignment) 형태를 띠며 CFG(Control Flow Graph)로 표현된다. 범용 목적이 아니기 때문에 오직 Jank만을 위한 강력한 최적화가 가능해진 것이다.

## 5단계 최적화: 5522ms에서 114ms로

Jank 팀은 재귀적 피보나치 수열을 벤치마크로 삼았다. 단순해 보이지만 다형성 산술 연산, 재귀, GC(Garbage Collection) 부하를 모두 테스트할 수 있는 훌륭한 타겟이다. 

JVM(OpenJDK 21) 환경에서 Clojure는 이 연산에 약 **200ms** 가 걸렸다. 반면 최적화 전 Jank의 초기 기록은 **5522ms** 였다. 처참한 수준이다. 하지만 5단계의 최적화를 거치며 이 수치는 극적으로 변한다.

### 1. 산술 연산 인라이닝 (5522ms -> 2309ms)
Clojure는 메타데이터를 통해 인라이닝을 처리한다. Jank는 `clojure.core/max` 같은 산술 함수 호출 시, 분석 단계에서 메타데이터를 확인하고 C++ 런타임 함수(`jank::runtime::max`)를 직접 호출하도록 만들었다. 

```clojure
(defn
  ^{:inline (fn [l r] (list 'cpp/jank.runtime.max l r))
    :inline-arities #{2}}
  max
  ([x] x)
  ([l r] (cpp/jank.runtime.max l r)))
```
이 방식은 매크로 확장과 유사하게 동작하며, 불필요한 `var` 역참조와 박싱(boxing) 오버헤드를 통째로 날려버린다. 이 조치 하나로 실행 시간이 절반 이하로 줄었다.

### 2. 불필요한 IR 명령어 제거 (2309ms -> 2247ms)
초기 IR에서는 boolean 결과를 굳이 객체로 박싱(`:cpp/into-object`)한 뒤, 다시 분기를 위해 truthy 여부를 검사(`:truthy`)하는 비효율이 있었다. C++ 런타임에서 넘어온 `bool` 값을 분기문에 직접 사용하도록 IR 생성기를 수정하여 소폭의 성능 향상을 이뤘다.

### 3. `nil` 초기화 병목 제거 (2247ms -> 1400ms)
C++은 번역 단위(Translation Unit) 간의 전역 변수 초기화 순서를 보장하지 않는다. Jank는 이 문제(Static Initialization Order Fiasco)를 피하려고 `nil`을 `jank_nil()`이라는 함수 뒤에 숨겨두었는데, 프로파일링 결과 여기서 엄청난 병목이 발생하고 있었다.

해결책은 우아했다. AOT 컴파일된 전역 변수들은 어차피 모듈 로드 시점에 재초기화된다는 점을 이용해, 박스드 포인터 타입이 기본적으로 `nullptr`로 초기화되도록 생성자를 수정한 것이다. 덕분에 `nil`을 함수 호출 없이 전역 값으로 직접 참조할 수 있게 되었다.

### 4. 포인터 태깅 (1400ms -> 282ms)
개인적으로 이번 최적화의 하이라이트라고 생각한다. 64비트 시스템에서 포인터는 8바이트 정렬(aligned)을 하므로, **포인터 주소의 하위 3비트는 항상 000** 이 된다. 

Jank는 이 하위 비트 중 최하단 비트를 `1`로 세팅하여 해당 값이 포인터가 아닌 '인코딩된 정수'임을 나타내는 **Pointer Tagging** 기법을 도입했다. 나머지 63비트에 실제 정수 데이터를 담는 것이다.

```text
일반 포인터: pppppppp pppppppp ... ppppp000
인코딩 정수: xxxxxxxx xxxxxxxx ... xxxxxxx1
```

이 기법 덕분에 63비트 이하의 정수 연산에서 발생하는 모든 동적 메모리 할당(GC 부하)이 완벽하게 사라졌다. 1400ms에서 282ms로의 비약적인 도약은 여기서 나왔다. V8이나 JVM 내부에서도 적극적으로 쓰이는 이 클래식한 기법이 새로운 런타임에 이식되는 걸 보는 건 언제나 즐겁다.

### 5. 강력한 C++ 인라이닝 (282ms -> 114ms)
마지막으로 Clang 컴파일러가 산술 연산을 무조건 인라이닝하도록 C++ 어트리뷰트를 강제했다.

```cpp
template <typename L, typename R>
[[gnu::always_inline, gnu::flatten, gnu::hot]]
auto add(L const l, R const r) { ... }
```

결과는 **114ms**. JVM의 200ms를 가볍게 뛰어넘었다.

## Hacker News의 반응과 나의 생각

이 글이 Hacker News에 올라온 후, 시니어 엔지니어들 사이에서 꽤 심도 있는 논의가 오갔다. 몇 가지 짚어볼 만한 포인트가 있다.

- **왜 MLIR을 쓰지 않았나?:** 누군가 "MLIR을 쓰면 되지 왜 굳이 커스텀 IR을 만드냐"고 질문했다. 사실 MLIR은 루프 최적화나 벡터화에 기가 막힌 도구다. 하지만 개인 개발자나 소규모 팀이 MLIR 다이얼렉트를 정의하고 유지보수하는 건 배보다 배꼽이 더 큰 작업이다. Jank의 선택은 지극히 실용적이었다.
- **JIT vs AOT의 본질적 차이:** Jank가 JVM을 이겼다고 해서 "Jank > JVM"이라고 결론 내리는 것은 섣부르다. JVM의 진가는 런타임에 프로그램의 동작을 분석하여 공격적으로 최적화하는 JIT 컴파일러에 있다. Jank는 AOT(Ahead-of-Time) 방식이므로 시작부터 빠르지만, JVM처럼 실행 시간이 길어질수록 점점 더 빨라지는 마법은 기대하기 어렵다.
- **동적 언어의 인라이닝 딜레마:** 인라이닝을 공격적으로 하면 성능은 오르지만, REPL에서 함수를 재정의(Redefinition)했을 때 호출자(Caller)들의 동작이 업데이트되지 않는 문제가 생긴다. HN 댓글에서는 Julia 언어의 **World-age 시스템** 이 완벽한 해결책으로 언급되었다. 함수나 상수가 재정의될 때마다 '세계의 나이'를 증가시키고, 의존성 백에지(backedge)를 추적해 무효화하는 방식이다. Jank 팀도 장기적으로는 이런 고도화된 무효화 메커니즘을 고민해야 할 것이다.

## 결론

Jank가 당장 내일 프로덕션에 투입될 수 있느냐고 묻는다면 대답은 '아니오'다. 피보나치는 어디까지나 마이크로 벤치마크일 뿐이며, 복잡한 상태를 다루는 실제 애플리케이션에서는 JVM의 성숙한 GC와 적응형 JIT 컴파일러가 여전히 압도적인 우위를 점할 것이다.

하지만 Jank 팀이 보여준 엔지니어링 접근 방식은 훌륭하다. 언어의 의미론을 보존하기 위해 과감히 커스텀 IR을 설계하고, 포인터 태깅 같은 로우레벨 최적화를 직접 구현하며 병목을 하나씩 박살 내는 과정은 컴파일러 개발의 정석을 보여준다. 

LLVM이라는 거대한 어깨 위에 올라타면서도, 그 한계를 명확히 인지하고 자신만의 추상화 계층을 쌓아 올린 Jank. 이 프로젝트가 앞으로 어떤 대규모 벤치마크 결과를 가져올지 진심으로 기대된다.
