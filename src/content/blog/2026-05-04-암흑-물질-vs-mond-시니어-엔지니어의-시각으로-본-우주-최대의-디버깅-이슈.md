엔지니어로서 우리는 코드에 등장하는 '매직 넘버(Magic Number)'나 원인을 알 수 없는 '땜빵(Fudge factor)'을 혐오한다. 시스템의 메트릭이 예상과 다르게 튈 때, 우리는 보통 두 가지 선택의 기로에 놓인다. 데이터베이스에 우리가 모르는 숨겨진 레코드가 존재한다고 가정하거나, 아니면 핵심 비즈니스 로직(Core Library) 자체에 버그가 있다고 의심하는 것이다.

최근 Science지에 흥미로운 기사가 올라왔다. 뉴턴의 중력 법칙이 역대 최대 규모의 테스트를 통과했다는 내용이다. 이는 단순히 물리학계의 뉴스를 넘어, 현재 우주론에서 가장 뜨거운 논쟁거리인 암흑 물질(Dark Matter)과 수정 뉴턴 역학(MOND) 간의 아키텍처 논쟁을 다시 수면 위로 끌어올렸다. 

15년 넘게 대규모 분산 시스템을 디버깅해 온 엔지니어의 관점에서, 이 논쟁은 거대한 레거시 시스템의 버그를 추적하는 과정과 소름 돋도록 닮아있다.

## 우주라는 시스템의 관측된 버그

문제의 발단은 단순하다. 은하의 회전 속도를 측정(Monitoring)해 보니, 중심부에서 멀어질수록 속도가 줄어들어야 정상인데 실제로는 속도가 떨어지지 않고 일정하게 유지되는 현상이 발견된 것이다. 뉴턴과 아인슈타인이 작성한 현재의 중력 API(`calculate_gravity`)에 따르면, 이 은하들은 진작에 산산조각 났어야 한다.

이 버그를 픽스하기 위해 물리학자들은 두 가지 대형 PR(Pull Request)을 제안했다.

### PR 1: 암흑 물질 (숨겨진 데이터 추가)
가장 주류가 되는 해결책이다. 눈에 보이지 않지만 질량을 가진 데이터(암흑 물질)가 시스템 도처에 깔려있다고 가정하는 것이다.

### PR 2: MOND (코어 로직 수정)
수정 뉴턴 역학(Modified Newtonian Dynamics)이라 불리는 이 접근법은, 가속도가 극도로 낮아지는 엣지 케이스(Edge case)에서는 중력 공식 자체가 다르게 동작해야 한다고 주장한다.

이 두 가지 접근법을 코드로 표현하면 다음과 같다.

```python
# 기존의 Standard API
def calculate_gravity(mass, distance):
    return G * mass / (distance ** 2)

# PR 1: 암흑 물질 (Dark Matter) 패치
# 관측되지 않는 숨겨진 질량을 추가하여 해결
def calculate_gravity_dm(visible_mass, distance):
    total_mass = visible_mass + get_hidden_dark_matter_halo()
    return G * total_mass / (distance ** 2)

# PR 2: MOND 패치
# 특정 임계치(a0) 이하의 엣지 케이스에서 로직 자체를 분기
def calculate_gravity_mond(mass, distance, acceleration):
    if acceleration < a0:
        return math.sqrt(G * mass * a0) / distance
    return calculate_gravity(mass, distance)
```

## 해커뉴스 커뮤니티의 뜨거운 논쟁: 발칸(Vulcan)인가 넵튠(Neptune)인가?

이번 기사가 Hacker News에 올라온 후, 시니어 개발자들과 과학 덕후들 사이에서 엄청난 토론이 벌어졌다. 그중 가장 내 눈길을 끈 것은 1800년대의 천문학 역사에 비유한 한 유저의 통찰이었다.

- **해왕성의 사례:** 과거 천왕성의 궤도가 뉴턴의 계산과 어긋났을 때, 과학자들은 '우리가 보지 못한 미지의 행성이 중력을 행사하고 있을 것'이라고 예측했다. 그리고 실제로 해왕성(Neptune)이 발견되었다. 이는 암흑 물질 접근법의 대성공 사례라 할 수 있다.
- **발칸의 사례:** 이후 수성의 궤도에서도 비슷한 오차가 발견되었다. 과학자들은 이번에도 태양과 수성 사이에 '발칸'이라는 미지의 행성이 있을 것이라 확신하고 수십 년간 찾았지만 실패했다. 결국 이 버그는 아인슈타인이 일반 상대성 이론이라는 '코어 로직 재작성'을 통해 해결했다. 이는 MOND 지지자들이 꿈꾸는 시나리오다.

과연 지금의 암흑 물질은 제2의 해왕성일까, 아니면 제2의 발칸일까?

## 총평: 왜 나는 암흑 물질(LCDM)의 손을 들어주는가

솔직히 말해서, 나는 주니어 시절 암흑 물질이라는 개념을 처음 접했을 때 이를 그저 수식을 맞추기 위한 땜빵(Fudge factor)이라고 생각했다. '관측은 안 되는데 아무튼 존재함'이라는 설명은 엔지니어의 직관에 강력히 위배되기 때문이다. 우주 공간을 채우고 있다는 과거의 '에테르(Aether)' 이론이나 다를 바 없어 보였다.

하지만 시스템 아키텍처를 깊이 파고들수록, 그리고 HN의 여러 토론과 총알 성단(Bullet Cluster) 같은 실제 관측 데이터를 살펴볼수록 내 생각은 바뀌었다. 

암흑 물질 모델(LCDM)은 단순히 은하의 회전 속도라는 하나의 버그만 잡는 것이 아니다. 우주 배경 복사(CMB)의 패턴, 중력 렌즈 현상 등 시스템 전반의 다양한 메트릭을 놀랍도록 정확하게 예측(Predict)해낸다. 반면 MOND는 은하 회전 곡선이라는 단일 이슈에는 기가 막히게 들어맞지만, 다른 지표들을 설명하기 위해서는 TeVeS 같은 복잡한 텐서-벡터-스칼라 이론을 덧붙이며 억지스러운 하드코딩을 남발해야 한다.

가장 결정적인 증거는 **암흑 물질이 없는 은하(NGC 1052-DF2 등)** 의 발견이다. 만약 중력의 코어 로직 자체가 문제(MOND)라면, 우주의 모든 은하가 동일한 버그를 겪어야 한다. 하지만 어떤 은하에는 암흑 물질의 효과가 있고 어떤 은하에는 없다면? 이는 코어 로직의 문제가 아니라, 특정 노드에만 캐시 데이터(암흑 물질)가 배포되지 않은 상태라고 보는 것이 타당하다.

우리는 아직 '모든 것의 이론(Theory of Everything)'이라는 완벽한 V2.0 아키텍처를 가지지 못했다. 누군가 획기적인 양자 중력 이론을 들고나와 이 모든 것을 리팩토링하기 전까지, 암흑 물질은 우리가 가진 가장 훌륭하고 실용적인 Production-ready 모델이다. 보이지 않는다고 해서 존재하지 않는 것은 아니다. 우리가 매일 의존하는 수많은 백그라운드 데몬 프로세스들처럼 말이다.
