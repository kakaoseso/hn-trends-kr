요즘 IT 업계를 보면 "AI가 게임 코딩을 다 해준다", "언리얼 엔진 5 나나이트(Nanite) 쓰면 폴리곤 무제한이니까 최적화 끝이다" 같은 소리가 심심찮게 들립니다. 현업에서 15년 넘게 구르고 있는 엔지니어로서 솔직히 헛웃음이 나옵니다. 물리 법칙과 하드웨어의 한계는 AI가 마법처럼 해결해주지 않습니다. 

제가 겪어본 그래픽스 최적화의 80%는 결국 하나의 철학으로 귀결됩니다. **"안 해도 되는 뻘짓을 어떻게 빨리 알아채고 안 할 것인가?"**

오늘은 Krupitskas의 블로그 글과 해커뉴스(Hacker News)의 열띤 토론을 바탕으로, 모던 렌더러들이 화면에 안 보이는 것들을 어떻게 지워내는지, 즉 **컬링(Culling)** 기법의 발전사를 딥다이브 해보겠습니다. 

## 1. 뻔하지만 가장 중요한 기본기

가장 싸고, 가장 범용적인 기술들입니다. 무거운 연산을 돌리기 전에 명백한 쓰레기들을 먼저 걸러내는 1차 방어선이죠.

- **Distance Culling:** 카메라에서 일정 거리 이상 멀어지면 그리지 않습니다. 아주 단순하지만 팝인(Pop-in) 현상을 막기 위해 Dithered fade-out이나 임포스터(Impostor, 원경용 빌보드) 처리가 필수적입니다.
- **Frustum Culling:** 카메라의 시야(Frustum)를 벗어난 오브젝트를 바운딩 박스(AABB)나 구형을 이용해 통째로 날립니다.
- **Backface Culling:** 닫힌 메쉬의 뒷면(내부)을 안 그립니다. 

여기서 주니어들이 흔히 착각하는 게 있습니다. 전통적인 파이프라인에서 Backface Culling은 **Vertex Shader가 끝난 후** 래스터라이저(Rasterizer) 단계에서 발생합니다. 즉, 화면에 안 보이는 뒷면이라 할지라도 Vertex 연산 비용은 꼬박꼬박 지불하고 있다는 뜻입니다. 이 낭비를 막기 위해 등장한 것이 뒤에서 설명할 GPU-Driven 파이프라인의 Meshlet 컬링입니다.

## 2. 오클루전 컬링(Occlusion Culling)과 메타버스의 딜레마

건물 뒤에 가려진 자동차를 굳이 그릴 필요가 있을까요? 오클루전 컬링은 '무엇이 다른 것에 가려졌는가'를 판별합니다. 도심지나 실내 씬에서 엄청난 성능 향상을 가져오지만, 그만큼 구현이 까다롭습니다.

요즘 업계 표준은 **Hi-Z(Hierarchical Z-Buffer)를 이용한 Two-Pass Occlusion Culling** 입니다. 이전 프레임의 Depth 버퍼를 Mip chain으로 만들어(Depth Pyramid) 대략적인 화면 공간 바운딩 박스와 테스트하는 방식입니다.

1. 이전 프레임에서 보였던 오브젝트들을 먼저 그리고, 새로운 Hi-Z를 만듭니다.
2. 이전 프레임에서 가려졌던(Culled) 오브젝트들을 새로운 Hi-Z에 다시 테스트합니다. 방금 시야에 들어온 녀석들에게 두 번째 기회를 주는 거죠.

해커뉴스 스레드에서 한 유저가 남긴 코멘트가 제 뼈를 때렸습니다. 유저가 직접 콘텐츠를 만드는 UGC 기반의 메타버스(예: 세컨드 라이프)에서 오클루전 컬링이 얼마나 지옥 같은지에 대한 이야기였습니다.

> "창문이 없는 방에선 프레임이 잘 나옵니다. 하지만 문을 열고 밖으로 나가면 투명한 유리창 너머로 건물 내부가 보이고, 시야는 끝없이 열려 있죠. 이런 환경에선 컬링 연산 비용이 그리는 비용을 초과하는 대참사가 일어납니다."

메타(Meta)가 그 천문학적인 돈을 쏟아붓고도 호라이즌 월드의 그래픽을 획기적으로 개선하지 못한 이유 중 하나가 바로 이겁니다. 예측 불가능한 씬 구조에서는 오클루전 컬링의 효율이 극단적으로 떨어집니다.

## 3. GPU-Driven Rendering: CPU를 해방시켜라

제가 2018년쯤 처음 이 구조를 프로덕션에 도입했을 때의 쾌감이 아직도 생생합니다. 과거에는 CPU가 `for` 루프를 돌며 가시성을 판단하고 수만 개의 Draw call을 쐈습니다. 이제는 그 역할을 GPU Compute Shader로 넘깁니다.

```cpp
// Compute Shader: 오브젝트 당 하나의 스레드 할당
[numthreads(64, 1, 1)]
void CullCS(uint id : SV_DispatchThreadID)
{
    if (id >= g_objectCount) return;
    ObjectData obj = g_objects[id];
    
    if (!SphereInFrustum(obj.center, obj.radius)) return;
    
    uint slot;
    InterlockedAdd(g_drawCount[0], 1, slot); // 살아남은 녀석들만 버퍼에 압축
    g_drawArgs[slot] = MakeDrawArgs(obj);
}
```

CPU는 그저 `ExecuteIndirect` 명령 한 방만 날리면 됩니다. GPU가 스스로 판단하고 스스로 그리는 것이죠.

여기서 한 발 더 나아간 것이 **Meshlet / Cluster Culling** 입니다. 거대한 메쉬를 64~128개의 삼각형 단위(Meshlet)로 쪼개고, 각 Meshlet의 Normal Cone(법선이 향하는 방향의 범위)을 계산합니다. 만약 Meshlet의 모든 법선이 카메라 반대편을 향한다면? 통째로 날려버립니다. 앞서 말한 Vertex Shader의 낭비를 원천 차단하는 겁니다.

## 4. Mesh Shader와 서브 픽셀의 세계

최신 API의 Mesh Shader를 사용하면 래스터라이제이션이 시작되기도 전에 **삼각형 단위(Triangle level)** 로 컬링이 가능합니다. `SV_CullPrimitive`를 통해 픽셀 셰이더 호출 없이 삼각형을 삭제할 수 있죠.

특히 **Small Triangle Culling** 이 예술입니다. 화면에서 1픽셀보다 작거나 픽셀 중심에 걸치지 않는 미세한 삼각형들은 어차피 그려봤자 화면에 나오지 않습니다. 고해상도 스캔 데이터나 멀리 있는 머리카락, 빽빽한 나뭇잎을 렌더링할 때 이 기법 하나로 엄청난 GPU 사이클을 아낄 수 있습니다.

## 5. 결론: 나나이트(Nanite)는 마법이 아니다

언리얼 엔진 5의 나나이트를 보고 많은 분들이 경악했지만, 사실 나나이트는 무에서 유를 창조한 기술이 아닙니다. 오늘 다룬 Cluster Culling, Screen-space error 판별, 그리고 미세 삼각형을 처리하기 위한 커스텀 Software Rasterizer를 극단적으로 잘 조합한 **엔지니어링의 정수** 일 뿐입니다.

해커뉴스 스레드에서 한 노병이 "90년대 소프트웨어 렌더러 시절, 매 프레임 Frustum 테스트를 하는 게 너무 무거워서 N 프레임마다 한 번씩만 검사하도록 꼼수를 썼다"고 회고하더군요. 카메라를 확 돌리면 팝인이 생겼겠지만, 당시 하드웨어에선 그게 최선의 엔지니어링이었을 겁니다.

렌더링에 은통알(Silver bullet)은 없습니다. 
무작정 최신 GPU-Driven 파이프라인을 쑤셔 넣기 전에, 여러분의 게임이 CPU 바운드인지, Vertex 바운드인지, 아니면 단순히 Overdraw가 심한 건지부터 프로파일링 하시기 바랍니다. 싸고 단순한 테스트(Distance, Frustum)부터 시작해서, 씬의 밀도에 맞춰 복잡한 레이어(Hi-Z, Meshlet)를 쌓아 올리는 것. 그것이 진짜 그래픽스 엔지니어링입니다.
