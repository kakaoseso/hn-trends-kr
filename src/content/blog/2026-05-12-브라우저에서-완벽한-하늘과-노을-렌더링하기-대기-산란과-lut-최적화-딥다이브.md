그래픽스 프로그래밍을 하다 보면 누구나 한 번쯤 빠지는 함정이 있습니다. 바로 '하늘 렌더링'이죠. 대충 파란색 그라데이션을 배경으로 깔아놓고 만족하려 하지만, 왠지 모르게 촌스럽고 어색합니다. 왜 그럴까요? 진짜 하늘의 색은 단순한 픽셀의 나열이 아니라, 빛과 대기 입자들이 만들어내는 거대한 물리 엔진의 결과물이기 때문입니다.

최근 Maxime Heckel의 블로그에서 이 주제를 아주 깊게 다룬 글을 읽었습니다. 단순히 하늘을 그리는 것을 넘어, 우주 궤도에서 바라보는 지구의 대기층과 노을, 그리고 행성 규모의 렌더링까지 다루는 꽤나 흥미로운 여정이더군요. 오늘은 이 아티클을 바탕으로 브라우저 상에서 대기 산란(Atmospheric Scattering)을 어떻게 구현하고 최적화하는지 기술적으로 파헤쳐 보겠습니다.

## 하늘의 물리학: Raymarching과 산란
가장 먼저 이해해야 할 것은 빛이 대기를 통과할 때 일어나는 현상입니다. 이를 시뮬레이션하기 위해 우리는 **Raymarching** 기법을 사용합니다. 카메라에서 씬으로 광선을 쏘고, 대기를 통과하며 빛이 얼마나 살아남는지(Transmittance), 그리고 얼마나 카메라 쪽으로 산란되는지(Scattering)를 계산하는 과정이죠.

- **Rayleigh Scattering:** 공기 분자와 같이 빛의 파장보다 작은 입자에 의한 산란입니다. 파란색처럼 파장이 짧은 빛이 더 많이 산란되기 때문에 낮에는 하늘이 파랗게 보입니다. 반대로 해가 질 때는 빛이 통과해야 하는 대기층이 길어져 파란색은 모두 튕겨나가고 붉은색만 우리 눈에 도달하게 되죠.
- **Mie Scattering:** 먼지나 수증기 같은 비교적 큰 입자에 의한 산란입니다. 태양 주변에 희뿌연 후광(Halo)이 생기거나 안개가 낀 것처럼 보이게 만듭니다.
- **Ozone Absorption:** 오존층이 특정 파장의 빛을 흡수하는 현상입니다. 해가 질 무렵 하늘에 오묘한 보라색이나 짙은 푸른색 톤을 더해주는 핵심 요소입니다.

아래는 Rayleigh 밀도를 구하고 Optical Depth를 누적하는 핵심 셰이더 코드입니다. 고도가 높아질수록 대기가 희박해지는 것을 지수 함수로 표현한 것이 인상적입니다.

```glsl
const float RAYLEIGH_SCALE_HEIGHT = 8.0; // km
const float ATMOSPHERE_HEIGHT = 100.0; // km

float rayleighDensity(float h) {
    return exp(-max(h, 0.0) / RAYLEIGH_SCALE_HEIGHT);
}
```

## 배경을 넘어 볼륨으로: Post-processing과 Depth
단순히 배경(Skybox)으로만 쓴다면 위의 로직만으로도 훌륭합니다. 하지만 씬 안에 있는 3D 오브젝트들과 상호작용하려면 이야기가 달라집니다. 대기 효과가 오브젝트 앞뒤로 자연스럽게 깔리게 하려면 이 효과를 Post-processing 파이프라인으로 가져와야 합니다.

이를 위해 Depth buffer를 읽어와 Screen UV 좌표를 World Space로 복원하는 과정이 필요합니다. 예전에 저도 볼류메트릭 라이팅을 구현할 때 이 부분에서 꽤나 고생했던 기억이 납니다.

```glsl
vec3 getWorldPosition(vec2 uv, float depth) {
    float clipZ = depth * 2.0 - 1.0;
    vec2 ndc = uv * 2.0 - 1.0;
    vec4 clip = vec4(ndc, clipZ, 1.0);

    vec4 view = projectionMatrixInverse * clip;
    vec4 world = viewMatrixInverse * view;

    return world.xyz / world.w;
}
```

여기에 행성 규모의 스케일을 다루기 위해 Logarithmic depth buffer를 활성화하고, Ray-sphere intersection 수학을 이용해 광선이 대기권(구체)을 언제 통과하는지 계산합니다. 이 수학적 토대가 완성되면 지구뿐만 아니라 상수 몇 개만 바꿔서 화성의 붉은 노을까지 렌더링할 수 있게 됩니다.

## 성능의 벽과 LUT 최적화
솔직히 말씀드리면, 앞서 설명한 방식대로 프래그먼트 셰이더에서 매 픽셀마다 이중 루프(Primary step + Lightmarch step)를 돌리는 것은 실무에서 **미친 짓** 에 가깝습니다. 모바일은 물론이고 웬만한 데스크탑 GPU도 비명을 지를 겁니다.

이 지점에서 엔지니어링의 꽃인 최적화가 등장합니다. 저자는 Sebastian Hillaire의 논문을 바탕으로 **LUT** (Look Up Table) 기반 접근법을 도입합니다. 무거운 계산을 미리 텍스처에 구워놓고(Baking), 렌더링할 때는 텍스처를 샘플링만 하는 아주 우아한 패턴입니다.

![Diagram showcasing the sampling processes for our Sky-View and Aerial Perspective LUTs](https://assets.maximeheckel.com/images/blog/lut-raymarching.webp?w=3840&q=100)

세 가지 주요 LUT를 생성합니다.
- **Transmittance LUT:** 빛이 대기를 통과하며 얼마나 살아남는지를 각도와 고도에 따라 미리 계산합니다.
- **Sky-View LUT:** 지상에서 특정 방향을 바라봤을 때 하늘의 색상을 저장합니다.
- **Aerial Perspective LUT:** 카메라와 씬의 오브젝트 사이 공간에 존재하는 대기 안개(Haze)의 양을 저장합니다.

이 방식은 연산량을 극단적으로 줄여줍니다. 빛의 도달 여부를 알기 위해 매번 태양을 향해 Raymarching을 할 필요 없이, Transmittance LUT의 픽셀 하나만 읽어오면 되니까요.

## Hacker News의 반응과 개인적인 생각
Hacker News 스레드를 보면 누군가 과거에 많이 쓰이던 Perez All-Weather나 Preetham 스카이 모델을 언급하더군요. 2010년대 초반 게임 엔진에서는 그런 분석적(Analytical) 모델을 많이 썼습니다. 하지만 최근 그래픽스 트렌드는 이 아티클처럼 물리 기반의 볼류메트릭 모델로 완전히 넘어왔습니다. 훨씬 더 사실적이고 다양한 시간대와 환경을 유연하게 커버할 수 있기 때문이죠.

다만, WebGL 환경에서 FBO를 여러 개 띄워가며 LUT를 굽는 과정은 프론트엔드 엔지니어에게 꽤나 고통스러운 작업입니다. 저자 본인도 회고했듯, 애초에 WebGPU의 Compute Shader를 활용했다면 아키텍처가 훨씬 깔끔했을 텐데 하는 아쉬움은 남습니다. Screen-space depth-aware 이펙트는 언제나 엣지 케이스(Banding, Flickering 등)와의 싸움이니까요.

## 총평
이 아티클은 단순한 튜토리얼을 넘어 셰이더 수학과 최적화 기법을 엿볼 수 있는 마스터클래스입니다. 당장 내일 회사 프로젝트의 랜딩 페이지에 이 무거운 셰이더를 올릴 일은 없겠지만, 빛과 대기가 상호작용하는 물리 법칙을 코드로 번역하고 이를 브라우저 위에서 실시간으로 돌리기 위해 최적화해 나가는 과정은 모든 엔지니어에게 큰 영감을 줍니다. 우주나 비행 시뮬레이션, 혹은 고품질의 3D 웹 경험을 준비하고 있다면 반드시 정독해야 할 자료입니다.
