15년 넘게 소프트웨어 엔지니어링과 시스템 최적화를 다루면서 깨달은 진리가 하나 있습니다. 성능 최적화의 끝판왕은 결국 '안 해도 되는 일을 찾아내서 안 하는 것'이라는 점입니다. 대규모 선형 대수 시스템이나 고성능 컴퓨팅(HPC) 분야에서 다루는 Sparse Matrix(희소 행렬) 연산이 정확히 이 철학을 따릅니다.

최근 Reid Atcheson의 블로그에서 [Sparse Cholesky Elimination Tree](https://www.reidatcheson.com/sparse/linear/cholesky/2026/04/09/etree.html)에 대한 아주 흥미로운 글을 읽었습니다. 보통 이런 주제는 딱딱한 그래프 이론과 수학적 증명으로 시작하기 마련인데, 이 글은 실제 Right-looking Cholesky 알고리즘 코드에서 출발해 Task DAG를 그리고, 이를 압축해 트리를 만들어내는 실용적인 접근을 보여줍니다. "아, 이게 진짜 엔지니어의 바텀업(Bottom-up) 접근 방식이지"라는 생각이 절로 들더군요.

오늘은 이 글을 바탕으로, 희소 행렬 연산의 핵심인 Elimination Tree(제거 트리)가 무엇인지, 그리고 이것이 왜 시스템 설계 관점에서도 훌륭한 인사이트를 주는지 깊게 파헤쳐 보겠습니다.

### 왜 Dense 알고리즘을 그대로 쓰면 안 될까?

Cholesky 분해는 대칭 정정부호 행렬(Symmetric Positive Definite Matrix) $A$를 하삼각 행렬 $L$과 그 전치 행렬의 곱, 즉 $A = LL^T$로 분해하는 알고리즘입니다. 

만약 행렬 $A$가 대부분 0으로 채워진 희소 행렬이라면 어떨까요? Dense(밀집) 행렬용 알고리즘을 그대로 돌리면 CPU 사이클과 메모리를 엄청나게 낭비하게 됩니다. 더 큰 문제는 **Fill-in(채워짐)** 현상입니다. 원래 행렬 $A$에는 0이었던 위치가, 연산 과정(Rank-1 update)을 거치면서 행렬 $L$에서는 0이 아닌 값으로 채워지는 현상이죠.

원문의 Dense Right-looking Cholesky 의사 코드를 잠시 보겠습니다.

```c
// 3. Right-looking rank-1 update of trailing matrix
// Note that it is only this step which will create fill-in.
for (int j = k + 1; j < n; ++j) {
    for (int i = j; i < n; ++i) { // lower triangle only
        L[i][j] -= L[i][k] * L[j][k];
    }
}
```

이 3번 스텝이 바로 Fill-in의 주범입니다. 여기서 매우 중요한 구조적 규칙(Structural rule)을 하나 도출할 수 있습니다.

- **규칙:** $k < j \le i$ 일 때, $L[i][k] \neq 0$ 이고 $L[j][k] \neq 0$ 이면, $L[i][j] \neq 0$ 이다.

즉, $k$번째 열(Column)을 처리할 때 이미 존재하는 0이 아닌 값들이 곱해지면서 미래의 $j$번째 열에 새로운 0이 아닌 값을 만들어낸다는 뜻입니다.

### Task DAG에서 Elimination Tree로

연산의 의존성을 그래프로 그려보면 자연스럽게 DAG(Directed Acyclic Graph)가 나옵니다. 하지만 희소 행렬의 특성상 불필요한 연산을 가지치기(Pruning)하고 나면, 꽤 복잡해 보이는 의존성 그래프가 남습니다.

여기서 원작자의 통찰이 빛을 발합니다. 앞서 말한 구조적 규칙을 다시 생각해 봅시다. 노드 0에서 1로 가는 엣지($L[1][0] \neq 0$)가 있고, 0에서 2로 가는 엣지($L[2][0] \neq 0$)가 있다면, 규칙에 의해 필연적으로 1에서 2로 가는 엣지($L[2][1] \neq 0$)가 존재해야 합니다.

그렇다면 0에서 2로 가는 직접적인 엣지는 사실상 **중복(Redundant)** 입니다. 어차피 0 -> 1 -> 2 순서로 연산이 진행되어야 하니까요. 이렇게 DAG에서 중복된 엣지를 모두 걷어내면, 놀랍게도 깔끔한 **Tree(트리)** 구조가 남게 됩니다. 이것이 바로 Elimination Tree입니다.

### Symbolic Factorization: 구조를 먼저, 계산은 나중에

솔직히 말해서 제가 이 글에서 가장 감탄한 부분은 이 트리를 활용하는 패턴입니다. HPC나 수치해석 라이브러리를 바닥부터 짜본 분들이라면 아시겠지만, 메모리 할당을 루프 안에서 동적으로 하는 것은 성능의 무덤입니다.

원문에서는 이를 해결하기 위해 두 단계로 작업을 분리합니다.

1. **Symbolic Phase:** $O(n)$ 시간에 Elimination Tree를 구축하고, 이를 따라가며 최종 행렬 $L$의 어디에 0이 아닌 값이 들어갈지(Fill-in 패턴) 미리 계산하여 메모리를 한 번에 할당합니다.
2. **Numeric Phase:** 이미 할당된 구조 위에서 실제 부동소수점 연산만 빠르게 수행합니다.

이러한 패턴은 비단 선형 대수뿐만 아니라, 컴파일러 최적화나 분산 시스템의 Task Scheduler를 설계할 때도 똑같이 적용되는 훌륭한 아키텍처 패턴입니다. 실행 전에 의존성 트리를 분석해 파이프라인과 자원을 미리 세팅하는 방식이죠.

```c
// O(n)으로 Elimination Tree를 계산하는 핵심 로직
int v = c;
while (v != -1 && v < r) {
    int next = ancestor[v];
    ancestor[v] = r; // 현재 경로를 상위 노드 r로 연결
    if (next == -1) {
        parent[v] = r; // 처음 도달한 상위 노드가 부모가 됨
        break;
    }
    v = next;
}
```
이 코드는 $A$의 초기 0이 아닌 값들만 보고 `ancestor` 배열을 활용해 바텀업으로 빠르게 트리를 구성합니다. 불필요한 탐색을 줄이는 아주 우아한 로직입니다.

### Hacker News의 반응과 나의 생각

이 글이 Hacker News에 올라왔을 때 폭발적인 댓글 토론이 일어나진 않았습니다. 아무래도 주제 자체가 수치해석의 꽤 깊은 영역을 다루고 있기 때문이겠죠. 하지만 이런 류의 '근본적인 알고리즘을 코드로 해부하는' 글은 시니어 엔지니어들의 북마크 리스트에 조용히, 하지만 확실하게 자리 잡습니다.

개인적인 경험을 덧붙이자면, 2018년쯤 사내에서 대규모 그래프 분석 엔진을 최적화할 때 이와 정확히 동일한 문제에 직면한 적이 있습니다. 당시 저희 팀은 무식하게 거대한 Hash Map을 써서 중간 결과를 저장하다가 Out Of Memory로 고통받았죠. 만약 그때 이 Elimination Tree의 개념을 명확히 이해하고 Symbolic 과정을 통해 메모리 레이아웃을 미리 확정 지었다면, 수주의 삽질을 아낄 수 있었을 겁니다.

### 결론 (Verdict)

대부분의 백엔드 엔지니어나 프론트엔드 개발자가 실무에서 직접 Sparse Cholesky 분해를 구현할 일은 제로에 가깝습니다. 이미 LAPACK, SuiteSparse, Eigen 같은 훌륭한 라이브러리들이 있으니까요.

하지만 이 글이 주는 진짜 가치는 **'복잡한 실행 흐름을 DAG로 모델링하고, 알고리즘의 내재적 규칙을 찾아내어 이를 단순한 트리로 압축하는 사고방식'** 에 있습니다. 추상적인 수학 이론을 코드의 실행 흐름으로 끌어내려 설명한 Reid Atcheson의 글쓰기 방식은 우리 모두가 배워야 할 훌륭한 기술 블로깅의 표본입니다. 성능 최적화의 한계에 부딪혀 본 엔지니어라면, 꼭 원문을 정독해 보시길 권합니다.
