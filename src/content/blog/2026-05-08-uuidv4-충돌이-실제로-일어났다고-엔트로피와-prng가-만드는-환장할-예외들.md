최근 해커뉴스(Hacker News)를 뜨겁게 달군 흥미로운 스레드가 하나 올라왔다. 누군가 프로덕션 환경에서 실제 **UUIDv4** 충돌을 겪었다는 글이다.

우리 모두 주니어 시절부터 귀에 못이 박히도록 들은 이야기가 있다. "UUIDv4가 충돌할 확률은 지구에 소행성이 충돌할 확률보다 낮다." 수학적으로 122비트의 난수 공간을 가지므로, 초당 10억 개의 UUID를 100년 동안 생성해도 충돌 확률은 50% 정도에 불과하다. 그래서 대부분의 엔지니어는 데이터베이스의 Primary Key로 UUID를 마음 편히 때려 박는다.

하지만 현실은 수학 교과서와 다르다. 이번 HN 스레드에서 드러난 문제의 핵심과, 15년 차 엔지니어로서 현업에서 겪은 기상천외한 UUID 삽질기들을 정리해 보겠다.

## 문제는 수학이 아니라 '엔트로피'다

수학적인 확률은 완벽한 난수(True Random)를 가정한다. 하지만 컴퓨터는 결정론적인 기계다. 우리가 사용하는 난수는 결국 **PRNG** (의사 난수 생성기)에서 나오며, 이 PRNG의 품질은 초기화에 사용되는 **Seeding** 과 시스템의 **Entropy** 에 전적으로 의존한다.

HN 스레드에서 한 유저가 핵심을 찔렀다.
> "UUIDv4의 보안과 고유성은 고품질 엔트로피 소스라는 가정하에 성립한다. 하드웨어 결함, 소프트웨어 버그, 혹은 '고품질 엔트로피'가 뭔지 모르는 개발자들에 의해 이 가정은 무참히 깨진다."

만약 자바스크립트 환경에서 `crypto.getRandomValues()` 대신 오래된 Polyfill이나 엉성한 난수 생성 로직을 탄다면 어떻게 될까?

```javascript
// 절대 프로덕션에서 이런 식의 커스텀 난수를 UUID에 섞지 마라
function badRandomUUID() {
  return 'xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx'.replace(/[xy]/g, function(c) {
    var r = Math.random() * 16 | 0;
    var v = c == 'x' ? r : (r & 0x3 | 0x8);
    return v.toString(16);
  });
}
```

`Math.random()`은 암호학적으로 안전하지 않다. V8 엔진 등에서 내부 상태가 겹치거나, 초기 시드가 동일한 두 프로세스가 동시에 뜬다면 완전히 똑같은 난수 시퀀스를 뱉어낸다. 원글 작성자의 경우, 과거에 유저의 **Client-side** (모바일 기기)에서 UUID를 생성해서 서버로 보냈다고 한다. 저가형 안드로이드 기기나 구형 브라우저의 조악한 엔트로피 환경을 고려하면, 충돌은 더 이상 소행성 충돌 급의 기적이 아니라 필연에 가깝다.

## 코미디가 된 마이크로서비스: UUID 검증팀

스레드를 읽다가 정말 육성으로 뿜었던 댓글이 하나 있다. 어떤 스타트업에 합류한 CTO가 발견한 기형적인 아키텍처 이야기다.

- **전담 팀:** 무려 3명의 엔지니어와 1명의 DB 관리자가 'UUID 발급 마이크로서비스'를 전담하고 있었다.
- **동작 방식:** 클라이언트가 UUID를 요청하면, 새로 생성한 UUID가 기존 DB에 존재하는지 검증한다. 없으면 INSERT 후 반환한다.
- **업무 프로세스:** 이 팀은 자신들만의 칸반 보드와 스프린트까지 가지고 있었다.

솔직히 말해서, 10년 전쯤 나도 비슷한 구조를 대기업 사내 시스템에서 본 적이 있다. "절대 충돌하면 안 된다"는 강박관념이 빚어낸 오버엔지니어링의 극치다. 결국 이 시스템은 성능 병목을 견디지 못하고 글로벌 Incrementing Counter(트위터의 Snowflake 방식)로 진화했다고 한다.

## 클라이언트의 난수를 믿지 마라

분산 시스템에서 중앙 집중형 DB 없이 고유 ID를 생성할 수 있다는 것이 UUID의 가장 큰 매력이다. 하지만 그 생성 주체가 **통제 불가능한 클라이언트** 라면 이야기가 달라진다.

VM 환경이나 컨테이너 환경에서도 엔트로피 고갈(Entropy Starvation) 문제는 심심찮게 발생한다. 하물며 유저의 스마트폰? 절대 믿으면 안 된다. ID 생성은 무조건 신뢰할 수 있는 서버 사이드에서, `/dev/urandom`이나 검증된 Crypto 라이브러리를 통해 이루어져야 한다.

## 대안과 결론: 우리는 여전히 UUIDv4를 써야 할까?

개인적으로 단순한 UUIDv4는 점차 레거시가 되어가고 있다고 생각한다. 최근의 분산 환경에서는 시간 순서대로 정렬 가능하면서도 충돌 방지 메커니즘이 포함된 ID 체계가 훨씬 실용적이다.

- **Twitter Snowflake:** 타임스탬프 + 데이터센터 ID + 워커 ID + 시퀀스를 조합해 충돌을 원천 차단한다.
- **ULID / CUID2:** 시간 기반 정렬이 가능하면서도 URL-safe하고, UUID와 호환되는 훌륭한 대안들이다.

결론적으로, 이번 해커뉴스 해프닝은 **수학적 확률이 시스템의 구현 결함까지 커버해주진 않는다** 는 뼈아픈 교훈을 남겼다. 여러분의 시스템이 아직도 `Math.random()` 기반의 정체불명 npm 패키지로 ID를 생성하고 있다면, 내일 당장 충돌 로그를 보게 될지도 모른다.
